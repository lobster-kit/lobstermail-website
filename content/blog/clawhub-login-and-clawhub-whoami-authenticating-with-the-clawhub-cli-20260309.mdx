You installed the ClawHub CLI, ran your first command, and got a `401 Unauthorized` error. Or maybe you authenticated last week and now `clawhub whoami` returns nothing useful. Either way, you're stuck before you've done anything interesting.

ClawHub is the skill directory for OpenClaw. It's how you search, install, and manage skills that give your agent capabilities. But none of that works until you're authenticated. The CLI offers three distinct ways to log in, each suited to different situations, and one verification command to confirm everything is working.

Here's the full picture.

## ClawHub authentication: 3 methods at a glance

1. **Browser flow** — run `clawhub login`, a browser window opens for OAuth authentication
2. **Token flag** — run `clawhub login --token <your-api-token>` for headless or CI use
3. **Environment variable** , set `export CLAWHUB_AUTH="your-auth-here"` for automated pipelines

After any of these, run `clawhub whoami` to verify your session is active.

Each method stores credentials differently and has different tradeoffs for security, convenience, and automation. Let's walk through them.

## Method 1: browser-based login

This is the default. Run:

```bash
clawhub login
The CLI opens your default browser to an OAuth consent screen. You approve access, the browser redirects back, and the CLI stores your session token locally. It's the simplest option when you're sitting at a computer and can interact with a browser window.

The stored token persists between terminal sessions. You don't need to re-authenticate every time you open a new shell. The CLI reads the cached credentials automatically until they expire or you explicitly log out.

This method works well for local development and manual skill management. It does not work in environments without a browser: remote servers, Docker containers, CI runners, or automated agent pipelines. For those, you need one of the other two methods.

## Method 2: token-based login

If you can't open a browser (or don't want to), pass a token directly:

```bash
clawhub login --token <your-api-token>
This skips the browser entirely. You generate an API token from your ClawHub account settings, then pass it as an argument. The CLI validates the token against the OpenClaw API and caches it the same way the browser flow would.

This is the right choice for CI/CD pipelines, remote servers, and any situation where interactive login isn't possible. A few things to keep in mind:

- Don't hardcode tokens in scripts that get committed to version control. Use secrets management (GitHub Actions secrets, environment variables injected at runtime, your CI platform's secret store).
- Tokens can expire or be revoked. If your pipeline starts failing with 401 errors after working fine for weeks, check whether the token is still valid.
- You can generate multiple tokens for different environments. One for your local machine, one for CI, one for your staging server. If one gets compromised, revoke it without disrupting everything else.

## Method 3: the CLAWHUB_AUTH environment variable

The third option bypasses the `clawhub login` command entirely:

```bash
export CLAWHUB_AUTH="your-auth-here"
```

When this environment variable is set, the CLI uses it for every request. No login command needed. No cached credentials on disk. The variable takes precedence over any stored session.

This is the most flexible option for automation. Your agent orchestration layer can set the variable at runtime, rotate it on a schedule, and never write credentials to the filesystem. It's also useful for quick testing: set the variable, run a few commands, unset it, done.

The tradeoff is that environment variables are visible to any process running in the same shell session. On a shared server, that's a real concern. On a single-user development machine or an isolated container, it's usually fine.

Here's a comparison of all three methods:

| Method | Best for | Requires browser | Persists to disk | Automation-friendly |
|---|---|---|---|---|
| `clawhub login` | Local development | Yes | Yes | No |
| `clawhub login --token` | CI/CD, remote servers | No | Yes | Yes |
| `CLAWHUB_AUTH` env var | Agent pipelines, ephemeral environments | No | No | Yes |

## Verifying your session with clawhub whoami

After authenticating, run:

```bash
clawhub whoami
```

This command calls the OpenClaw API to validate your current credentials and returns your account information. If authentication is working, you'll see your username and account details. If something is wrong, you'll get an error.

`clawhub whoami` is useful in three situations:

**Confirming a fresh login worked.** Right after `clawhub login`, run `whoami` to make sure the session was stored correctly. Don't assume success based on the login command's output alone.

**Debugging CI failures.** When a pipeline step fails with authentication errors, add `clawhub whoami` as an early step to isolate whether the problem is auth or something else. If `whoami` fails, you know the token is bad before you waste time investigating skill installation issues.

**Checking for stale credentials.** Tokens expire. If your agent has been running for a while without re-authenticating, `whoami` tells you whether the session is still valid. This is especially useful when your agent orchestration layer needs to verify session state programmatically before attempting operations.

You can script around the output. If `clawhub whoami` returns a non-zero exit code, trigger a re-authentication flow. If it returns successfully, proceed with skill operations. This turns a manual verification step into an automated health check.

## Logging out

When you're done, or when you need to switch accounts:

```bash
clawhub logout
```

This clears the stored credentials from disk. After logging out, any `clawhub` command that requires authentication will fail until you log in again. The `CLAWHUB_AUTH` environment variable is not affected by `logout` since it lives in your shell, not in the CLI's credential store.

If you're switching between accounts (personal vs. team, for example), the sequence is:

```bash
clawhub logout
clawhub login --token <different-token>
clawhub whoami  # confirm you're now the right user
```

## Troubleshooting 401 errors

A `401 Unauthorized` from `clawhub whoami` means the CLI sent credentials that the API rejected. The most common causes, in order of likelihood:

**Expired token.** Tokens don't last forever. Re-authenticate with `clawhub login` or generate a fresh API token.

**Revoked token.** If someone (or an automated policy) revoked the token from the account settings, it won't work anymore regardless of when it was issued. Generate a new one.

**Wrong environment variable.** If `CLAWHUB_AUTH` is set to an old or invalid value, it overrides your cached login. Check with `echo $CLAWHUB_AUTH`. Unset it with `unset CLAWHUB_AUTH` if you want to fall back to the cached credentials.

**Network issues.** The CLI needs to reach the OpenClaw API to validate credentials. If you're behind a corporate proxy or firewall, the request might be blocked. This usually manifests as a timeout rather than a 401, but misconfigured proxies can return unexpected status codes.

**Version mismatch.** Older versions of the ClawHub CLI might use a deprecated authentication endpoint. Run `clawhub --cli-version` and update if you're behind. You can manage CLI versions with:

```bash
clawhub --cli-version
```

## Agents and headless authentication at scale

If you're running multiple agents that each need to interact with ClawHub, the browser flow obviously won't work. Here's what does:

Generate a dedicated API token for each agent or agent group. Store tokens in your secrets manager. At agent startup, either pass the token via `clawhub login --token` or set `CLAWHUB_AUTH` in the agent's environment. Run `clawhub whoami` as a health check before the agent begins its work.

For agents that run continuously, build in periodic re-authentication. Check the exit code of `clawhub whoami` every few hours. If it fails, trigger the token refresh flow. This prevents an agent from running for days with stale credentials and only discovering the problem when a critical operation fails.

If your agents use LobsterMail for email (where the agent self-provisions its own inbox without human intervention), you already understand the pattern. The agent handles its own setup. ClawHub authentication works the same way: give the agent a token, let it verify with `whoami`, and let it get to work.

## Wrapping up

Start with `clawhub login` if you're at your desk. Use `clawhub login --token` for anything automated. Use `CLAWHUB_AUTH` when you don't want credentials touching the filesystem. And always run `clawhub whoami` after authenticating to confirm the session is real.

If you're building agents that need to authenticate with multiple services (ClawHub for skills, email for communication, APIs for data), the token-plus-verification pattern applies everywhere. Get the credential, verify it works, then proceed. For the email piece, [LobsterMail](/) handles agent-provisioned inboxes so your agent doesn't need a human to set up email access.
