Your agent just tried to read an inbox through the Gmail API. Google responded with HTTP 429 and a message that says something like `"User-rate limit exceeded. Retry after 2026-03-15T14:22:07.000Z."` Your agent, being an agent, immediately retried. Got another 429. Retried again. Another 429. Now Google is actively throttling you harder, and no emails are getting through at all.

This is one of the most common failure modes for any automated system built on top of Gmail's API. The 429 status code means "Too Many Requests," and Google includes a `Retry-After` value telling you exactly when you're allowed to try again. Most agents ignore it. That's the problem.

Let's break down what the Gmail API 429 error actually means, how to read the retry timestamp, and how to build retry logic that doesn't make things worse.

## What a Gmail API 429 error is telling you

Google's Gmail API enforces two layers of rate limiting. The first is a per-user quota: 250 quota units per user per second, calculated as a moving average. The second is a project-level daily cap, which defaults to 1,000,000,000 quota units per day (though most projects will never touch that ceiling).

When you exceed the per-user rate, Google returns either an HTTP 429 or an HTTP 403 with a `rateLimitExceeded` error reason. The distinction matters less than people think. Both mean the same thing: slow down.

The 429 response body typically includes a timestamp in ISO 8601 format. This is the `Retry-After` value. It's not telling you how many seconds to wait. It's giving you a specific moment in time after which Google will start accepting requests from you again. The difference matters when your server clock is off by a few seconds, or when you're parsing the value in a language that expects different formats.

Here's what the error looks like in practice:

```json
{
  "error": {
    "code": 429,
    "message": "Rate Limit Exceeded",
    "status": "RESOURCE_EXHAUSTED",
    "details": [
      {
        "reason": "RATE_LIMIT_EXCEEDED",
        "metadata": {
          "consumer": "projects/123456",
          "quota_limit": "defaultPerMinutePerUser",
          "quota_limit_value": "250"
        }
      }
    ]
  }
}
Sometimes the retry timestamp appears in the response headers. Sometimes it's in the error body. Sometimes Google doesn't include one at all, and you're left guessing. Your code needs to handle all three cases.

## How to handle Gmail API 429 errors: step-by-step

1. Detect the 429 (or 403 `rateLimitExceeded`) response code
2. Parse the `Retry-After` ISO timestamp from the error body or headers
3. Calculate wait duration as `retryAfterTimestamp - Date.now()`
4. Sleep for that duration plus 500-2000ms of random jitter
5. Retry the request once
6. If it fails again, apply exponential backoff with a 60-second cap
7. After 5 consecutive failures, enqueue for later processing and alert

That's the whole strategy. The nuance is in the implementation.

## Parsing the Retry-After timestamp

The Gmail API doesn't always follow a single format. You might see an ISO 8601 timestamp like `2026-03-15T14:22:07.309Z`, or you might see a raw integer representing seconds. Your parsing logic should try both:

```typescript
function getRetryDelay(response: Response): number {
  const retryAfter = response.headers.get('Retry-After');
  if (!retryAfter) return 5000; // default 5s if missing

  // Try parsing as seconds first
  const seconds = parseInt(retryAfter, 10);
  if (!isNaN(seconds)) return seconds * 1000;

  // Try parsing as ISO timestamp
  const timestamp = new Date(retryAfter).getTime();
  if (!isNaN(timestamp)) {
    const delay = timestamp - Date.now();
    return Math.max(delay, 1000); // at least 1 second
  }

  return 5000; // fallback
}
The `Math.max` guard is important. If your server clock is slightly ahead of Google's, the calculated delay could be negative. A negative delay means "retry immediately," which is exactly what got you rate-limited in the first place.

## Exponential backoff that actually works

Exponential backoff is simple in theory: wait 1 second, then 2, then 4, then 8. In practice, most implementations miss two things.

First, you need jitter. If 50 agent workers all hit a 429 at the same time and all back off to exactly 2 seconds, they'll all retry at exactly the same time and immediately get rate-limited again. Adding a random delay between 0 and 1 second breaks up the thundering herd.

Second, you need a cap. Pure exponential backoff grows to absurd wait times. After 10 retries you'd be waiting 1,024 seconds (over 17 minutes). Cap it at 60 seconds for most use cases.

```typescript
function backoffDelay(attempt: number): number {
  const base = Math.min(60000, 1000 * Math.pow(2, attempt));
  const jitter = Math.random() * 1000;
  return base + jitter;
}
```

Here's the combined approach: use the `Retry-After` timestamp for the first retry, then fall back to exponential backoff if subsequent retries also fail.

```typescript
async function gmailRequestWithRetry(
  fn: () => Promise<Response>,
  maxRetries = 5
): Promise<Response> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const res = await fn();

    if (res.status !== 429 && res.status !== 403) return res;

    const delay = attempt === 0
      ? getRetryDelay(res)
      : backoffDelay(attempt);

    console.warn(
      `Gmail 429: retry ${attempt + 1}/${maxRetries}, waiting ${delay}ms`
    );
    await new Promise(r => setTimeout(r, delay));
  }

  throw new Error('Gmail API rate limit: max retries exceeded');
}
```

## The multi-agent problem nobody talks about

Most Stack Overflow answers assume a single application making requests on behalf of one user. That's not how agents work in practice.

If you're running multiple agents that share a single Gmail OAuth token (common in multi-tenant setups), all of them draw from the same 250-quota-units-per-second bucket. Agent A reads emails. Agent B sends a message. Agent C labels something. They're all competing for the same pool, and none of them know what the others are doing.

This is where things get genuinely difficult. A 429 hitting Agent C might be caused by Agent A's burst of reads 200 milliseconds earlier. Agent C backs off, but Agent A doesn't know to slow down. The result is a whack-a-mole game where random agents keep tripping the limit while others wait.

The fix is centralized rate tracking. Instead of letting each agent manage its own retries, route all Gmail API calls through a shared rate limiter (a token bucket or sliding window counter) that enforces the per-user quota across all agents before any request goes out. Redis works well for this. A simple sorted set with timestamps can track requests per user per second.

But let's be honest: if you're building centralized rate limiters with Redis coordination layers just to read email, you've wandered pretty far from your agent's actual job.

## What happens when you ignore 429 errors

Nothing good. Google's rate limiting is progressive. The first 429 might have a retry window of 30 seconds. If you keep hammering, the next window could stretch to several minutes. In extreme cases, Google can temporarily suspend API access for the entire project, not just the offending user.

I've seen projects where an agent entered a tight retry loop on a 429, burned through thousands of failed requests in minutes, and triggered a project-wide lockout that took hours to resolve. The agent was trying to be helpful. It saw "request failed," so it tried again. Faster. That instinct (which works fine for network timeouts) is exactly wrong for rate limits.

## When the Gmail API returns 429 with barely any activity

This comes up a lot in forums, and the answer is almost always one of three things:

**Quota units aren't 1:1 with requests.** A `messages.get` call costs 5 quota units. A `messages.send` costs 100. A single send already eats 40% of your per-second allowance. Two sends in rapid succession and you're already at 80%.

**The moving average catches bursts.** Google calculates the per-user rate as a moving average. If your agent fires off 10 reads in a single millisecond (250 units instantly), the moving average for that second spikes even though the absolute count seems low.

**Shared credentials.** If another application or agent is using the same OAuth token, their usage counts against your quota too. Check the Google Cloud Console's API dashboard to see actual usage graphs broken down by method.

## A simpler path for agents

The Gmail API was designed for applications acting on behalf of human users. The entire quota model assumes a human at the other end who sends a few messages per minute and checks their inbox a few times per hour. Agents don't work that way. They poll. They batch. They burst. Every design assumption in the Gmail API fights against automated workflows.

If your agent needs email as a capability (sending verification emails, receiving confirmations, processing inbound messages), you don't necessarily need Gmail at all.
