In January 2026, a review published in *Information* found that prompt injection now ranks as the #1 vulnerability in the OWASP Top 10 for LLM Applications, showing up in over 73% of production AI deployments assessed. That stat is bad enough for single-agent setups. But when agents talk to each other, the math gets worse fast.

## What is multi-agent prompt infection spreading?

Multi-agent prompt infection spreading is an attack where a compromised LLM agent injects malicious instructions into neighboring agents through shared context, tool outputs, or message passing. The infection self-replicates as each newly compromised agent propagates the payload to others it interacts with, enabling coordinated data exfiltration, unauthorized actions, or disinformation across the entire system.

This isn't theoretical. Researchers at the University of Texas demonstrated it in their [Prompt Infection paper](https://arxiv.org/html/2410.07283v1), showing that a single compromised agent in a multi-agent system can spread its payload to every reachable agent. The infection doesn't need a human in the loop. It moves through the channels agents already use to collaborate.

## How it actually works

Traditional prompt injection targets one model. You sneak instructions into a user input, and the model follows them instead of its system prompt. Multi-agent prompt infection takes this further by turning each victim into a carrier.

Here's the sequence:

1. An attacker crafts a malicious payload and delivers it to one agent (through an email, a web page, a document, anything the agent processes as input).
2. The compromised agent embeds the payload into its outputs: tool calls, messages to other agents, or shared memory.
3. The next agent reads that output as trusted context. It follows the injected instructions.
4. That agent now carries the infection and passes it along to the agents downstream from it.

The researchers found that self-replicating infections (where the payload includes instructions to copy itself) significantly outperformed non-replicating ones. In larger agent populations, the infection spread faster, not slower. The propagation follows a logistic growth curve: slow at first, then exponential, then saturating once most agents are compromised.

Think of it like a biological virus. Patient zero infects two contacts. Each of them infects two more. Within a few cycles, the entire population is carrying the payload.

## Why email is the most dangerous entry point

Most discussions about multi-agent prompt infection focus on agent-to-agent messaging. But the initial infection has to come from somewhere, and email is the widest open door.

Consider a typical agent workflow: your agent checks an inbox, reads an email, extracts relevant information, and passes that to other agents for processing (scheduling, CRM updates, code generation, customer replies). If the email body contains injected instructions, the receiving agent treats them as legitimate content. It follows the instructions, and every downstream agent inherits the infection.

Email is uniquely dangerous here for three reasons.

First, email is unstructured. Unlike API calls with typed schemas, email bodies are free text. There's no inherent boundary between "content" and "instruction" for an LLM parsing the message.

Second, email is externally sourced. Anyone on the internet can send your agent an email. You don't control the input the way you might control internal API traffic.

Third, email often triggers multi-step workflows. A single inbound message might touch a classification agent, an extraction agent, a response-drafting agent, and a CRM agent. One malicious email, four compromised agents.

## What LLM Tagging does (and doesn't do)

LLM Tagging is a defense mechanism where each agent marks its outputs with metadata indicating the source and processing history. The idea is that downstream agents can verify provenance before trusting content.

It helps. But it doesn't solve the problem on its own.

LLM Tagging assumes agents check tags before processing. In practice, many multi-agent frameworks pass messages as plain text without structured provenance fields. Even when tags exist, a sufficiently clever injection can instruct the agent to strip or forge them before forwarding. The infection payload can include "ignore all source tags" as part of its instructions.

Tagging is one layer. You need more.

## Infrastructure-level defenses that actually help

The missing piece in most prompt infection discussions is infrastructure. Everyone talks about model-level defenses (system prompts, guardrails, output filtering). Almost nobody talks about what happens at the transport layer, where messages enter and move between agents.

Here's what makes a real difference:

**Injection risk scoring at the inbox.** Before your agent ever sees an email, the infrastructure should scan the content for injection patterns and assign a risk score. LobsterMail does this automatically. Every inbound email comes with security metadata and an injection risk score, so your agent can decide whether to process, quarantine, or reject the message before it enters the pipeline.

**Sandboxed inboxes.** Each agent should have its own isolated inbox. If one inbox receives a malicious message, the blast radius is limited to that agent's scope. With [LobsterMail's inbox model](/docs/getting-started), agents self-provision isolated addresses. There's no shared inbox where a single poisoned email reaches every agent in the system.

**Message provenance tracking.** When agents pass information to each other, the infrastructure should track where each piece of content originated. Did this text come from an external email? From an internal agent? From a verified API? Provenance matters because it determines trust level.

**Agent identity verification.** In a multi-agent system, agents should authenticate to each other. An infected agent trying to impersonate a trusted peer should fail the identity check.

## The first 60 minutes after detection

Most resources on prompt infection focus on prevention. Nobody talks about what to do when you catch one mid-spread. Here's a practical playbook:

**Minutes 0-10: Isolate.** Stop inter-agent message passing immediately. If your agents communicate through a message bus or shared context, pause it. The goal is to halt propagation before assessing damage.

**Minutes 10-25: Identify patient zero.** Check your email inboxes for the initial infection vector. Look at the injection risk scores on recent inbound messages. Find the email (or external input) that started the chain.

**Minutes 25-45: Audit downstream agents.** Review the outputs of every agent that received data from the compromised agent. Check for anomalous tool calls, unexpected external communications, or data access patterns that don't match normal behavior.

**Minutes 45-60: Reset and restart.** Clear the context and conversation history of all affected agents. Reload their system prompts. Resume message passing only after confirming clean state.

This is easier when your email infrastructure provides an audit trail. If you're routing agent email through Gmail or a personal inbox, reconstructing the chain of infection is manual and painful.

## Which architectures are most vulnerable

Not all multi-agent setups carry equal risk. Architecture matters.

**Hub-and-spoke** (one orchestrator, many workers) is moderately vulnerable. If a worker gets infected, it can only pass the infection back to the orchestrator. But if the orchestrator gets infected, every worker is exposed.

**Peer-to-peer** (agents communicate freely with each other) is the most vulnerable. Any agent can infect any other agent, and the infection spreads through multiple paths simultaneously. The logistic growth curve hits hardest here.

**Hierarchical** (strict layers, agents only talk to their immediate neighbors) is the most resistant. Infection can only spread one layer at a time, giving you more time to detect and isolate.

If you're designing a multi-agent system that processes external inputs (especially email), hierarchical architectures with strict message boundaries give you the best chance of containing an infection before it goes systemic.

## What to do right now

If you're running agents that process email and pass information to other agents, you have three immediate steps:

1. Add injection scanning at the email layer, before your agents process content. LobsterMail's built-in [security and injection scoring](/docs/getting-started) handles this automatically.
2. Isolate your agent inboxes. One agent, one inbox. No shared mailboxes where a single poisoned message reaches your entire pipeline.
3. Design your inter-agent communication with containment in mind. Hierarchical beats peer-to-peer when external inputs are involved.

Multi-agent prompt infection spreading is a real, demonstrated attack vector. The defenses aren't complicated, but they need to exist at the infrastructure level, not just in your prompts.
