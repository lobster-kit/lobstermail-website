Every email you've ever received has hidden metadata that determines whether it shows up as part of a conversation or lands as a standalone message. Three headers do the heavy lifting: `Message-ID`, `In-Reply-To`, and `References`. If you're building anything that sends email programmatically (an agent, a transactional system, a notification pipeline), getting these right is the difference between clean threaded conversations and a scattered mess in your recipient's inbox.

Here's how they work, where they differ, and how to set them correctly.

## In-Reply-To vs References: the quick comparison

The **In-Reply-To** header contains the `Message-ID` of the single email being replied to. The **References** header contains a chain of `Message-ID` values representing the entire conversation history.

| | In-Reply-To | References |
|---|---|---|
| Definition | Points to the direct parent message | Lists the full ancestry of the thread |
| Value format | Single `Message-ID` | Space-separated list of `Message-ID` values |
| Number of IDs | Always one | One to many (recommended cap: ~10) |
| Primary purpose | Identifies which message this is a reply to | Lets clients reconstruct the full thread tree |
| Fallback behavior | Clients may fall back to subject-line matching | Clients walk the list to find any known ancestor |

Most email clients use both headers together. But when they conflict or one is missing, behavior gets unpredictable. More on that below.

## Message-ID: the foundation

Before you can understand threading, you need to understand `Message-ID`. Every email gets a unique identifier when it's sent, formatted like an email address wrapped in angle brackets:

Message-ID: <abc123@mail.example.com>
This ID is generated by the sending mail server or client. It should be globally unique. RFC 5322 defines the format, and most implementations use a combination of timestamp, random string, and domain name to guarantee uniqueness.

The `Message-ID` is what `In-Reply-To` and `References` point back to. Without it, threading can't work at all. If your sending infrastructure strips or fails to generate `Message-ID` headers, every downstream reply will be orphaned.

## How In-Reply-To works

When someone replies to an email, their mail client takes the `Message-ID` of the original message and places it in the `In-Reply-To` header of the reply:

```
Message-ID: <reply-456@mail.example.com>
In-Reply-To: <abc123@mail.example.com>
```

This creates a direct parent-child relationship. The email client receiving this reply can now display it underneath the original message.

Simple enough. But `In-Reply-To` only tells you about the immediate parent. In a long conversation with ten messages, the fifth reply only knows about the fourth message. It has no awareness of messages one through three. That's where `References` comes in.

## How References works

The `References` header contains the full chain of `Message-ID` values from the root of the conversation down to the parent:

```
Message-ID: <reply-789@mail.example.com>
In-Reply-To: <reply-456@mail.example.com>
References: <abc123@mail.example.com> <reply-456@mail.example.com>
```

The convention, defined in RFC 5322 (and its predecessor RFC 2822), is straightforward: copy the `References` header from the parent message, then append the parent's `Message-ID` to the end. The first ID in the list is always the root message. The last ID is always the direct parent.

For deeply nested threads, RFC 5322 recommends trimming the list if it exceeds roughly ten entries. When trimming, keep the first ID (the root) and remove IDs from the middle, preserving the most recent ancestors. This prevents headers from growing unbounded while keeping enough context for clients to stitch threads together.

## How email clients use these headers

Different email clients prioritize these headers differently, and this is where things get interesting.

**Gmail** is aggressive about threading. It uses `References` and `In-Reply-To` as the primary signals, but it also groups messages by subject line. This means two completely unrelated emails with the same subject can end up in the same conversation view. Gmail also threads messages sent from the same account to the same recipient with the same subject, even without any threading headers at all. This behavior is well-documented, frequently complained about, and unlikely to change.

**Outlook** relies more strictly on `In-Reply-To` and `References`. Subject-line matching plays a smaller role. If threading headers are missing, Outlook is more likely to display messages separately. It also supports its own proprietary `Thread-Index` and `Thread-Topic` headers, which Outlook clients use internally but other providers ignore.

**Apple Mail** follows the RFC standards closely. It uses `In-Reply-To` and `References` for threading and does not aggressively group by subject line. Of the major clients, Apple Mail is the most predictable.

**Thunderbird** and most open-source clients follow RFC 5322 faithfully. Threading works as specified: `References` for tree construction, `In-Reply-To` for direct parent identification.

The takeaway: if you set both headers correctly, your emails will thread properly in every major client. If you only set one, you're gambling on client-specific fallback behavior.

## What happens when threading headers are missing

When `In-Reply-To` and `References` are both absent, email clients have to guess. Gmail falls back to subject-line matching, which is why you sometimes see unrelated messages grouped together if they share a generic subject like "Re: Update" or "Re: Question." Outlook tends to leave headerless messages ungrouped. Apple Mail does the same.

When `In-Reply-To` is missing but `References` is present, most clients can still thread correctly by walking the `References` list. The reverse (only `In-Reply-To`, no `References`) works for simple two-message exchanges but breaks down in longer threads because the client can't reconstruct the conversation tree.

The worst case is when headers are present but contain incorrect or spoofed `Message-ID` values. A message with a fabricated `In-Reply-To` can inject itself into an existing thread, which has real security implications for agents processing email. If your agent trusts that all messages in a thread are from the same conversation, a spoofed `In-Reply-To` header can trick it into treating a malicious message as a legitimate reply. This is one reason [multi-agent email coordination](/blog/multi-agent-email-coordination) requires careful validation beyond just checking thread membership.

## Setting threading headers in programmatic email

If your agent or application sends email through an API or SMTP, you're responsible for setting these headers yourself. Most email APIs (SendGrid, Postmark, SES, LobsterMail) let you pass custom headers with each message.

Here's the pattern for a correct reply:

```typescript
// When sending a reply to a received email:
const replyHeaders = {
  'In-Reply-To': originalEmail.messageId,
  'References': originalEmail.references
    ? `${originalEmail.references} ${originalEmail.messageId}`
    : originalEmail.messageId,
  'Subject': `Re: ${originalEmail.subject}`,
};
```

A few things to note:

1. **Always set both headers.** Some guides say `In-Reply-To` alone is sufficient. It isn't, at least not reliably across all clients.

2. **Preserve the `References` chain.** Don't just set `References` to the parent's `Message-ID`. Copy the parent's `References` value and append the parent's `Message-ID`. This is how threads stay intact across many replies.

3. **Keep the subject line consistent.** Prepend `Re:` if it isn't already there, and don't modify the rest. Gmail in particular uses subject matching as a secondary threading signal.

4. **Don't fabricate `Message-ID` values.** If you're replying to a message, use the actual `Message-ID` from the original. Generating a fake one will break the thread.

5. **Store `Message-ID` and `References` on your side.** If your agent needs to [schedule meetings over email](/blog/agent-schedule-meetings-email) or manage ongoing conversations, you'll want to persist these values so future replies reference the correct chain.

## Threading for AI agents

Agents that send and receive email at scale hit threading edge cases that human users rarely encounter. An agent might process hundreds of conversations simultaneously, each with its own thread. Mixing up a `Message-ID` between conversations means cross-contaminating threads, which can confuse both the recipient and the agent itself.

A few practical recommendations:

Store the `Message-ID`, `In-Reply-To`, and `References` of every message your agent sends and receives. Index them by conversation or task ID. When the agent needs to reply, look up the most recent message in that conversation and construct headers from its stored metadata.

If your agent starts a new conversation, don't set `In-Reply-To` or `References` at all. Let the first message establish the root `Message-ID`, and only add threading headers on subsequent replies.

When an agent receives a reply, validate that the `In-Reply-To` value matches a `Message-ID` your agent actually sent. This is a basic defense against thread injection attacks, where a sender spoofs threading headers to insert messages into an existing conversation.

LobsterMail returns threading metadata (`messageId`, `inReplyTo`, `references`) on every received email, so your agent can read and set these headers without parsing raw MIME. The `receive()` method surfaces these fields directly, and `send()` accepts them as parameters.

## Testing your threading

Before sending at scale, verify your headers are working. Send a test email, reply to it programmatically with correct headers, and check the result in Gmail, Outlook, and Apple Mail. All three should show a threaded conversation.

If threads split unexpectedly, check these common causes: the `Message-ID` format is malformed (missing angle brackets or domain portion), the `References` header was overwritten instead of appended to, or the subject line was modified between messages. Any of these can break threading in at least one major client.

  <FAQItem question="What is the In-Reply-To header and what value should it contain?">
    The `In-Reply-To` header identifies which message an email is replying to. It should contain exactly one `Message-ID` value from the parent email, wrapped in angle brackets (e.g., `<abc123@mail.example.com>`).
  </FAQItem>

  <FAQItem question="What is the References header and how does it differ from In-Reply-To?">
    The `References` header contains a space-separated list of `Message-ID` values representing the full conversation ancestry, from root to parent. `In-Reply-To` only points to the direct parent, while `References` maps the entire thread tree.
  </FAQItem>

  <FAQItem question="Do I need to set both In-Reply-To and References, or is one enough?">
    Set both. While some clients can thread with just one header, behavior varies across Gmail, Outlook, and Apple Mail. Setting both gives you reliable threading everywhere.
  </FAQItem>

  <FAQItem question="How does Gmail's threading differ from Outlook's?">
    Gmail threads aggressively using `References`, `In-Reply-To`, and subject-line matching. Outlook relies more strictly on the threading headers and gives less weight to subject lines. Outlook also uses proprietary `Thread-Index` and `Thread-Topic` headers that other clients ignore.
  </FAQItem>

  <FAQItem question="What is a Message-ID and how is it generated?">
    A `Message-ID` is a globally unique identifier assigned to every email by the sending server. It typically combines a timestamp, random string, and domain name in the format `<unique-string@domain.com>`. It's defined by RFC 5322.
  </FAQItem>

  <FAQItem question="What happens when both In-Reply-To and References are missing?">
    Email clients fall back to heuristics. Gmail groups messages by matching subject lines, which can incorrectly merge unrelated emails. Outlook and Apple Mail are more likely to display them as separate messages.
  </FAQItem>

  <FAQItem question="How should I construct the References header for a deeply nested reply?">
    Copy the `References` header from the parent message and append the parent's `Message-ID` to the end. If the list exceeds about ten IDs, remove entries from the middle while keeping the first (root) and most recent ancestors.
  </FAQItem>

  <FAQItem question="Can unrelated emails accidentally end up in the same thread?">
    Yes. Gmail in particular will group messages with the same subject line into one thread, even without threading headers. This is a known behavior. Using unique, descriptive subject lines reduces the risk.
  </FAQItem>

  <FAQItem question="How should an AI agent set threading headers when sending programmatic replies?">
    Store the `messageId` and `references` from every received email. When replying, set `In-Reply-To` to the parent's `Message-ID` and `References` to the parent's `References` value plus the parent's `Message-ID`. LobsterMail's SDK surfaces these fields on every received email and accepts them on `send()`.
  </FAQItem>

  <FAQItem question="Does the subject line affect email threading?">
    In Gmail, yes. Gmail uses subject-line matching as a secondary threading signal, so changing the subject mid-conversation can split the thread. Outlook and Apple Mail rely more on `In-Reply-To` and `References` headers than subject lines.
  </FAQItem>

  <FAQItem question="Why do email threads sometimes break or split even when In-Reply-To is set correctly?">
    Common causes include malformed `Message-ID` values (missing angle brackets), overwriting `References` instead of appending to it, or modifying the subject line between messages. Any of these can cause at least one major client to split the thread.
  </FAQItem>

  <FAQItem question="What RFC standards define In-Reply-To and References behavior?">
    RFC 5322 (Internet Message Format) is the current standard. It superseded RFC 2822, which superseded RFC 822. All three define the same basic threading header behavior, with RFC 5322 being the authoritative reference.
  </FAQItem>

  <FAQItem question="How can I test that my threading headers work correctly before sending at scale?">
    Send a test message, reply to it programmatically with correct headers, then verify the thread displays correctly in Gmail, Outlook, and Apple Mail. Check that `Message-ID` values include angle brackets and a domain, that `References` chains are intact, and that subject lines are consistent.
  </FAQItem>

  <FAQItem question="Can someone spoof In-Reply-To to inject messages into an existing thread?">
    Yes. If a sender sets `In-Reply-To` to a valid `Message-ID` from an existing conversation, most email clients will display the spoofed message as part of that thread. Agents should validate that `In-Reply-To` values match `Message-ID` values they actually sent.
  </FAQItem>
