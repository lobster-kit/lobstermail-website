Himalaya is a solid CLI email client. It speaks IMAP, SMTP, outputs JSON, and plays well with scripts. If you're wiring up email for a personal workflow or a cron job on your laptop, it's a good pick. But the moment you try to configure Himalaya OAuth2 for something that runs unattended (say, an AI agent on a server), you hit a wall that the docs don't really prepare you for.

I spent a weekend getting Himalaya's OAuth2 flow working with Gmail and Outlook. Then I spent another weekend figuring out why it kept breaking in production. Here's what I learned, what the actual setup steps look like, and where the approach falls apart for automated systems.

## What Himalaya OAuth2 actually does

Himalaya supports OAuth2 as an alternative to app passwords for authenticating with Gmail, Outlook, and other providers. Instead of storing a plaintext password in your config file, you register an OAuth2 application with your email provider, get a client ID and secret, and let Himalaya handle the token exchange.

The auth method comes in two flavors: `xoauth2` and `oauthbearer`. Gmail uses `xoauth2`. Outlook supports both, but `xoauth2` is the safer bet. The difference is in how the access token gets encoded in the IMAP AUTH command. For most setups, `xoauth2` is what you want.

When Himalaya needs a token, it opens your default browser, redirects you to Google or Microsoft's consent screen, waits for the callback on a local port, captures the authorization code, exchanges it for an access token, and stores the result in your system keyring. It works. On a machine with a browser.

## How to configure Himalaya with OAuth2

Here's the step-by-step process, distilled from too many failed attempts:

1. Register an OAuth2 application with your email provider (Google Cloud Console for Gmail, Azure AD for Outlook) and note the client ID and client secret.
2. Set the redirect URI to `http://localhost:9090` in your provider's app registration (Himalaya listens on this port by default).
3. Open your Himalaya config file (usually `~/.config/himalaya/config.toml`) and add the TOML block with `backend.auth.type = "oauth2"`.
4. Set `backend.auth.auth-url` and `backend.auth.token-url` to your provider's authorization and token endpoints.
5. Store your client secret in the system keyring using `secret-tool` (Linux) or Keychain (macOS) under the key Himalaya expects.
6. Run `himalaya envelope list` to trigger the browser-based authorization flow and complete the consent screen.
7. Verify that tokens are stored in your keyring and that subsequent commands don't prompt for auth again.

That's the happy path. Let me show you what the config actually looks like for each provider.

## Gmail OAuth2 configuration

For Gmail, you need a Google Cloud project with the Gmail API enabled and an OAuth2 consent screen configured. The scopes you need are `https://mail.google.com/` for full IMAP/SMTP access. Google doesn't offer granular read-only IMAP scopes, so it's all or nothing.

```toml
[accounts.gmail]
email = "you@gmail.com"
backend.type = "imap"
backend.host = "imap.gmail.com"
backend.port = 993
backend.login = "you@gmail.com"
backend.auth.type = "oauth2"
backend.auth.method = "xoauth2"
backend.auth.client-id = "your-client-id.apps.googleusercontent.com"
backend.auth.client-secret.keyring = "gmail-oauth2-client-secret"
backend.auth.access-token.keyring = "gmail-oauth2-access-token"
backend.auth.refresh-token.keyring = "gmail-oauth2-refresh-token"
backend.auth.auth-url = "https://accounts.google.com/o/oauth2/v2/auth"
backend.auth.token-url = "https://oauth2.googleapis.com/token"
backend.auth.scopes = "https://mail.google.com/"
```

The SMTP side mirrors this structure but points at `smtp.gmail.com` on port 465. You can reuse the same OAuth2 tokens.

One thing Google doesn't make obvious: if your OAuth2 app is in "testing" mode, refresh tokens expire after 7 days. You need to publish the app (which requires a review process) or you'll wake up to broken auth every week.

## Outlook OAuth2 configuration

Microsoft's setup has its own quirks. You register an app in Azure AD, enable the `IMAP.AccessAsUser.All` and `SMTP.Send` permissions, and configure the redirect URI.

```toml
[accounts.outlook]
email = "you@outlook.com"
backend.type = "imap"
backend.host = "outlook.office365.com"
backend.port = 993
backend.login = "you@outlook.com"
backend.auth.type = "oauth2"
backend.auth.method = "xoauth2"
backend.auth.client-id = "your-azure-client-id"
backend.auth.client-secret.keyring = "outlook-oauth2-client-secret"
backend.auth.access-token.keyring = "outlook-oauth2-access-token"
backend.auth.refresh-token.keyring = "outlook-oauth2-refresh-token"
backend.auth.auth-url = "https://login.microsoftonline.com/common/oauth2/v2.0/authorize"
backend.auth.token-url = "https://login.microsoftonline.com/common/oauth2/v2.0/token"
backend.auth.pkce = true
backend.auth.scopes = "https://outlook.office.com/IMAP.AccessAsUser.All https://outlook.office.com/SMTP.Send"
```

Notice `backend.auth.pkce = true`. Microsoft requires PKCE (Proof Key for Code Exchange) for public clients. If you skip this, the token exchange fails with a vague error that doesn't mention PKCE at all. That detail alone cost me two hours.

## The redirect error everyone hits

The single most common Himalaya OAuth2 problem, based on GitHub issues and Reddit threads, is the "cannot wait for redirection" error. It happens when Himalaya can't bind to the local port to catch the OAuth2 callback.

Common causes:

- Another process is already using port 9090
- You're running Himalaya over SSH without port forwarding
- You're on a headless server with no browser at all
- A firewall is blocking localhost connections on that port

The fix for the first case is straightforward: kill whatever's on port 9090 or configure a different port. For SSH, you can forward the port with `ssh -L 9090:localhost:9090`. But the headless server case? That's where things get philosophical.

## Where this breaks for agents

OAuth2's browser redirect flow assumes a human is sitting at a computer, ready to click "Allow" in a browser window. That assumption holds for a developer checking personal email from the terminal. It does not hold for an AI agent running on a server somewhere.

When an OpenClaw agent needs to send a verification email or receive a signup confirmation, it can't open a browser. It can't click consent buttons. It can't complete a redirect flow. The agent needs email that works without human intervention at auth time.

Some people work around this by pre-authenticating Himalaya on a local machine, copying the keyring tokens to the server, and hoping the refresh token doesn't expire. This works until it doesn't. Gmail's 7-day expiry in testing mode, Microsoft's conditional access policies, or a simple token rotation can break the flow silently. The agent stops receiving email, and nobody notices until a workflow fails downstream.

There's a Reddit thread from an OpenClaw user who had Himalaya working perfectly from the command line but couldn't get their agent to find the tool at all. The OAuth2 config was correct. The tokens were valid. The problem was the gap between "works for a human in a shell" and "works for an agent in a runtime."

## A different approach for agent email

This is the core tension: OAuth2 was designed to protect human users from credential theft by keeping passwords out of third-party apps. It's a good security model for that purpose. But agents aren't third-party apps accessing a human's mailbox. Agents need their own inboxes, provisioned programmatically, with no browser flow in the loop.

LobsterMail takes a different approach. Instead of wrapping OAuth2 around an existing mailbox, the agent creates its own inbox with a single SDK call:

```typescript
import { LobsterMail } from '@lobsterkit/lobstermail';

const lm = await LobsterMail.create();
const inbox = await lm.createSmartInbox({ name: 'My Agent' });
const emails = await inbox.receive();
```

No client IDs, no redirect URIs, no keyring, no token refresh. The agent gets `my-agent@lobstermail.ai` and can send and receive immediately. The free tier includes 1,000 emails per month with no credit card required.

This isn't a replacement for Himalaya if you need to access your personal Gmail inbox from the terminal. Himalaya is great for that. But if your agent needs its own email address that works in headless environments without human auth ceremonies, the two tools solve fundamentally different problems.

## When to use which

Use Himalaya with OAuth2 when you're a human who wants to read personal or work email from the command line, when you're scripting against your own mailbox on a machine with a browser available, or when you need to access an existing inbox that already has mail in it.

Use agent-native email (like LobsterMail) when an AI agent needs its own address, when the system runs headless with no browser, when you can't guarantee a human will re-authenticate every time a token expires, or when you need the agent to self-provision inboxes without any manual setup.

The mistake I see people make is trying to shoehorn Himalaya into an agent pipeline because it's a good CLI tool. It is a good CLI tool. But "good CLI tool" and "good agent infrastructure" aren't the same thing, and OAuth2 is precisely where that difference becomes painful.

If you're building an agent that needs email and you don't want to babysit OAuth2 tokens, [the getting started guide](/docs/getting-started) covers the setup in about two minutes.

<FAQ>
  <FAQItem question="What is OAuth2 in the context of Himalaya CLI and why is it preferred over password authentication?">
    OAuth2 lets Himalaya authenticate with email providers using tokens instead of storing your password in a config file. It's preferred because Gmail and Outlook are deprecating app passwords, and OAuth2 tokens can be scoped to specific permissions and revoked independently.
  </FAQItem>

  <FAQItem question="How do I register an OAuth2 application to get a client ID and client secret for use with Himalaya?">
    For Gmail, create a project in Google Cloud Console, enable the Gmail API, and configure an OAuth2 consent screen. For Outlook, register an app in Azure AD. Both will give you a client ID and secret to use in Himalaya's TOML config.
  </FAQItem>

  <FAQItem question="What is the difference between the xoauth2 and oauthbearer auth methods in Himalaya?">
    Both are ways to pass an OAuth2 access token during IMAP authentication. `xoauth2` base64-encodes the username and token together, while `oauthbearer` uses a different encoding format. Gmail requires `xoauth2`. Outlook supports both, but `xoauth2` is more widely compatible.
  </FAQItem>

  <FAQItem question="What does backend.auth.pkce = true do in Himalaya's OAuth2 config?">
    PKCE (Proof Key for Code Exchange) adds an extra verification step to the OAuth2 flow to prevent authorization code interception. Microsoft requires it for public clients. If you're connecting to Outlook and skip this setting, the token exchange will fail.
  </FAQItem>

  <FAQItem question="Which OAuth2 scopes are required for IMAP and SMTP access in Gmail and Outlook?">
    Gmail requires `https://mail.google.com/` for both IMAP and SMTP. Outlook uses `https://outlook.office.com/IMAP.AccessAsUser.All` for reading and `https://outlook.office.com/SMTP.Send` for sending.
  </FAQItem>

  <FAQItem question="Why does Himalaya throw a 'cannot wait for redirection' error during OAuth2 setup?">
    Himalaya needs to listen on a local port (default 9090) to catch the OAuth2 callback. This error means something else is using that port, you're on a headless server with no browser, or a firewall is blocking the connection. Try killing other processes on port 9090 or forwarding it over SSH.
  </FAQItem>

  <FAQItem question="Can I use Himalaya OAuth2 in a headless or server environment without a browser redirect?">
    Not easily. The OAuth2 flow requires a browser to complete the consent screen. You can pre-authenticate on a local machine and copy tokens to the server, but they'll eventually expire and require manual re-authentication. For headless environments, agent-native email like [LobsterMail](/) avoids this problem entirely.
  </FAQItem>

  <FAQItem question="How do I refresh expired OAuth2 tokens in Himalaya automatically?">
    Himalaya handles token refresh automatically using the refresh token stored in your keyring, as long as the refresh token itself is still valid. Gmail refresh tokens expire after 7 days if your OAuth2 app is in testing mode. Publishing the app through Google's review process gives you long-lived refresh tokens.
  </FAQItem>

  <FAQItem question="How do I store my OAuth2 client secret and tokens in the system keyring with Himalaya?">
    On Linux, use `secret-tool store` with the keyring name you specified in your config (e.g., `gmail-oauth2-client-secret`). On macOS, Himalaya uses the system Keychain. The access and refresh tokens are stored automatically after the first successful authorization.
  </FAQItem>

  <FAQItem question="How do I enable OAuth2 support when installing Himalaya via Cargo?">
    Run `cargo install himalaya --features oauth2`. The OAuth2 feature is not included in the default build. Without this flag, the `oauth2` auth type in your config will be unrecognized.
  </FAQItem>

  <FAQItem question="How does Himalaya OAuth2 compare to app passwords for security?">
    OAuth2 tokens are scoped to specific permissions, expire automatically, and can be revoked without changing your account password. App passwords grant full account access and never expire unless manually revoked. OAuth2 is strictly better for security, which is why Google and Microsoft are pushing everyone toward it.
  </FAQItem>

  <FAQItem question="Is it possible to use Himalaya OAuth2 as a backend for AI agents or email automation?">
    Technically yes, but it requires working around the browser-based consent flow. Agents can't click "Allow" in a browser window. Pre-provisioned tokens work until they expire. For agent use cases, purpose-built infrastructure like [LobsterMail](/docs/getting-started) is designed to work without any human auth steps.
  </FAQItem>

  <FAQItem question="Is Himalaya OAuth2 compatible with Microsoft 365 modern authentication?">
    Yes. Microsoft 365 uses the same Azure AD OAuth2 endpoints as consumer Outlook. Set the `auth-url` and `token-url` to the Microsoft identity platform v2.0 endpoints and enable PKCE. Organizational accounts may have conditional access policies that restrict which apps can authenticate.
  </FAQItem>
</FAQ>
