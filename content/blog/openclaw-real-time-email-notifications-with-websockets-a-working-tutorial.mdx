Your agent fires a request to check for email. Nothing. Fires again. Nothing. Again. Nothing. Somewhere around the 40th empty response, an email finally lands, and your agent catches it on the next cycle. By then, 5 to 10 seconds have passed since delivery. That lag sounds trivial until it's a verification code with a 60-second expiry window, and the link has already expired by the time your agent sees it.

Polling is the default because it's simple. But simple comes with costs that compound fast when your OpenClaw agent is running real workflows against real services.

This tutorial covers a different approach: keeping a WebSocket connection open so LobsterMail pushes emails to your agent the moment they arrive.

## The real cost of polling

A polling loop running every 5 seconds generates 17,280 API requests per inbox per day. At 10-second intervals, that's still 8,640 requests. Multiply across multiple inboxes or multiple agents, and you're burning through rate limits on requests that almost always return empty.

But the bigger problem is latency. If your agent submitted a form and is waiting on a magic link, the gap between email delivery and your agent seeing it averages half the polling interval. For a 5-second loop, that's 2.5 seconds of dead time where the email exists but your agent doesn't know it.

For some workflows, 2.5 seconds is nothing. For others, it's the difference between catching a verification window and missing it. We covered this tradeoff in our [webhooks vs polling comparison](/blog/webhooks-vs-polling-email), but the short version is: if your agent needs to react to inbound email quickly, polling is always a compromise.

## How LobsterMail's WebSocket delivery works

When your agent creates an inbox with LobsterMail, the platform maintains a WebSocket connection for that inbox. The moment an inbound email is processed, LobsterMail pushes the full message object through the open connection. No polling interval. No wasted requests. Notification latency measured in milliseconds.

The SDK exposes this through `inbox.listen()`. You register a callback, and that callback fires once per incoming email with the complete message, including headers, body, and LobsterMail's built-in injection risk score.

```typescript
import { LobsterMail } from '@lobsterkit/lobstermail';

const lm = await LobsterMail.create();
const inbox = await lm.createSmartInbox({ name: 'openclaw-agent' });

console.log(`Inbox ready: ${inbox.address}`);

inbox.listen((email) => {
  console.log(`Got email from ${email.from}: ${email.subject}`);
  // your agent handles it here
});
```

That's the core pattern. `LobsterMail.create()` handles [agent self-signup](/blog/agent-self-signup-explained) automatically on first run, with no human dashboard step. Token gets stored at `~/.lobstermail/token` and reused on every subsequent call.

## Building a working OpenClaw integration

OpenClaw agents are event-driven, so the WebSocket pattern fits naturally. Register the email listener when your agent starts, handle each email as an event. Here's a minimal but complete setup:

```typescript
import { LobsterMail, Email } from '@lobsterkit/lobstermail';

async function startAgent() {
  const lm = await LobsterMail.create();
  const inbox = await lm.createSmartInbox({ name: 'my-openclaw-agent' });

  console.log(`Agent inbox: ${inbox.address}`);

  inbox.listen(async (email: Email) => {
    if (email.security.injectionRisk === 'high') {
      console.warn('Suspicious email, skipping');
      return;
    }

    await handleEmail(email);
  });
}

async function handleEmail(email: Email) {
  const body = email.text ?? email.html ?? '';

  const codeMatch = body.match(/\b\d{6}\b/);
  if (codeMatch) {
    console.log(`Verification code: ${codeMatch[0]}`);
    // continue the workflow
  }
}

startAgent();
```

A few things worth calling out here.

First, `createSmartInbox()` generates a human-readable address from the name you pass. If `my-openclaw-agent@lobstermail.ai` is already taken, it tries variations automatically. If you'd rather have a random address with zero collision risk, `createInbox()` gives you something like `lobster-xxxx@lobstermail.ai` immediately.

Second, that injection risk check on line 10 matters more than it might look. If your agent acts on email content from unknown senders, prompt injection is a real attack surface. We've written about the [security side of agent email](/blog/openclaw-agent-email-security) in detail, but the short version is: LobsterMail scores every inbound email and surfaces that score on the message object so you can gate on it without rolling your own detection logic.

Third, this works on the free tier. Your agent gets one inbox, 1,000 emails per month, no credit card required. If you need more inboxes or higher volume, the Builder plan at $9/month covers up to 10 inboxes and 5,000 emails/month.

## A concrete example: catching a verification code

Say your OpenClaw agent is registering for a new API service. It fills out the signup form using the inbox address and needs to catch the verification email to complete the flow. With polling at 5-second intervals, your agent sits idle for up to 5 seconds after the email lands. With the WebSocket approach, the typical round-trip from email sent to callback firing is under 900ms.

Here's what the waiting part looks like in practice:

```typescript
async function waitForVerificationCode(
  inbox: Inbox,
  timeout = 30_000
): Promise<string | null> {
  return new Promise((resolve) => {
    const timer = setTimeout(() => resolve(null), timeout);

    inbox.listen((email) => {
      const code = (email.text ?? '').match(/\b\d{6}\b/)?.[0];
      if (code) {
        clearTimeout(timer);
        resolve(code);
      }
    });
  });
}
```

Thirty-second timeout as a safety net. In practice, most services deliver the email within a few seconds, and your agent catches it in under a second after that. If the promise resolves to null, your agent can request a resend or exit cleanly. Either way, the outcome is known quickly rather than hanging in a polling loop hoping the next cycle catches it.

This same pattern works for any "wait for an email, then continue" workflow: password resets, two-factor codes, account confirmations, approval emails. The shape is always the same. Open a listener, match the content you're waiting for, resolve, and move on.

## Connection handling and edge cases

The SDK handles reconnection automatically. If the connection drops, it reconnects and delivers any emails that arrived during the gap. Your agent won't silently miss messages.

One caveat: WebSocket connections require a persistent process. If your OpenClaw instance runs in a serverless environment, the connection closes when the process exits. For those setups, consider the [webhook approach with HMAC verification](/blog/openclaw-inbound-email-webhook-setup-with-hmac-verification) instead. Both deliver in real time. WebSockets need a long-running process, webhooks need an HTTP endpoint.

For most OpenClaw installations running on a local machine or persistent server, WebSockets are simpler. No endpoint to expose, no signature verification to implement. The connection is outbound from your agent, so it works behind NATs and firewalls without port forwarding.

## What this doesn't cover

This tutorial covers inbound real-time notifications only. Outbound sending, threading, and reply handling are in the [SDK docs](/docs/getting-started). If your agent handles incoming messages from real users rather than service emails, read [why mainstream email isn't quite ready for agents](/blog/openclaw-mainstream-email-not-ready) first. The trust model is different.
