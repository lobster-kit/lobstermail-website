Most Python agent frameworks expect your agent to talk to APIs, call tools, and reason through multi-step tasks without human intervention. But the moment that agent needs to sign up for a service, catch a confirmation code, or send a follow-up email, the whole thing stalls. The agent can reason and plan, but it can't check its email because it doesn't have an email address.

Gmail won't work here. OAuth requires a human to open a browser, click Allow, and authorize the app. That's fine for consumer apps. For an agent running unattended on a server at 2am, it's a dead end. And even if you manage to wire up a personal inbox, now your agent is reading your private messages and sending from your identity. One prompt injection in a forwarded newsletter and your agent is replying to your boss as you.

LobsterMail gives your Python agent its own isolated inbox. No OAuth, no human signup, no shared credentials. The agent provisions the address itself and starts communicating. This tutorial walks through the full setup.

## Install the SDK

```bash
pip install lobstermail
```

On first run, the SDK creates a free account automatically and saves a token to `~/.lobstermail/token`. No dashboard, no email confirmation, no API key to generate first. If you're running in a stateless environment (Lambda, Cloud Run, a container), set `LOBSTERMAIL_API_KEY` in your environment variables and the SDK picks it up instead.

## Provision an inbox

```python
import asyncio
from lobstermail import LobsterMail

async def main():
    lm = await LobsterMail.create()
    inbox = await lm.create_smart_inbox(name="my-agent")
    print(inbox.address)  # my-agent@lobstermail.ai

asyncio.run(main())
```

`create_smart_inbox()` takes a human-readable name and handles collisions for you. If `my-agent@lobstermail.ai` is already taken, the SDK tries variations until it finds something available. You get a clean, readable address without writing any availability logic.

For throwaway tasks where the address only needs to exist for a few minutes, `create_inbox()` gives you a random address like `lobster-a4f2@lobstermail.ai`. Sign up for something, grab the code, delete the inbox. Done.

## Receive and read emails

```python
emails = await inbox.receive()
for email in emails:
    print(email.subject)
    print(email.from_address)
    print(email.body)
    print(email.injection_risk)  # 'low' | 'medium' | 'high'
```

Every email that arrives comes with an `injection_risk` score. This is specific to agent workloads, and you won't find it in SendGrid, Resend, or Mailgun. The score rates how likely the email body contains [prompt injection patterns](/blog/what-is-prompt-injection-in-email) designed to hijack your agent's behavior. A high score (roughly above 0.7) means the content has text written to look like system instructions, role overrides, or tool invocations hidden inside normal-looking prose.

If your agent passes email content into a model prompt (which it almost certainly does), treat high-risk messages differently. Quote the content explicitly rather than inserting it raw, or route it through a human review step. We covered the full taxonomy of injection categories in our [prompt injection security guide](/blog/agentmail-vs-lobstermail-comparing-prompt-injection-security-for-ai-agents).

## The verification code pattern

This is the scenario that breaks agent workflows most often. The agent registers for a third-party service, a confirmation email lands in an inbox the agent can't access, and the whole pipeline stalls. Here's a clean way to handle it:

```python
import asyncio
import re
from lobstermail import LobsterMail

async def register_and_verify(service_name: str, registration_fn):
    lm = await LobsterMail.create()
    inbox = await lm.create_smart_inbox(name=f"agent-{service_name.lower()}")
    print(f"Inbox ready: {inbox.address}")

    # Your registration function uses inbox.address as the email field
    await registration_fn(email=inbox.address)

    # Poll for the verification email, 60-second window
    for _ in range(12):
        emails = await inbox.receive()
        for email in emails:
            keywords = ("verify", "confirm", "activate", "welcome")
            if any(kw in email.subject.lower() for kw in keywords):
                match = re.search(r"\b(\d{4,8})\b", email.body)
                if match:
                    return match.group(1)
        await asyncio.sleep(5)

    raise TimeoutError(f"No verification email from {service_name} after 60s")
```

The 12-iteration loop with 5-second pauses gives a 60-second window. Most services deliver within 10 seconds, but this handles slower platforms without blocking for minutes. You can tune the interval to match your use case. The beauty of this pattern is that the agent creates a purpose-built inbox for each registration, uses it to grab the code, and can [delete it immediately after](/blog/lobstermail-sdk-quickstart-give-your-agent-an-email-in-5-minutes). No inbox clutter, no shared state between tasks.

## Wiring into LangChain, CrewAI, or AutoGen

The inbox object doesn't care what framework you're using. For [LangChain agents](/blog/langchain-email-integration), initialize the inbox before the agent loop, inject `inbox.address` wherever the agent needs contact info, and wrap `inbox.receive()` and `inbox.send()` as callable tools.

[CrewAI agents](/blog/crewai-agent-email) follow the same approach. Create the inbox before the crew starts, pass the address into the agent's backstory, and expose the inbox methods as tools the agent can call mid-task.

For [AutoGen](/blog/autogen-agent-email), you wrap each LobsterMail operation as an AutoGen `FunctionTool` and hand them to your `AssistantAgent`. The agent chains the calls itself: provision the inbox, send, check for replies.

The point is that the inbox is just an async Python object. It fits wherever async functions fit.

## Managing inbox lifecycle

```python
# List all inboxes on this account
inboxes = await lm.list_inboxes()
for i in inboxes:
    print(i.address, i.created_at)

# Delete when the task is done
await lm.delete_inbox(inbox.id)
```

If you're spinning up ephemeral inboxes for one-off tasks (sign up, grab code, move on), cleaning them up keeps your account tidy. The free tier supports one active inbox. Builder supports ten. For agents that need more, each inbox is isolated from the others, so a compromised address doesn't expose anything beyond its own messages.

## A note on async

The SDK is async-first because most modern agent frameworks are too. LangChain, CrewAI, AutoGen, smolagents — they all run async under the hood. For quick scripts, `asyncio.run()` at the top level handles it. In Jupyter notebooks, you can `await` directly in cells since Jupyter runs its own event loop.

If you hit "event loop already running" errors in Jupyter, one line of `nest_asyncio.apply()` clears it up. There's no official sync wrapper yet. For synchronous codebases, wrapping individual calls in `asyncio.run()` works fine for low-volume use cases.

  <FAQItem question="What's the difference between create_smart_inbox() and create_inbox()?">
    `create_smart_inbox(name="...")` generates a readable address and handles name collisions automatically. `create_inbox()` gives you a random address instantly. Use smart inboxes for persistent agents with a recognizable address, random inboxes for throwaway tasks like verification codes.
  </FAQItem>

  <FAQItem question="What does the injection_risk score measure?">
    It scores how likely an email body contains prompt injection attempts, including boundary manipulation, system prompt overrides, data exfiltration instructions, role hijacking, tool invocation tricks, and encoding exploits. Scores above 0.7 warrant extra caution before passing content to a model. See our [injection security guide](/blog/agentmail-vs-lobstermail-comparing-prompt-injection-security-for-ai-agents) for details.

  <FAQItem question="Can I use LobsterMail in a serverless function or Docker container?">

  <FAQItem question="Can an agent manage multiple inboxes?">
    Yes. Call `create_smart_inbox()` or `create_inbox()` multiple times. Each call returns a distinct inbox object. The free tier supports one active inbox; Builder supports up to ten. Use `lm.list_inboxes()` and `lm.delete_inbox(id)` to manage them.

  <FAQItem question="What happens if my LobsterMail token is exposed?">
    Rotate the token in your account settings. The old token is invalidated immediately. Because the token is scoped to your LobsterMail inboxes only, exposure doesn't compromise anything outside LobsterMail, unlike a Google OAuth token that might carry broader account access.
