You generated a fresh token from ClawHub Settings, pasted it into `clawhub login --token`, and got back a single word: `Unauthorized`. You tried again with a new token. Same result. You tried `--no-browser`. Still dead.

This has been one of the most persistent bugs in the OpenClaw ecosystem since late January 2026. Issues [#6933](https://github.com/openclaw/openclaw/issues/6933), [#5483](https://github.com/openclaw/openclaw/issues/5483), and [#72](https://github.com/openclaw/clawhub/issues/72) all describe the same core problem: tokens that look valid, were just created seconds ago, and still get rejected by the CLI. It's not you. But there are real fixes, and most of them take under five minutes.

## How to fix clawhub login --token unauthorized (step-by-step)

1. Regenerate a fresh token at `clawhub.ai/settings/cli/auth` (not the general API tokens page)
2. Confirm the token format starts with `clh_h_`, not `clh_` or `ghp_`
3. Test the token directly with curl before touching the CLI:
   ```bash
   curl -H "Authorization: Bearer clh_h_YOUR_TOKEN" https://www.clawhub.ai/api/auth/verify
4. Set the registry base URL to the canonical domain to prevent redirect header-dropping:
   ```bash
   clawhub config set registry https://www.clawhub.ai
   ```
5. Clear any cached credentials from previous sessions:
   ```bash
   rm -rf ~/.config/clawhub/auth.json
   ```
6. Run the login command again with the fresh token:
   ```bash
   clawhub login --token clh_h_YOUR_TOKEN
   ```
7. If you're on a headless server, add `--no-browser` to skip the OAuth fallback entirely

If step 3 returns a 200 response but step 6 still fails, you're almost certainly hitting the redirect bug. Step 4 is the fix.

## Why valid tokens get rejected

The root cause in most cases is an HTTP redirect that strips the `Authorization` header. When the CLI sends your token to `https://clawhub.ai/api/auth/verify`, some environments (particularly behind proxies or on certain Node.js versions) follow a 301 redirect to `https://www.clawhub.ai/api/auth/verify`. HTTP clients are supposed to drop the `Authorization` header on redirects to a different origin. Even though `clawhub.ai` and `www.clawhub.ai` serve the same app, they're technically different origins.

Your token is valid. The server never sees it.

This is why the curl test in step 3 matters. If you hit the canonical URL directly (`www.clawhub.ai`), no redirect happens, no header gets dropped, and the token works fine. Setting the registry base URL (step 4) tells the CLI to skip the redirect entirely.

## Token types and why they matter

ClawHub has two types of tokens, and using the wrong one is the second most common cause of the Unauthorized error.

**Browser OAuth tokens** are created when you run `clawhub login` and complete the browser flow. These tokens are tied to your browser session, scoped to your user account, and stored automatically by the CLI. They work for most operations but can't be copy-pasted between machines.

**Personal CLI tokens** start with `clh_h_` and are generated from Settings, under CLI Auth. These are the tokens you pass to `--token`. They carry full CLI permissions including skill publishing and deletion. The older `clh_` prefix (without the `_h_`) was used for read-only API access and doesn't have write permissions. If your token starts with just `clh_`, that's why `clawhub login` accepts it but subsequent operations like `clawhub skill delete` fail with Unauthorized.

GitHub OAuth tokens (`ghp_` prefix) don't work with ClawHub at all, despite both platforms using OAuth. They're completely separate authentication systems.

## Headless environments: VPS, Docker, CI/CD

If you're running the ClawHub CLI on a headless VPS, inside a Docker container, or in a CI/CD pipeline, the browser OAuth flow isn't an option. You need token auth to work.

The recommended approach:

```bash
# In your CI config or Dockerfile
export CLAWHUB_TOKEN=clh_h_YOUR_TOKEN
clawhub login --token "$CLAWHUB_TOKEN" --no-browser
Two things to watch for in automated environments. First, some CI runners reset the filesystem between steps, which means the credential cache at `~/.config/clawhub/auth.json` disappears. Either re-run `clawhub login` at the start of each job or set the `CLAWHUB_TOKEN` environment variable, which the CLI checks before looking at the config file.

Second, if your pipeline publishes skills to ClawHub and also runs email-triggered workflows, you're managing two separate token lifecycles. A ClawHub token expiring mid-run means your skill publish fails silently. There's no built-in token rotation in the ClawHub CLI yet, so the practical fix is to generate long-lived tokens and store them in your CI secret manager.

## Diagnosing 401 vs. 403 vs. connection errors

Not all "Unauthorized" errors are the same, even though the CLI prints the same message for several different failure modes.

A **401 Unauthorized** means the server received your request but the token was missing, malformed, or expired. This is the redirect bug or wrong-token-type issue described above.

A **403 Forbidden** means the token is valid but doesn't have permission for the operation you're attempting. This happens when you use a read-only `clh_` token for a write operation, or when the token was created by a different user than the skill owner.

A **connection error** that the CLI reports as "Unauthorized" is a known bug in `clawhub` npm versions prior to 2026.1.28. The CLI catches network timeouts and misreports them as auth failures. Updating to the latest version fixes this:

```bash
npm install -g clawhub@latest
```

You can distinguish these cases by testing with curl first. A 401 response body includes `{"error": "unauthorized"}`. A 403 includes `{"error": "forbidden", "reason": "insufficient_scope"}`. A timeout gives you no response body at all.

## Multi-account and token switching

If you work across multiple ClawHub accounts (personal projects plus a team account, for example), stale credentials are a common trap. The CLI stores one active token at a time in `~/.config/clawhub/auth.json`. Logging in with a new token overwrites the old one without warning.

There's no built-in profile switching. The workaround is to use environment variables:

```bash
# Personal account
CLAWHUB_TOKEN=clh_h_personal_token clawhub skill list

# Team account
CLAWHUB_TOKEN=clh_h_team_token clawhub skill publish
```

The `CLAWHUB_TOKEN` environment variable takes priority over the config file, so you can switch contexts without touching the stored credentials.

## When none of this works

If you've followed every step above and the CLI still returns Unauthorized, there are two remaining possibilities. The ClawHub auth service itself could be down (check [status.openclaw.ai](https://status.openclaw.ai)). Or your account could be flagged, which happens occasionally with new accounts that trigger abuse detection. In that case, reach out to the OpenClaw team on the community forum.

The good news: once you fix the registry URL and use the right token type, this problem stays fixed. It's a one-time configuration issue, not a recurring headache.
