Most agent frameworks give you ten ways to initialize an agent and zero guidance on which one to pick. LlamaIndex is no exception. If you've been staring at the docs trying to figure out whether you need `FunctionCallingAgent`, `ReActAgent`, or the now-deprecated `OpenAIAgent`, this is the article I wish I'd had.

`FunctionCallingAgent.from_tools` is the recommended way to create a tool-calling agent in LlamaIndex. It works across multiple LLM providers, handles parallel tool calls where supported, and gives you a clean interface for wiring up callable functions. Let's walk through it from scratch.

## How to use FunctionCallingAgent.from_tools in llama_index

Here's the quickest path from zero to a working agent:

1. **Install llama-index and your LLM provider package.**

```bash
pip install llama-index llama-index-llms-openai
2. **Import FunctionCallingAgent and FunctionTool.**

```
```python
from llama_index.core.agent import FunctionCallingAgent
from llama_index.core.tools import FunctionTool
3. **Define your tools as plain Python functions wrapped in FunctionTool.**

```
```python
def get_weather(city: str) -> str:
    """Get the current weather for a city."""
    return f"The weather in {city} is 72°F and sunny."

weather_tool = FunctionTool.from_defaults(fn=get_weather)
```

4. **Initialize your LLM.**

```python
from llama_index.llms.openai import OpenAI

llm = OpenAI(model="gpt-4o")
```

5. **Create the agent with `FunctionCallingAgent.from_tools`.**

```python
agent = FunctionCallingAgent.from_tools(
    tools=[weather_tool],
    llm=llm,
    verbose=True,
)

response = agent.chat("What's the weather in Montreal?")
print(response)
```

That's it. Five steps. The agent receives your message, decides whether to call `get_weather`, parses the result, and returns a natural language response.

## What FunctionCallingAgent actually does

When you call `FunctionCallingAgent.from_tools`, LlamaIndex creates an agent that delegates tool selection to the LLM's native function-calling API. This is different from `ReActAgent`, which uses prompt engineering (a reasoning-then-acting loop) to decide when to call tools.

The practical difference: `FunctionCallingAgent` is faster and more reliable for LLMs that natively support function calling (OpenAI, Anthropic, Bedrock). `ReActAgent` is more flexible and works with any LLM, including local models that don't have a function-calling API.

If your LLM supports function calling, use `FunctionCallingAgent`. If it doesn't, use `ReActAgent`. That's the decision tree.

## Defining tools properly

The quality of your agent depends on the quality of your tool definitions. LlamaIndex reads the function signature and docstring to generate the tool schema the LLM sees. Sloppy docstrings produce sloppy tool calls.

```python
def search_emails(query: str, max_results: int = 10) -> str:
    """Search the inbox for emails matching a query string.

    Args:
        query: Natural language search query (e.g. 'invoices from March')
        max_results: Maximum number of results to return. Defaults to 10.

    Returns:
        A formatted string listing matching email subjects and senders.
    """
    # your implementation here
    return results
```

A few things that matter here:

- **Type hints are required.** The LLM uses them to understand what to pass. If you skip type hints, the tool schema will be vague and the agent will hallucinate arguments.
- **The docstring is your tool's instruction manual.** Write it like you're explaining the function to someone who can't read the implementation. Include what the function does, what each parameter means, and what it returns.
- **Default values work.** LlamaIndex respects them, and the LLM will omit optional parameters when they're not needed.

You can register multiple tools at once:

```python
agent = FunctionCallingAgent.from_tools(
    tools=[weather_tool, search_tool, calendar_tool],
    llm=llm,
    verbose=True,
)
```

The LLM sees all available tools and picks which ones to call based on the user's message. If the message doesn't require any tools, the agent responds directly.

## Parallel tool calls

Some LLMs (GPT-4o, Claude) can call multiple tools in a single turn. If the user asks "What's the weather in Montreal and Toronto?", the LLM can fire off two `get_weather` calls simultaneously instead of making them sequentially.

In LlamaIndex, this is controlled by the `allow_parallel_tool_calls` parameter:

```python
agent = FunctionCallingAgent.from_tools(
    tools=[weather_tool],
    llm=llm,
    allow_parallel_tool_calls=True,
    verbose=True,
)
```

There's a catch. Not all LLM providers actually support this. OpenAI does. Anthropic does. Bedrock via `BedrockConverse` has an [open issue](https://github.com/run-llama/llama_index/issues/17606) where the flag doesn't reliably propagate. If you're on Bedrock and parallel calls aren't working, that's why. Check the LlamaIndex GitHub issues for your specific provider before assuming the flag is broken in your code.

The latency implications are real. Parallel calls can cut agent response time significantly when the user's query involves multiple independent lookups. For agents handling email triage (check sender reputation, scan attachments, classify priority), parallel execution turns a three-second waterfall into a one-second parallel batch.

## Adding a system prompt

You'll almost certainly want to give your agent a persona or instructions. Use the `system_prompt` parameter:

```python
agent = FunctionCallingAgent.from_tools(
    tools=[weather_tool, email_tool],
    llm=llm,
    system_prompt="You are a helpful assistant that manages email and calendar tasks. Always confirm before sending emails.",
    verbose=True,
)
```

The system prompt is sent as the first message in the conversation. It shapes how the agent interprets requests and when it decides to use tools. A good system prompt can be the difference between an agent that helpfully confirms before sending a client email and one that fires it off immediately.

## Handling errors in tool calls

This is where most tutorials stop and most production agents break. What happens when a tool raises an exception?

By default, `FunctionCallingAgent` catches the exception and passes the error message back to the LLM as the tool's response. The LLM then decides what to do, often telling the user something went wrong or trying a different approach.

If you want more control, wrap your tool functions with explicit error handling:

```python
def send_email(to: str, subject: str, body: str) -> str:
    """Send an email to the specified recipient."""
    try:
        result = email_client.send(to=to, subject=subject, body=body)
        return f"Email sent successfully to {to}. Message ID: {result.id}"
    except RateLimitError:
        return "Rate limit exceeded. Try again in 60 seconds."
    except InvalidAddressError as e:
        return f"Invalid email address: {e}. Ask the user to verify."
```

Return error information as a string rather than raising. This keeps the agent in control and lets the LLM reason about what went wrong. An unhandled exception becomes a generic error message the LLM can't do much with. A descriptive error string gives the agent something to work with.

## Persisting chat history

`FunctionCallingAgent` maintains an in-memory chat history by default. Each call to `agent.chat()` appends to the conversation. But if your process restarts, that memory is gone.

For persistent memory, pass a `ChatMemoryBuffer`:

```python
from llama_index.core.memory import ChatMemoryBuffer

memory = ChatMemoryBuffer.from_defaults(token_limit=4096)

agent = FunctionCallingAgent.from_tools(
    tools=[weather_tool],
    llm=llm,
    memory=memory,
    verbose=True,
)
```

You can serialize and restore the memory between sessions by saving the chat history to a database or file. This is essential for agents that handle multi-turn conversations over hours or days, like an email assistant that needs to remember what it discussed with the user yesterday.

## Migrating from the deprecated OpenAIAgent

If you're still using `OpenAIAgent`, the migration is straightforward. `FunctionCallingAgent` is its direct replacement with a provider-agnostic interface.

```python
# Old (deprecated)
from llama_index.agent.openai import OpenAIAgent
agent = OpenAIAgent.from_tools(tools, llm=llm)

# New
from llama_index.core.agent import FunctionCallingAgent
agent = FunctionCallingAgent.from_tools(tools, llm=llm)
```

The API is nearly identical. The main difference is that `FunctionCallingAgent` lives in `llama_index.core.agent` instead of a provider-specific package, and it works with any LLM that supports function calling. Your tool definitions, system prompts, and chat logic don't need to change.

## Connecting agents to real infrastructure

A tool-calling agent is only as useful as the tools you give it. Most tutorials show toy examples (weather lookups, calculators). Real agents need real infrastructure.

Email is one of the most common capabilities agents need. Signing up for services, reading verification codes, sending follow-ups. If you're building a `FunctionCallingAgent` that handles email, the tool implementation matters. You can wrap any email API as a `FunctionTool`. The agent calls the function, the function hits the API, and the result comes back as a string the LLM can reason about.

For agents that need to self-provision their own inboxes without human setup, [LobsterMail](/) handles that. Your agent creates an inbox, receives mail, and sends responses through simple function calls, which map naturally to the `FunctionTool` pattern in LlamaIndex.

## What to do next

Start with one tool. Get the agent working end-to-end with `verbose=True` so you can see every tool call and response. Then add tools incrementally. The most common mistake is wiring up ten tools at once and wondering why the agent picks the wrong one. Start small, test each tool in isolation, and expand from there.

If you're building an agent that needs email capabilities, the [LobsterMail getting started guide](/docs/getting-started) shows how to set that up in under five minutes.
