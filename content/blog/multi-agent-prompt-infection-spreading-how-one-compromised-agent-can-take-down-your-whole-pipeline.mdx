In October 2024, researchers from the Technion and Intuit published a paper called "Prompt Infection." It described something most people building multi-agent systems hadn't considered: a compromised agent doesn't just fail. It recruits. One infected agent rewrites the instructions passed to the next agent in the chain, which rewrites instructions for the next, and so on. The infection spreads like a biological virus through shared context windows.

By January 2026, OWASP listed prompt injection as the #1 critical vulnerability in their Top 10 for LLM Applications. It appeared in over 73% of production AI deployments they assessed. And multi-agent prompt infection spreading is the version of this problem that keeps scaling with your architecture.

## What is multi-agent prompt infection?

Multi-agent prompt infection is an attack where a compromised AI agent embeds malicious instructions into its output, causing downstream agents that consume that output to become compromised themselves. Unlike standard prompt injection (where a human crafts a malicious input for a single model), prompt infection spreads autonomously from agent to agent through their normal communication channels. It's dangerous because it requires no direct access to the target agents and can propagate through an entire system from a single entry point.

The original research demonstrated two variants: non-replicating infections, where the malicious payload executes but doesn't spread further, and self-replicating infections, where each compromised agent embeds the infection into its own output. The self-replicating version is the one that earned comparisons to biological viruses and computer worms.

## How infection moves through an agent network

Picture a typical email processing pipeline. Agent A monitors an inbox, summarizes new messages, and passes summaries to Agent B. Agent B extracts action items and routes them to Agent C, which books calendar events or drafts replies.

An attacker sends a carefully crafted email to the monitored inbox. The email body contains hidden instructions (maybe in white text, maybe in a Unicode trick, maybe just phrased conversationally enough that the summarizer treats it as content). Agent A processes the email and produces a "summary" that now contains the injected instructions. Agent B reads that summary, follows the embedded instructions, and its output to Agent C carries the infection forward.

At no point did the attacker need access to Agent B or C. They sent one email. The agents did the rest.

The researchers modeled this propagation using a logistic growth curve. In smaller networks (5-10 agents), infection saturated the population within a handful of interaction cycles. In larger networks, the absolute number of compromised agents grew faster because there were more communication pathways. A network of 50 agents with regular inter-agent messaging could reach full compromise in under 20 interaction rounds with a self-replicating payload.

## Why email is the most exposed attack surface

Most multi-agent infection research focuses on agents sharing context through function calls or message queues. But email is worse for three reasons.

First, email is an open ingress channel. Anyone on the internet can send your agent a message. There's no authentication requirement for inbound mail. Compare that to an internal API where at least you control who can call it.

Second, email content is rich and unstructured. HTML formatting, attachments, forwarded threads, signature blocks. Every layer is a place to hide injected instructions. An attacker can embed a payload in an HTML comment that a human would never see but an LLM happily processes.

Third, email workflows tend to be long chains. Receive, summarize, extract, respond, forward, schedule. Each step is an agent consuming the previous agent's output. That's exactly the topology where prompt infection thrives.

This is the specific gap that most security guidance misses. Papers talk about "agent communication channels" abstractly. In practice, for most production agent deployments, the primary external communication channel is email. Hardening that transport layer matters more than any amount of prompt engineering on the agents themselves.

## What a compromised agent can actually do

The damage depends on what tools the infected agent has access to. The Prompt Infection paper categorized outcomes into two buckets: content manipulation and tool misuse.

Content manipulation means the agent produces wrong outputs. Summaries that omit key details, translations that change meaning, analyses that reach the opposite conclusion. Subtle enough that a human reviewer might not catch it.

Tool misuse is more direct. If a compromised agent has access to email sending, it can exfiltrate data by mailing it to an external address. If it has file system access, it can read or modify documents. If it has calendar access, it can create events with phishing links. The infection doesn't just spread between agents. It weaponizes whatever capabilities those agents already have.

In an email pipeline, an infected agent with send permissions could forward every inbound message to an attacker-controlled address. Or it could modify outbound replies to include phishing links. The agent's legitimate access becomes the attack surface.

## Why "just add guardrails" doesn't solve this

A common response is to add safety filters or system prompt reinforcement. The research tested this. Standard safety training and system prompts reduced infection rates but didn't eliminate them. Self-replicating infections were particularly resilient because they mutated slightly with each propagation step, evading pattern-matching defenses.

The researchers proposed a technique called LLM Tagging, where each agent signs its output with a cryptographic tag that downstream agents verify. This helped, but it requires every agent in the system to implement the protocol. One untagged agent breaks the chain.

The deeper problem is architectural. If your agents consume each other's natural language output as instructions, you've built a system where data and code exist in the same channel. That's the same design flaw that gave us SQL injection in the 2000s. Guardrails are input validation. They help, but they don't fix the underlying architecture.

## Infrastructure-level defenses that actually work

The most effective mitigations operate below the agent layer, at the transport and infrastructure level.

**Sender authentication at the envelope level.** SPF, DKIM, and DMARC validation on every inbound message before it reaches any agent. This doesn't prevent all injection (a legitimate sender can still craft malicious content), but it eliminates spoofed sources entirely.

**Content sandboxing.** Strip HTML, remove hidden elements, normalize Unicode, and extract plain text before passing email content to an agent. Most injection payloads rely on formatting tricks that disappear when you reduce the content to clean plaintext.

**Injection risk scoring.** Analyze inbound content for patterns associated with prompt injection before the agent processes it. Flag or quarantine messages that score above a threshold. This is exactly what LobsterMail's security layer does: every email comes back with injection risk metadata so your agent (or your orchestration logic) can decide whether to process it.

**Capability isolation.** Don't give every agent in your pipeline full tool access. The summarization agent doesn't need send permissions. The calendar agent doesn't need file access. Limiting blast radius per agent is basic security hygiene, but it's especially important when infection can jump between agents.

**Envelope separation.** Keep the email metadata (sender, recipient, headers, authentication results) in a separate data structure from the email content. Pass the content to the LLM, but make routing and authentication decisions based on the envelope, which the LLM never touches.

## Translating academic risk into operational terms

The logistic growth model from the research has practical implications. If your email pipeline has five agents in sequence, a self-replicating infection can compromise all five within the first interaction cycle. That's not a theoretical risk over thousands of messages. That's the first malicious email.

For teams running agent-based email workflows in regulated industries (finance, healthcare, legal), this creates liability exposure. If a compromised agent sends incorrect information to a client, or exfiltrates protected data through an outbound email, the regulatory consequences land on you, not on the LLM provider.

The mitigation path isn't complicated, but it does require treating email infrastructure as a security boundary rather than a commodity utility. Your email layer needs to validate senders, score content for injection risk, and enforce per-agent capability limits before any LLM processes anything.
