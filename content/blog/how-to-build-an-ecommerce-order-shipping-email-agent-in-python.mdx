Most ecommerce shipping notification pipelines look the same: a webhook fires from Shopify, a Zapier step sends a templated email through SendGrid, and the customer gets a generic "Your order has shipped!" message. It works. But there's zero intelligence in it. The email doesn't mention the weather delay affecting FedEx Ground this week. It doesn't adjust tone for a first-time buyer versus a repeat customer. And when something goes wrong (a carrier exception, a failed delivery attempt), silence.

An AI agent can do better. Give it access to a shipping API and its own email address, and it handles the full loop: poll carrier status, detect changes, compose a context-aware message, send it from a dedicated inbox. No templates. No Zapier. Here's how to build one in Python.

## How to build an ecommerce order shipping email agent in Python

1. Install the LobsterMail SDK and a shipping API client (Shippo, EasyPost, or KeyDelivery)
2. Let the agent provision its own email inbox with `LobsterMail.create()`
3. Fetch order and shipment objects from your shipping API
4. Set up a status-polling loop that detects carrier status changes
5. Map each status change to a personalized email composed by the agent
6. Send the email through the agent's inbox and log it to prevent duplicates
7. Handle edge cases: failed deliveries, carrier delays, and retry logic

Each step below walks through the code.

## Give the agent its own inbox

The first architectural decision is where emails come from. Traditional setups use SES or SendGrid with a shared sending domain. That works for batch marketing blasts. For an autonomous agent that sends individually, you want inbox isolation: the agent owns its address and manages its own reputation.

```bash
## Connect to a shipping API
```

Shippo, EasyPost, and KeyDelivery all return similar tracking objects. Here's a minimal function that fetches the current status for a tracking number:

```python
import shippo
import os

shippo.config.api_key = os.environ["SHIPPO_API_KEY"]

def get_tracking_status(tracking_number: str, carrier: str) -> dict:
    tracking = shippo.Tracking.get_status(carrier, tracking_number)
    return {
        "status": tracking.tracking_status.status,
        "detail": tracking.tracking_status.status_detail,
        "location": tracking.tracking_status.location,
        "eta": tracking.eta,
    }
```

The important fields are the status code (`TRANSIT`, `DELIVERED`, `FAILURE`, `RETURNED`) and the human-readable detail. Your agent needs both to compose a useful email.

## Detect status changes and send emails

This is the core loop. The agent polls the shipping API, compares the current status against what it last recorded, and fires off an email when something changes.

```python
import json
import asyncio
from pathlib import Path

STATE_FILE = Path("shipping_state.json")

def load_state() -> dict:
    return json.loads(STATE_FILE.read_text()) if STATE_FILE.exists() else {}

def save_state(state: dict):
    STATE_FILE.write_text(json.dumps(state))

async def check_and_notify(orders: list[dict], inbox):
    state = load_state()

    for order in orders:
        tracking = get_tracking_status(order["tracking_number"], order["carrier"])
        last_status = state.get(order["order_id"])

        if tracking["status"] != last_status:
            body = compose_shipping_email(order, tracking)
            await inbox.send(
                to=order["customer_email"],
                subject=f"Update on your order #{order['order_id']}",
                body=body,
            )
            state[order["order_id"]] = tracking["status"]

    save_state(state)
```

The state file is the guard against duplicate sends. If the agent restarts mid-loop, it picks up where it left off without re-emailing customers who already got their update. For production, swap the JSON file for Redis or Postgres. The logic stays identical.

This is an agent-driven approach rather than a webhook-driven one. Instead of depending on carrier webhooks (which vary wildly in reliability across providers), your agent actively checks on whatever schedule you set. We compared both patterns in [webhooks vs polling: how your agent should receive emails](/blog/webhooks-vs-polling-email). Fifteen-second intervals for express shipments, five minutes for ground.

## Handle what goes wrong

Carrier exceptions happen. FedEx marks a package as "delivery exception" and goes silent for 48 hours. Your agent should anticipate this.

For failed delivery attempts, send a proactive email with redelivery options before the customer opens a WISMO ticket. For carrier delays, acknowledge the delay and provide a revised ETA, but don't send the same "still in transit" message every 15 minutes. The state file prevents that automatically, since the status code hasn't actually changed. For address validation errors, flag the order for human review and send the customer a polite confirmation request.

If you're processing thousands of orders per day, the Builder tier at $9/month gives you 5,000 emails and up to 10 inboxes. Run separate inboxes for different notification types: shipping confirmations, delivery alerts, exception notices. That separation keeps your [agent's email organized](/blog/what-agents-do-with-email) and makes sender reputation easier to manage.

Five minutes is a reasonable default. Customers don't need second-by-second updates, and most carriers refresh status every 30 minutes to a few hours. If you need faster notification for specific events like delivery confirmation, layer in a carrier webhook alongside the polling loop and deduplicate with the same state store.
