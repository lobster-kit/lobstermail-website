In January 2026, OWASP published its updated Top 10 for LLM Applications. Prompt injection claimed the number one spot, showing up in over 73% of production AI deployments assessed during the review period. That's not a theoretical concern. That's the majority of shipping systems.

But the version of prompt injection most people worry about (a user typing "ignore previous instructions" into a chatbox) isn't even close to the scariest variant. The real problem is what happens when agents talk to each other. When a malicious prompt lands in one agent, replicates itself into that agent's output, and rides downstream into every agent connected to it. Researchers call this "prompt infection." It behaves less like a hack and more like a biological virus.

## What is multi-agent prompt infection?

Multi-agent prompt infection is a self-replicating attack where a malicious prompt injected into one LLM agent autonomously propagates to other connected agents, coordinating them to exfiltrate data, spread disinformation, or execute harmful tool calls. It behaves like a computer worm inside AI pipelines, spreading without any further human involvement after the initial injection.

The term was formalized in an October 2024 paper from researchers who demonstrated it across multi-agent populations. Their key finding: self-replicating infections dramatically outperformed non-replicating ones. And larger agent populations didn't slow the spread. They accelerated it, following a logistic growth curve similar to biological epidemics.

## How infection actually moves between agents

Traditional prompt injection is a one-shot attack. You trick one model into doing something it shouldn't. Prompt infection is different because the compromised agent doesn't just misbehave. It embeds the malicious instruction into its own output, so the next agent that reads that output gets infected too.

Here's a simplified chain:

1. Agent A receives an email containing a hidden instruction buried in the body text.
2. Agent A processes the email. The instruction tells it to include a specific phrase in all future outputs.
3. Agent A sends a summary to Agent B. The summary contains the replicated instruction.
4. Agent B reads the summary, gets infected, and passes the instruction along to Agent C.
5. The infection spreads outward through every connected agent.

The scary part isn't step 1. It's steps 3 through 5, where no human is involved and no external attacker needs to do anything else. The initial payload does the rest.

This is what makes the "AI worm" label accurate. Like Morris Worm or WannaCry, the attack is self-propagating. Unlike traditional malware, it doesn't exploit a software vulnerability. It exploits the fact that LLMs can't reliably distinguish instructions from data.

## Why email is the highest-risk vector

Most discussions about multi-agent prompt infection spreading focus on API calls between agents or shared memory stores. But email is arguably the most dangerous channel, for three reasons.

**Email is unsolicited by design.** Your agent's inbox receives messages from anyone. There's no allowlist, no handshake, no mutual authentication before content arrives. Every inbound email is untrusted input that your agent will likely read, summarize, or act on.

**Email content is rich and unstructured.** HTML bodies, nested quotes, forwarded threads, attachments, headers. There are dozens of places to hide a malicious instruction where a human reviewer wouldn't see it but an LLM parser would process it.

**Email workflows are chained.** An agent that reads email rarely just reads it. It extracts data, updates a CRM, drafts a reply, forwards a summary to another agent, triggers a calendar booking. Each downstream action is a new infection opportunity.

Consider a common pattern: Agent A monitors a shared inbox, extracts action items, and routes them to specialized agents (Agent B handles scheduling, Agent C handles billing). A single poisoned email to that inbox now has a direct path to every agent in the system.

## Self-replicating vs. non-replicating infections

The research draws a clear line between these two categories, and the difference matters for defense planning.

Non-replicating injection is bad but contained. A malicious prompt tricks one agent into one bad action: leaking data, calling the wrong API, returning a wrong answer. The damage stops there.

Self-replicating infection includes a propagation instruction. The payload tells the infected agent to reproduce the malicious prompt in its outputs. This turns every compromised agent into a carrier. The infection follows a logistic growth curve: slow at first, then exponential through the middle of the population, then tapering as it runs out of uninfected agents.

In the researchers' experiments, self-replicating infections compromised the majority of agents in a system within a handful of communication rounds. Non-replicating infections affected only the directly targeted agent.

## What a defense actually looks like

There's no single fix. But there are concrete strategies that reduce the blast radius, especially at the email layer.

### Input labeling and message provenance

The most effective defense the research identified is called LLM Tagging: wrapping external content in explicit markers so the model can distinguish data from instructions. Something like `[EXTERNAL_EMAIL_CONTENT]...[/EXTERNAL_EMAIL_CONTENT]` around every inbound message body.

This isn't foolproof. Models can still be tricked. But it raised the bar significantly in testing.

LobsterMail's security layer does something similar. Every inbound email gets an injection risk score before your agent ever sees the content. High-risk messages get flagged with security metadata so your agent can decide how to handle them, rather than blindly processing everything.

### Architecture matters: isolation and topology

Hub-and-spoke architectures (one central agent routing to specialists) create single points of failure but also single points of defense. If the hub agent sanitizes inputs before routing, it can break the chain.

Mesh topologies, where every agent talks to every other agent, are the worst case. Infection spreads through every path simultaneously, and there's no chokepoint to filter at.

Sequential pipelines fall somewhere in between. Infection can only move forward, not laterally, which gives you more time to detect anomalies in output quality.

If you're building a multi-agent email workflow, think about where your sanitization boundaries are. The agent that first reads an email should be isolated from the agents that take action on it. Process the content in a sandbox, extract only structured data, and pass that structured data (not raw email content) to downstream agents.

### Monitoring and early detection

Prompt infections change agent behavior in measurable ways. Watch for:

- Sudden changes in output length or format from an agent that was previously consistent
- Unexpected tool calls, especially data exfiltration patterns like new outbound emails to unknown addresses
- Agents including strange phrasing in their outputs that echoes across multiple agents in the pipeline

These signals won't catch everything. But they'll catch the obvious cases, which, based on current research, is most of them. Today's prompt infections aren't subtle. They're effective because nobody is looking.

## The uncomfortable truth

We're building systems where autonomous agents read untrusted input, make decisions, and pass information to other autonomous agents. That's the whole point of multi-agent architectures. It's also exactly the structure that makes prompt infection so effective.

The fix isn't to stop building multi-agent systems. It's to treat every message boundary between agents the same way we treat network boundaries in traditional security: as a potential infection vector that needs inspection, labeling, and monitoring.

For email specifically, that means your agent shouldn't be parsing raw email bodies and dumping them into prompts. Use an email provider that scores and labels content before your agent processes it. LobsterMail does this at the infrastructure level, so your agent gets security metadata alongside every message.

If you're running multi-agent email workflows, start with the architecture. Isolate your email-reading agents from your action-taking agents. Label external content explicitly. Monitor for behavioral drift. And assume that every inbound email is a potential infection vector, because it is.
