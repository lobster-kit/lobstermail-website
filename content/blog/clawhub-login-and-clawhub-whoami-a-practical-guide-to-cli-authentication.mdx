You installed ClawHub, ran `clawhub search` to find a skill, and got hit with a `401 Unauthorized`. The CLI isn't broken. You just haven't authenticated yet.

ClawHub is the skill directory for OpenClaw. It lets you discover, install, update, and publish skills from the command line. But most operations require a valid session, and the CLI has multiple ways to authenticate depending on your situation. If you're sitting at a laptop with a browser, one command gets you in. If you're running an agent in a headless environment (a server, a CI pipeline, a container), you need a different approach.

This guide covers every authentication path, what each command actually does under the hood, and how to troubleshoot when things go wrong.

## ClawHub login and whoami: quick reference

Here's the short version if you already know what you're doing:

1. **Browser login** — run `clawhub login`, a browser window opens for OAuth authorization
2. **Token login** — run `clawhub login --token <token>` and paste your API token
3. **Verify session** , run `clawhub whoami` to print the authenticated user handle
4. **Logout** , run `clawhub logout` to clear stored credentials
5. **Manual auth** , set `export CLAWHUB_AUTH='your-token'` for headless or CI environments

If you need more detail on any of these, keep reading.

## Logging in with the browser flow

The default `clawhub login` command opens your system browser and walks you through an OAuth authorization flow. You approve access, the CLI receives a token, and it stores that token locally for future commands.

```bash
clawhub login
After the browser redirects back, you should see a confirmation message in your terminal. From that point, every `clawhub` command uses the stored token automatically.

This is the right choice when you're working on your own machine, installing skills interactively, or setting things up for the first time. It's the fastest path from zero to authenticated.

One thing to know: the browser flow won't work if there's no display server available. SSH sessions, Docker containers, and CI runners all fall into this category. For those, you need token-based auth.

## Logging in with an API token

If you can't open a browser (or don't want to), pass a token directly:

```bash
clawhub login --token <your-api-token>
This stores the token in the same local credential file as the browser flow. Every subsequent command uses it, same as before.

Where do you get the token? Generate one from your ClawHub account settings in the web dashboard. You can create multiple tokens with different labels so you know which one is used where. This matters when you're running agents across several environments and need to revoke access to one without breaking the others.

For automated setups where you don't want to run `clawhub login` at all, set the `CLAWHUB_AUTH` environment variable:

```bash
export CLAWHUB_AUTH="your-api-token-here"
```

When `CLAWHUB_AUTH` is set, the CLI uses it directly, skipping the local credential file entirely. This is the cleanest approach for containers and CI pipelines because there's no stored state to manage. The token lives in the environment and disappears when the process ends.

## Verifying your session with clawhub whoami

The `clawhub whoami` command prints the handle of the currently authenticated user:

```bash
clawhub whoami
# → @your-username
```

It sends the stored token to ClawHub's API and returns the associated account. This is a server-side check, not a local file read. If the token has been revoked, expired, or is malformed, `clawhub whoami` will fail with an error rather than showing stale data.

Use it to confirm you're logged into the right account (especially if you manage multiple accounts), to verify that a token is still valid before running a long operation, or as a health check in automated scripts.

A common pattern in agent startup scripts:

```bash
if ! clawhub whoami > /dev/null 2>&1; then
  echo "ClawHub auth failed, re-authenticating..."
  clawhub login --token "$CLAWHUB_TOKEN"
fi
```

This tries `clawhub whoami` silently. If it fails for any reason, the script re-authenticates with a stored token. Simple, and it prevents your agent from running unauthenticated commands and hitting 401 errors halfway through a workflow.

## Logging out and re-authenticating

`clawhub logout` clears the locally stored credentials:

```bash
clawhub logout
```

This removes the token from your machine. It does not revoke the token on the server side. If someone copied your token before you logged out, they could still use it until you revoke it from the web dashboard.

The distinction matters. `clawhub logout` is for switching accounts or cleaning up a local session. Revoking a compromised token is a separate action you do in the ClawHub web UI.

To switch accounts, the sequence is:

```bash
clawhub logout
clawhub login  # or clawhub login --token <different-token>
clawhub whoami  # confirm you're on the right account
```

## Using ClawHub auth in agent and CI environments

If you're building agents that install or publish skills as part of their workflow, interactive browser login isn't an option. Here's what works:

**Environment variable approach** (recommended for most agent setups):

```bash
# In your .env or CI secrets
CLAWHUB_AUTH=lm_your_token_here
```

The CLI picks this up automatically. No login command needed. Your agent starts, runs `clawhub install some-skill`, and auth happens transparently.

**Token file approach** (useful for long-running agents on a single machine):

```bash
# One-time setup
clawhub login --token "$CLAWHUB_TOKEN"
# Token is now stored locally and persists across restarts
```

**Per-command token** (useful for one-off operations):

Some CI setups prefer passing the token per command rather than setting a global variable. You can combine environment variables with individual commands:

```bash
CLAWHUB_AUTH="$SECRET_TOKEN" clawhub publish ./my-skill
```

This sets the variable only for that single command and doesn't persist it anywhere.

For agents that run for days or weeks, token expiry is a real concern. ClawHub tokens don't expire by default as of early 2026, but if your organization enforces token rotation policies, your agent needs to handle re-authentication. The `clawhub whoami` health check pattern from earlier is the simplest way to detect an expired token and recover.

### A note on security

Don't commit tokens to version control. Don't log them to stdout. Don't store `CLAWHUB_AUTH` in a Dockerfile's `ENV` instruction (it gets baked into the image layer). Use secrets management: CI secret variables, encrypted .env files, or a vault service.

If you're running agents that also need email (for receiving verification codes, sending notifications, or communicating with users), [LobsterMail](/) handles inbox provisioning the same way ClawHub handles skill auth. Your agent creates its own inbox with a single SDK call, no manual setup required. The pattern of "agent self-provisions what it needs" applies to both skills and email.

## Common authentication errors and fixes

**401 Unauthorized on any command:** Your token is missing, expired, or revoked. Run `clawhub whoami` to confirm. If it fails, re-authenticate with `clawhub login` or `clawhub login --token`.

**"Could not open browser" during clawhub login:** You're in a headless environment. Use `clawhub login --token <token>` or set `CLAWHUB_AUTH` instead.

**clawhub whoami shows the wrong account:** You have a locally stored token from a previous session. Run `clawhub logout`, then log in with the correct account.

**"Permission denied" when publishing a skill:** Your token might not have publish scope. Generate a new token with the right permissions from the ClawHub dashboard. Not all tokens are created equal; a read-only token won't let you push skills to the registry.

**CLAWHUB_AUTH is set but the CLI ignores it:** Check for typos in the variable name. Check that the token value doesn't include extra whitespace or quotes. Run `echo $CLAWHUB_AUTH` to see what the shell actually has.

## What the ClawHub CLI is actually for

If you landed here looking for auth commands, you might still be getting oriented. ClawHub CLI is the tool for managing OpenClaw skills. The core workflow:

- `clawhub search <query>` finds skills in the registry
- `clawhub install <slug>` downloads a skill to your local setup
- `clawhub list` shows what you've installed
- `clawhub update --all` upgrades everything to the latest versions
- `clawhub inspect <slug>` previews a skill's definition without installing it
- `clawhub publish` pushes your own skill to the registry

All of these (except `search` and `inspect` for public skills) require authentication. Which is why sorting out `clawhub login` is usually the first thing you do after installing the CLI.

To install ClawHub itself, check the [official repo](https://github.com/openclaw/clawhub) for platform-specific instructions. It's a single binary on most systems.

## The pattern that keeps repeating

ClawHub auth follows a pattern you'll see in every modern CLI tool: OAuth browser flow for interactive use, API tokens for automation, environment variables for ephemeral environments. If you've used `gh auth login` for GitHub or `npm login` for the npm registry, this will feel familiar.

The main thing to remember: `clawhub login` gets you in, `clawhub whoami` confirms you're in, and `clawhub logout` gets you out. For agents and pipelines, skip the login command entirely and set `CLAWHUB_AUTH` in your environment.
