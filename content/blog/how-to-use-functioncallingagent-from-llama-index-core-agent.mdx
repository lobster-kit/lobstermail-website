If you've been building with LlamaIndex, you've probably seen this import floating around in tutorials and GitHub discussions:

```python
from llama_index.core.agent import FunctionCallingAgent
It looks simple enough. But the docs split responsibilities across `FunctionCallingAgent`, `FunctionCallingAgentWorker`, and `AgentRunner`, and most tutorials skip the parts that matter when you're wiring an agent to do real things in the world. Things like sending emails, calling external APIs, or reacting to incoming messages.

Here's what FunctionCallingAgent actually is, how to set it up, and how to extend it with tools that do more than query a vector index.

## What FunctionCallingAgent is (and isn't)

FunctionCallingAgent is LlamaIndex's high-level wrapper for agents that use an LLM's native function-calling (tool-use) capability. Instead of the ReAct pattern, where the model reasons in a text loop of "Thought → Action → Observation," FunctionCallingAgent relies on the model itself to select and invoke tools through structured function calls. OpenAI, Anthropic, and AWS Bedrock models all support this natively.

The practical difference: ReAct agents parse their own reasoning text to decide which tool to call. FunctionCallingAgent lets the model return a structured tool invocation directly. This tends to be more reliable with models that were trained for it, and less reliable with models that weren't.

## How to use FunctionCallingAgent from llama_index.core.agent

The setup follows five steps:

1. Install `llama-index-core` and your LLM provider package
2. Import `FunctionCallingAgent` from `llama_index.core.agent`
3. Define your tools as `FunctionTool` or `QueryEngineTool`
4. Initialize the agent with `FunctionCallingAgent.from_tools()`
5. Call `agent.chat()` to start interacting

Here's the minimal working example:

```python
from llama_index.core.agent import FunctionCallingAgent
from llama_index.core.tools import FunctionTool
from llama_index.llms.openai import OpenAI

def get_weather(city: str) -> str:
    """Get the current weather for a city."""
    return f"The weather in {city} is 72°F and sunny."

weather_tool = FunctionTool.from_defaults(fn=get_weather)
llm = OpenAI(model="gpt-4o")

agent = FunctionCallingAgent.from_tools(
    [weather_tool],
    llm=llm,
    verbose=True,
)

response = agent.chat("What's the weather in Montreal?")
print(response)
The agent receives your message, the LLM decides to call `get_weather` with `city="Montreal"`, and the result flows back into the conversation. No prompt engineering for tool selection. No regex parsing.

## FunctionCallingAgent vs FunctionCallingAgentWorker

This is where most people get confused. LlamaIndex has two layers:

**FunctionCallingAgent** is the all-in-one convenience class. You call `.from_tools()`, hand it an LLM, and you're done. It handles the agent loop internally.

**FunctionCallingAgentWorker** is the lower-level component. It handles a single step of reasoning (one LLM call, one tool execution). To run a full conversation, you pair it with an `AgentRunner`, which manages the loop, task state, and memory.

```python
from llama_index.core.agent import FunctionCallingAgentWorker, AgentRunner

worker = FunctionCallingAgentWorker.from_tools(
    [weather_tool],
    llm=llm,
    verbose=True,
)

agent = AgentRunner(worker)
response = agent.chat("What's the weather?")
```

When should you use the worker pattern? When you need fine-grained control over each step, custom task management, or integration into a multi-agent framework like `llama-agents`. For single-agent workflows, `FunctionCallingAgent` does the job with less code.

## Adding real tools that do things

Most tutorials stop at query engines. The agent searches a document index, returns an answer, and that's it. But FunctionCallingAgent can call any Python function you wrap as a `FunctionTool`. That's where it gets interesting.

Say your agent needs to send an email after processing a customer request. You can define an email tool that provisions an inbox and sends the message:

```python
from llama_index.core.tools import FunctionTool
from lobstermail import LobsterMail

async def send_email(to: str, subject: str, body: str) -> str:
    """Send an email from the agent's inbox."""
    lm = await LobsterMail.create()
    inbox = await lm.createSmartInbox(name="support-agent")
    result = await inbox.send(to=to, subject=subject, body=body)
    return f"Email sent to {to} with ID {result.id}"

email_tool = FunctionTool.from_defaults(fn=send_email)
```

Now wire it into the agent alongside your other tools:

```python
agent = FunctionCallingAgent.from_tools(
    [query_tool, email_tool, weather_tool],
    llm=llm,
    system_prompt="You are a support agent. Answer questions using the knowledge base. If the user requests a follow-up, send them an email.",
    verbose=True,
)
```

The LLM decides when to use each tool based on the conversation. Ask it a factual question, it queries the index. Ask it to send a confirmation, it calls the email tool. No routing logic on your end.

The email piece is worth calling out because most agent frameworks treat communication as an afterthought. Your agent can reason all day, but if it can't send an email to confirm a booking or deliver a report, the workflow dead-ends. With [LobsterMail](https://lobstermail.ai), the agent provisions its own inbox at runtime. No human signup, no API key management, no DNS configuration. The agent just calls `LobsterMail.create()` and it has a working email address.

## Using FunctionCallingAgent with local models

FunctionCallingAgent works with any LLM that supports function calling. OpenAI's GPT-4o and GPT-3.5-turbo are the most common, but you can also use:

- **Ollama** (local models): Use `llama_index.llms.ollama`. Note that not all local models handle function calling well. Llama 3 and Mistral v0.3+ have the best support.
- **Azure OpenAI**: Use `llama_index.llms.azure_openai` with your deployment name and API version.
- **Bedrock Converse**: Use `llama_index.llms.bedrock_converse` for Claude or Llama models on AWS.
- **LangChain-wrapped LLMs**: Use `llama_index.llms.langchain.LangChainLLM` to bridge any LangChain model.

Here's the Ollama example, since it's the most requested:

```python
from llama_index.llms.ollama import Ollama
from llama_index.core.agent import FunctionCallingAgent

llm = Ollama(model="llama3", request_timeout=120.0)

agent = FunctionCallingAgent.from_tools(
    [weather_tool],
    llm=llm,
    verbose=True,
)
```

A word of caution: if you see errors like `Could not extract tool calls` or the agent loops without calling any tools, your model likely doesn't support structured function calling. Fall back to `ReActAgent` for those models.

## Adding memory for multi-turn conversations

By default, FunctionCallingAgent maintains conversation history within a single session. For persistent memory across sessions, you can plug in a memory module. The Mem0 integration is one option:

```python
from llama_index.core.agent import FunctionCallingAgent
from mem0 import MemoryClient

memory = MemoryClient(api_key="your-mem0-key")

agent = FunctionCallingAgent.from_tools(
    tools,
    llm=llm,
    memory=memory,
    verbose=True,
)
```

This lets the agent recall facts from previous conversations. Useful for support agents, personal assistants, or any workflow where context carries over between sessions.

## Parallel tool calls

Some models (GPT-4o, for example) support calling multiple tools in a single turn. If your agent needs to look up weather in three cities simultaneously, the model can emit three tool calls at once instead of doing them sequentially.

This is on by default when the model supports it. To disable it (useful for debugging or when tool order matters):

```python
agent = FunctionCallingAgent.from_tools(
    tools,
    llm=llm,
    allow_parallel_tool_calls=False,
)
```

One known issue: Bedrock Converse doesn't always respect this flag consistently across model versions. If you're seeing unexpected behavior with parallel calls on AWS, pin your model version and test explicitly.

## Building agentic RAG with FunctionCallingAgentWorker

The most common production pattern combines function calling with retrieval-augmented generation. You wrap your query engines as tools and let the agent decide which index to search:

```python
from llama_index.core.tools import QueryEngineTool

tool1 = QueryEngineTool.from_defaults(
    query_engine=engine_papers,
    name="research_papers",
    description="Search academic papers on machine learning.",
)

tool2 = QueryEngineTool.from_defaults(
    query_engine=engine_docs,
    name="product_docs",
    description="Search product documentation.",
)

agent = FunctionCallingAgent.from_tools(
    [tool1, tool2],
    llm=llm,
    verbose=True,
)

response = agent.chat("Compare the attention mechanism in the latest paper with our current implementation.")
```

The agent routes the query to the right index (or both) based on the question. This is more flexible than a single RAG pipeline because the agent can combine information from multiple sources in one response.

## Common errors and fixes

**`ValueError: Model does not support function calling`**: Your LLM doesn't expose a function-calling API. Switch to a model that does, or use `ReActAgent` instead.

**`ImportError: cannot import name 'FunctionCallingAgent'`**: You're on an older version of `llama-index-core`. Run `pip install --upgrade llama-index-core`.

**Agent loops indefinitely without calling tools**: The system prompt might be confusing the model. Simplify it. Also check that your tool descriptions are clear. Vague descriptions like "does stuff" won't help the model decide when to use the tool.

**`TypeError: coroutine was never awaited`**: You're mixing async tools with a sync agent runner. Either make all tools sync or use `await agent.achat()`.

## Where to go from here

If you're building an agent that needs to do more than answer questions, start with `FunctionCallingAgent.from_tools()` and a couple of simple tools. Get the loop working. Then add complexity: more tools, memory, a second agent.

For agents that need email capabilities (sending confirmations, receiving verification codes, processing inbound messages), check out [LobsterMail's getting started guide](/docs/getting-started). Your agent can provision its own inbox with one function call and wire it directly into a FunctionCallingAgent as a tool.

FunctionCallingAgent uses the model's native tool-calling API for structured invocations. FunctionCallingAgent is more reliable with supported models, while ReAct works with any text-generation model.
