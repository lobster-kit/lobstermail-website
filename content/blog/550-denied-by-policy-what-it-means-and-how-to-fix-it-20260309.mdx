Your agent sent an email. Instead of a delivery confirmation, it got back a `550 denied by policy` response. The message wasn't queued or deferred. The recipient's mail server looked at your message, checked it against its rules, and rejected it outright.

For AI agents running automated email workflows, this is a real problem. A 550 is permanent. Retrying the same message won't help. It'll just burn your sending reputation faster.

The good news: "denied by policy" is a specific sub-category of 550 errors, and the fix is usually straightforward once you know which policy triggered the rejection.

## What does 550 denied by policy mean?

A 550 denied by policy error is a permanent SMTP rejection code returned by a recipient's mail server, indicating that your message violates a configured policy. Common triggers include SPF or DMARC authentication failures, IP or domain blacklisting, relay restrictions, and domain-level content filtering. Unlike temporary 4xx errors, a 550 tells the sending server not to retry.

The "denied by policy" part narrows things down. It means the rejection wasn't caused by an invalid address or a full mailbox. The server actively enforced a rule, and your message didn't pass. Here are the most common causes:

1. **Missing or misconfigured SPF record** that doesn't authorize your sending IP
2. **DKIM signature failure** because the message was altered in transit or the key doesn't match
3. **DMARC policy set to reject** and the message failed both SPF and DKIM alignment
4. **Sending IP listed on a blacklist** (Spamhaus, Barracuda, etc.)
5. **Relay restrictions** on the recipient's server blocking external senders
6. **Security gateway rules** (Mimecast, Proofpoint, Barracuda) filtering by domain reputation or content

The exact wording varies by mail server. Microsoft Exchange might return `550 5.7.1 Service unavailable, client host blocked`. Mimecast might say `550 Denied by policy`. Google Workspace returns `550-5.7.26 This mail has been blocked because the sender is unauthenticated`. Different words, same category of problem.

## SPF, DKIM, and DMARC failures

Authentication failures cause the majority of 550 policy rejections I see in agent workflows. Here's why: when an agent provisions a new inbox and starts sending from it, the DNS records for that domain need to explicitly authorize the sending server. If they don't, the recipient's mail server has every reason to reject the message.

### SPF (Sender Policy Framework)

SPF tells receiving servers which IP addresses are allowed to send on behalf of your domain. If your sending server's IP isn't listed in the SPF record, the recipient gets a failure signal.

Check your SPF record with:

```bash
dig TXT yourdomain.com | grep spf
A valid SPF record looks something like:
v=spf1 include:_spf.lobstermail.ai ~all
```

The `~all` (softfail) is lenient. Some domains use `-all` (hardfail), which tells recipients to reject unauthorized senders outright. If your SPF record is missing entirely, that's an instant red flag for most mail servers.

### DKIM (DomainKeys Identified Mail)

DKIM attaches a cryptographic signature to each outgoing message. The recipient's server checks the signature against a public key published in your DNS. If the signature doesn't verify (because the key is wrong, the record is missing, or the message body was modified by a relay), you'll get a policy rejection.

### DMARC (Domain-based Message Authentication)

DMARC ties SPF and DKIM together and tells receiving servers what to do when both fail. A DMARC policy of `p=reject` means: if SPF and DKIM both fail alignment, reject the message. A `p=quarantine` policy sends it to spam. A `p=none` policy just reports.

If the domain you're sending from has `p=reject` in its DMARC record and your authentication isn't properly aligned, the recipient's server will return a 550.

Check your DMARC record:

```bash
dig TXT _dmarc.yourdomain.com
```

## When the problem is on the recipient's side

Not every 550 is your fault. Sometimes the recipient's mail server is configured to block entire categories of senders, and there's nothing wrong with your setup.

Common recipient-side causes:

- **Distribution lists restricted to internal senders.** Microsoft 365 distribution groups default to rejecting external email. The recipient's admin needs to allow external senders on that specific group. This is one of the most common 550 errors reported on Microsoft Q&A forums.
- **Security gateways with aggressive filtering.** Mimecast, Proofpoint, and Barracuda can block messages based on domain age, IP reputation, content patterns, or custom admin rules. If your domain is new (under 30 days), expect higher rejection rates from these gateways.
- **IP reputation blocks.** Even if your authentication is perfect, a shared sending IP with a bad reputation will get you blocked. This is especially common with budget SMTP providers that recycle IPs across thousands of customers.

When the issue is on the recipient's side, your options are limited. You can ask the recipient to whitelist your domain or IP. You can also try sending from a different address or domain. But you can't fix their server configuration.

## How to diagnose a 550 denied by policy error

The bounce message itself is your best diagnostic tool. Most 550 responses include an enhanced status code and a human-readable explanation. Here's how to read them:

| Response | Likely cause |
|---|---|
| `550 5.7.1 Relaying denied` | Your server isn't authenticated to relay through the recipient's server |
| `550 5.7.0 Local policy violation` | SPF, DKIM, or DMARC failure on the recipient's end |
| `550 5.7.23 SPF validation failed` | Your SPF record doesn't include the sending IP |
| `550 5.7.26 Unauthenticated mail is not accepted` | DMARC policy rejection (Google Workspace) |
| `550 Denied by policy` | Generic Mimecast or gateway-level block |
| `550 5.7.1 Service unavailable, client host blocked` | IP blacklisted (Microsoft Exchange) |

If you're sending programmatically through an API, the 550 code should appear in the API response or bounce webhook payload. Don't wait for a human to notice a bounce notification in their inbox. Parse the response code and the enhanced status code, then route to the appropriate fix.

For agents, the ideal workflow looks like this: send the message, check the response, and if it's a 550, log the enhanced status code and stop retrying immediately. Retrying a 550 is worse than doing nothing, because it signals to the recipient's server that you're ignoring their rejection.

## How to fix it step by step

Once you've identified the cause from the bounce message, here's what to do:

**For SPF failures:** Add the sending server's IP or include directive to your domain's SPF record. If you're using LobsterMail, the `@lobstermail.ai` domain already has SPF configured. For [custom domains](/docs/getting-started), the setup guide walks through the required DNS records.

**For DKIM failures:** Verify your DKIM public key is published in DNS and matches the private key your sending server uses to sign messages. Rotate the key if it's been compromised or changed.

**For DMARC rejections:** Start with `p=none` while you're debugging, then move to `p=quarantine` and finally `p=reject` once SPF and DKIM are both passing consistently.

**For blacklisted IPs:** Check your sending IP against Spamhaus, Barracuda, and Spamcop. If you're listed, follow their delisting process. If you're on a shared IP, consider moving to a dedicated one or switching providers.

**For relay denied errors:** Make sure your sending client is authenticating with the SMTP server before attempting to send. Unauthenticated relay is blocked by virtually every modern mail server.

**For recipient-side blocks:** Contact the recipient directly (through a different channel) and ask them to whitelist your sending domain or IP. For Microsoft 365 distribution lists, the group owner needs to enable "Allow messages from people outside of this organization."

## Preventing 550 errors before they happen

The best fix is never hitting a 550 in the first place. A few things that help:

Set up SPF, DKIM, and DMARC before you send your first message. Not after your first bounce. Before. If you're using `@lobstermail.ai` addresses, authentication is handled for you automatically. Your agent doesn't need to think about DNS records.

Warm up new domains gradually. Don't send 500 messages on day one from an address that's never sent anything before. Gmail caps new senders at roughly 500 per day, and Google Workspace accounts top out at 2,000. Exceeding those limits can trigger blacklisting that takes weeks to reverse.

Monitor your bounce rates. A bounce rate above 2% is a warning sign. Above 5%, major providers will start throttling or blocking your domain. If your agent is sending programmatically, build in bounce rate monitoring and automatic throttling.

Use a sending infrastructure that handles authentication automatically. This is where managed email services earn their keep. When your agent creates an inbox through LobsterMail, SPF and DKIM records are already configured for the `@lobstermail.ai` domain. The agent doesn't need to touch DNS, and you don't need to debug authentication failures at 2 AM.

## 550 vs. other bounce codes

Not all SMTP errors are equal. A quick reference:

| Code | Type | Meaning | Retry? |
|---|---|---|---|
| 421 | Temporary | Server busy, try again later | Yes |
| 450 | Temporary | Mailbox temporarily unavailable | Yes |
| 451 | Temporary | Local processing error | Yes |
| 550 | Permanent | Mailbox unavailable or policy rejection | No |
| 553 | Permanent | Mailbox name not allowed | No |
| 554 | Permanent | Transaction failed (generic) | No |

The distinction between 4xx and 5xx matters. A 421 or 451 means "try again in a few minutes." A 550 means "stop trying, your message is rejected." Agents that don't distinguish between these will waste resources retrying permanent failures and burning sender reputation in the process.

If your agent is handling email programmatically, build the retry logic around this distinction. Queue 4xx errors for retry with exponential backoff. Log 5xx errors for investigation and move on.
