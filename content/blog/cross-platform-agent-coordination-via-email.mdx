Last week I watched someone try to get a Claude agent to talk to a GPT-4 agent. They spent two days wiring up a custom WebSocket bridge, writing serialization logic, and debugging authentication tokens between two systems that had never been designed to interoperate. The whole thing broke when one side updated its API.

Meanwhile, email has been routing messages between incompatible systems since 1971.

The multi-agent future everyone keeps talking about isn't theoretical anymore. Slack published a guide to agentic platforms in 2026 that describes agents orchestrating cross-channel campaigns by coordinating messaging across email, social, and other channels. Codebridge's research on multi-agent orchestration identifies centralized and decentralized coordination patterns, both of which need a reliable message transport. The question isn't whether agents will coordinate across platforms. It's how.

And the answer might be the most boring protocol in computing.

## Why agent-to-agent protocols keep failing

The AI community has produced several agent coordination protocols in the past year. MCP (Model Context Protocol) handles tool coordination. ACP (Agent Client Protocol) manages agent processes. A2A (Agent-to-Agent Protocol) provides cross-platform federation. Projects like [Routa](https://github.com/phodal/routa) attempt to unify all three.

These protocols are genuinely useful inside a controlled environment. If you're running five agents on the same infrastructure, a purpose-built coordination layer makes sense.

The problem shows up when you cross boundaries. Your Claude agent needs to coordinate with a customer's GPT-based agent. Or your sales agent needs to trigger a workflow in a partner's system running on a completely different stack. Suddenly you need both sides to support the same protocol version, handle authentication, agree on message formats, and maintain persistent connections.

Real-world multi-agent coordination almost never happens within a single platform. A DEV Community guide on building multi-agent systems describes the core challenge: "Someone needs to decide which agent processes what and when to hand off work." When those agents live on different platforms, run different models, and are maintained by different teams, purpose-built protocols become a liability.

## Email as a coordination layer

Email solves the interoperability problem by not trying to be clever about it.

Every platform can send email. Every platform can receive email. The format is standardized (RFC 5322 has been stable for over fifteen years). Delivery is asynchronous, so neither side needs to be online simultaneously. Authentication is handled by the infrastructure (SPF, DKIM, DMARC), not by the application layer.

Here's what cross platform agent coordination via email looks like in practice:

**Agent A** (built on Claude, running on Platform X) needs to request data from **Agent B** (built on GPT-4, running on Platform Y). Agent A composes a structured email with the request parameters in the body, sends it to Agent B's inbox. Agent B receives the email, parses the request, processes it, and sends the response back to Agent A's address. No shared API. No bilateral authentication negotiation. No protocol version conflicts.

This pattern works for several reasons that purpose-built protocols struggle with:

1. **No bilateral setup.** Agent A doesn't need Agent B's API key, SDK version, or endpoint URL. It needs an email address.
2. **Async by default.** Email doesn't require both agents to be online. Messages queue. Retries happen at the transport layer. This matches how real agent workloads operate: tasks take variable time, and blocking calls create fragile chains.
3. **Human-auditable.** Every coordination message is a readable email sitting in an inbox. When something goes wrong (and it will), a human can open the inbox and see exactly what happened. Try doing that with binary protocol messages on a WebSocket.
4. **Universal addressing.** An email address works from any language, any framework, any cloud provider. It's the closest thing to a universal agent identifier we have.

## Structured messages in unstructured envelopes

The obvious objection: email is for humans. It's unstructured. You can't build reliable agent coordination on a format designed for "Hey, just following up on that thing."

Fair point, but it misses how agents actually use email.

An agent sending a coordination message doesn't write prose. It sends a structured payload (JSON, YAML, whatever) inside the email body or as an attachment. The email envelope handles routing and delivery. The body handles the data. You get the reliability of email infrastructure with the precision of structured data.

```json
{
  "task_id": "research-4821",
  "action": "request_summary",
  "parameters": {
    "topic": "Q1 competitor pricing changes",
    "format": "bullet_points",
    "max_length": 500
  },
  "reply_to": "agent-alpha@lobstermail.ai",
  "deadline": "2026-03-09T18:00:00Z"
}
The receiving agent parses this the same way it would parse an API request. The difference is that the transport layer (email) is already handled. No server to run. No endpoint to expose. No firewall rules to configure.

## When email coordination makes sense (and when it doesn't)

I want to be honest about the tradeoffs. Email isn't the right choice for every agent interaction.

**Email works well for:**
- Cross-organization agent coordination (where you can't share infrastructure)
- Asynchronous task handoffs (research requests, report generation, approval workflows)
- Coordination that needs an audit trail
- Agents built on different platforms, languages, or frameworks
- Workflows where latency tolerance is measured in seconds or minutes, not milliseconds

**Email is the wrong choice for:**
- Sub-second real-time coordination (use direct API calls or message queues)
- High-frequency state synchronization (hundreds of messages per minute between the same two agents)
- Streaming data between agents

The sweet spot is coordination between agents that don't share infrastructure and don't need real-time interaction. That covers a surprising amount of real-world multi-agent work. Think of it this way: if a human would coordinate this workflow over email, an agent probably can too.

## Security at the message layer

One thing people overlook about email-based coordination is that the security model is well understood. SPF records verify the sending server. DKIM signatures verify message integrity. DMARC policies tell receivers what to do with messages that fail verification.

For agents specifically, there's another concern: prompt injection. A malicious message could try to manipulate the receiving agent by embedding instructions in the email body. This matters more for agents than for humans, because agents parse and act on message content programmatically.

If you're setting up agent email infrastructure, look for injection detection that scores incoming messages for manipulation risk before your agent processes them. LobsterMail, for example, includes injection risk scoring on every received email, which is useful when your agent's inbox is receiving messages from agents (or humans) you don't fully control.

## Getting started with email-based coordination

The practical steps are simpler than the protocol-based alternatives:

1. **Give each agent its own inbox.** One address per agent, not one shared inbox. This keeps audit trails clean and prevents message routing confusion.
2. **Define a message schema.** Agree on a JSON structure for requests and responses. Include a task ID, action type, and reply-to address at minimum.
3. **Build parsing into your agent's receive loop.** When your agent checks email, parse the body as structured data first. Fall back to natural language processing for unstructured messages.
4. **Set timeout expectations.** Email delivery typically happens within seconds, but plan for minutes. Include deadline fields in your messages so agents know when to stop waiting.

## The pattern is older than we think

Kanerika's research on AI agent orchestration in 2026 describes coordination patterns (centralized supervisors, decentralized peer-to-peer) as if they're new inventions. They're not. Email has been doing both patterns for decades. A central coordinator agent can dispatch tasks to worker agents via email (centralized). Worker agents can message each other directly (decentralized). The addressing system supports both topologies without any infrastructure changes.

The multi-agent ecosystem is going to settle on common coordination patterns eventually. Some will use purpose-built protocols. Some will use email. The agents that can do both will have an advantage, because they won't be locked into a single coordination mechanism that only works with agents on the same platform.

For now, if you need two agents on different platforms to work together and you don't want to spend a week on integration plumbing, give them each an email address and let them talk. It's not glamorous. It works.
