Himalaya is a terminal-based email client that speaks IMAP and SMTP, outputs JSON, and plays nicely with scripts. If you want to connect it to Gmail or Outlook in 2026, you need OAuth2. App passwords are either deprecated or locked behind admin policies, and basic auth has been dead at Google since 2022 and at Microsoft since late 2024.

The setup isn't hard, but it has enough moving parts (provider consoles, client secrets, keyring storage, TOML config blocks) that it's easy to get stuck on one step and waste an hour. This guide walks through the whole process for both Gmail and Outlook, covers the headless-server problem, and explains where this approach breaks down for AI agents.

## How to configure OAuth2 in Himalaya (Gmail and Outlook)

Here's the full sequence, provider by provider:

1. **Register an OAuth2 application** with your email provider (Google Cloud Console or Microsoft Azure AD).
2. **Copy your client ID and client secret** from the provider's dashboard.
3. **Add a TOML config block** in `~/.config/himalaya/config.toml` with `auth.type = "oauth2"` and the correct auth/token URLs for your provider.
4. **Store your client secret in the system keyring** using `secret-tool` (Linux), Keychain (macOS), or Credential Manager (Windows).
5. **Run `himalaya envelope list`** (or any command) to trigger the initial browser-based authorization flow, which exchanges your login for access and refresh tokens.

That's the shape of it. Let me fill in the details.

## Gmail OAuth2 setup

### Register the app in Google Cloud Console

Go to [console.cloud.google.com](https://console.cloud.google.com), create a project (or use an existing one), and navigate to **APIs & Services > Credentials**. Click **Create Credentials > OAuth client ID**, choose "Desktop app" as the application type, and give it a name.

You'll get a client ID and client secret. Save both.

While you're in the console, enable the **Gmail API** under **APIs & Services > Library**. Without this, IMAP connections will fail silently even if your OAuth2 tokens are valid.

One thing that trips people up: Google marks new OAuth apps as "Testing" by default, which limits them to 100 users and makes tokens expire after 7 days. For personal use this is fine. For anything long-running, you'll want to either publish the app or re-authorize weekly.

### Configure Himalaya

Your `~/.config/himalaya/config.toml` should look like this:

```toml
[accounts.gmail]
email = "you@gmail.com"
backend.type = "imap"
backend.host = "imap.gmail.com"
backend.port = 993
backend.login = "you@gmail.com"
backend.auth.type = "oauth2"
backend.auth.client-id = "YOUR_CLIENT_ID.apps.googleusercontent.com"
backend.auth.client-secret.keyring = "gmail-oauth2-client-secret"
backend.auth.access-token.keyring = "gmail-oauth2-access-token"
backend.auth.refresh-token.keyring = "gmail-oauth2-refresh-token"
backend.auth.auth-url = "https://accounts.google.com/o/oauth2/v2/auth"
backend.auth.token-url = "https://oauth2.googleapis.com/token"
backend.auth.scope = "https://mail.google.com/"

message.send.backend.type = "smtp"
message.send.backend.host = "smtp.gmail.com"
message.send.backend.port = 465
message.send.backend.login = "you@gmail.com"
message.send.backend.auth.type = "oauth2"
message.send.backend.auth.client-id = "YOUR_CLIENT_ID.apps.googleusercontent.com"
message.send.backend.auth.client-secret.keyring = "gmail-oauth2-client-secret"
message.send.backend.auth.access-token.keyring = "gmail-oauth2-access-token"
message.send.backend.auth.refresh-token.keyring = "gmail-oauth2-refresh-token"
message.send.backend.auth.auth-url = "https://accounts.google.com/o/oauth2/v2/auth"
message.send.backend.auth.token-url = "https://oauth2.googleapis.com/token"
message.send.backend.auth.scope = "https://mail.google.com/"
The scope `https://mail.google.com/` grants full IMAP and SMTP access. Google doesn't offer narrower scopes for IMAP, so this is what you need.

### Store the secret in your keyring

On Linux with `secret-tool`:

```bash
secret-tool store --label="gmail-oauth2-client-secret" service gmail-oauth2-client-secret
It will prompt for the value. Paste your client secret. On macOS, Himalaya uses Keychain automatically. On Windows, it uses Credential Manager.

### Authorize

Run any Himalaya command:

```bash
himalaya envelope list --account gmail
```

Himalaya opens your browser to Google's consent screen. Log in, grant access, and it stores the resulting tokens in your keyring. Subsequent runs use the refresh token automatically.

## Outlook OAuth2 setup

Microsoft's flow is similar but has one extra requirement: PKCE (Proof Key for Code Exchange).

### Register in Azure AD

Go to [portal.azure.com](https://portal.azure.com) > **Azure Active Directory > App registrations > New registration**. Set the redirect URI to `http://localhost` (type: Mobile and desktop applications).

Under **API permissions**, add:
- `https://outlook.office.com/IMAP.AccessAsUser.All`
- `https://outlook.office.com/SMTP.Send`

Generate a client secret under **Certificates & secrets**.

### Configure Himalaya

```toml
[accounts.outlook]
email = "you@outlook.com"
backend.type = "imap"
backend.host = "outlook.office365.com"
backend.port = 993
backend.login = "you@outlook.com"
backend.auth.type = "oauth2"
backend.auth.method = "xoauth2"
backend.auth.client-id = "YOUR_AZURE_CLIENT_ID"
backend.auth.client-secret.keyring = "outlook-oauth2-client-secret"
backend.auth.access-token.keyring = "outlook-oauth2-access-token"
backend.auth.refresh-token.keyring = "outlook-oauth2-refresh-token"
backend.auth.auth-url = "https://login.microsoftonline.com/common/oauth2/v2.0/authorize"
backend.auth.token-url = "https://login.microsoftonline.com/common/oauth2/v2.0/token"
backend.auth.pkce = true
backend.auth.scope = "https://outlook.office.com/IMAP.AccessAsUser.All https://outlook.office.com/SMTP.Send"

message.send.backend.type = "smtp"
message.send.backend.host = "smtp-mail.outlook.com"
message.send.backend.port = 587
message.send.backend.starttls = true
message.send.backend.login = "you@outlook.com"
message.send.backend.auth.type = "oauth2"
message.send.backend.auth.method = "xoauth2"
message.send.backend.auth.client-id = "YOUR_AZURE_CLIENT_ID"
message.send.backend.auth.client-secret.keyring = "outlook-oauth2-client-secret"
message.send.backend.auth.access-token.keyring = "outlook-oauth2-access-token"
message.send.backend.auth.refresh-token.keyring = "outlook-oauth2-refresh-token"
message.send.backend.auth.auth-url = "https://login.microsoftonline.com/common/oauth2/v2.0/authorize"
message.send.backend.auth.token-url = "https://login.microsoftonline.com/common/oauth2/v2.0/token"
message.send.backend.auth.pkce = true
message.send.backend.auth.scope = "https://outlook.office.com/SMTP.Send"
```

Notice `backend.auth.pkce = true`. Microsoft requires PKCE for public clients, and Himalaya treats desktop apps as public clients. Skip this line and you'll get a vague "cannot wait for redirection" error that tells you nothing useful.

### The xoauth2 vs oauthbearer question

Himalaya supports two SASL mechanisms for OAuth2: `xoauth2` and `oauthbearer`. Gmail works with either. Outlook only supports `xoauth2`. If you don't specify `backend.auth.method`, Himalaya defaults to `xoauth2`, which is the safer bet for cross-provider compatibility.

The practical difference is minimal. Both send an OAuth2 access token to the mail server. `oauthbearer` is the newer RFC 7628 standard, while `xoauth2` is Google's original mechanism that became a de facto standard. Unless your provider specifically requires `oauthbearer`, stick with `xoauth2`.

## The headless server problem

Here's where Himalaya's OAuth2 gets awkward. The initial authorization requires a browser. Himalaya starts a local HTTP server, opens your browser to the provider's consent page, and waits for the redirect callback with the authorization code.

On a headless server, VPS, or CI/CD pipeline, there's no browser. You have a few options:

- **Authorize locally, copy tokens to the server.** Run Himalaya on your laptop first, then copy the keyring entries (or export them) to the server. This works until the refresh token expires or gets revoked.
- **Use an OAuth2 proxy** like `oauth2ms` or `mutt_oauth2.py`. These tools handle the browser flow separately and output tokens that Himalaya can consume.
- **Service accounts** (Gmail only). Google Workspace supports service accounts with domain-wide delegation, which skip the browser entirely. Himalaya doesn't have native service account support, but you can use a helper script to generate access tokens and inject them into the keyring.

None of these are great. They all require a human in the loop at some point, whether that's the initial authorization, the token rotation, or the service account setup in a Google Workspace admin console.

## Where this breaks down for AI agents

If you're building an AI agent that needs to send and receive email, Himalaya with OAuth2 creates a dependency on human-interactive authorization flows. Your agent can't open a browser, click "Allow," and paste a code. Every token refresh that fails (and they do fail, because providers revoke tokens, change policies, and expire grants) requires a human to re-authorize.

This is the core tension: OAuth2 was designed to let *humans* grant access to *applications*. It wasn't designed for autonomous agents that need to self-provision email without human intervention.

There's a Reddit thread from an OpenClaw user who spent days trying to get Himalaya working inside their agent's toolchain. The agent could see the CLI was installed, but the OAuth2 flow required interactive browser auth that couldn't run inside the agent's sandbox. The tool was technically "available" but practically unusable.

If your use case is personal email management from the terminal, Himalaya with OAuth2 is solid. If your use case is giving an autonomous agent its own email, you need infrastructure designed for that. [LobsterMail](/) takes the opposite approach: your agent provisions its own inbox with a single SDK call, no OAuth dance, no browser redirect, no keyring. The agent owns the inbox from the start.

```typescript
import { LobsterMail } from '@lobsterkit/lobstermail';

const lm = await LobsterMail.create();
const inbox = await lm.createSmartInbox({ name: 'My Agent' });
// That's it. inbox.address is ready to send and receive.
```

## Verifying your setup

After completing either the Gmail or Outlook configuration, test it:

```bash
# List recent emails
himalaya envelope list --account gmail

# Send a test email
himalaya template send --account gmail <<EOF
From: you@gmail.com
To: you@gmail.com
Subject: Himalaya OAuth2 test

It works.
EOF
```

If `envelope list` returns results without opening a browser, your refresh token is working. If it opens the browser again, your tokens may have expired (check Google's 7-day expiry for "Testing" apps).

  <FAQItem question="What OAuth2 auth methods does Himalaya support?">
    Himalaya supports two SASL mechanisms: `xoauth2` (Google's original, widely supported) and `oauthbearer` (RFC 7628, newer standard). Set the method with `backend.auth.method` in your TOML config. Default is `xoauth2`, which works with both Gmail and Outlook.
  </FAQItem>

  <FAQItem question="How do I register an OAuth2 app in Google Cloud Console for Himalaya?">
    Go to console.cloud.google.com, create a project, navigate to APIs & Services > Credentials, and create an OAuth client ID with "Desktop app" as the type. Also enable the Gmail API under the Library section.
  </FAQItem>

  <FAQItem question="How do I register an OAuth2 app in Azure AD for Outlook in Himalaya?">
    In portal.azure.com, go to Azure Active Directory > App registrations > New registration. Set the redirect URI to `http://localhost` as a mobile/desktop type. Add IMAP and SMTP permissions under API permissions, then generate a client secret.
  </FAQItem>

  <FAQItem question="What scopes are required for Gmail OAuth2 in Himalaya?">
    Use `https://mail.google.com/` for both IMAP and SMTP. Google doesn't offer more granular IMAP scopes, so this single scope covers everything.
  </FAQItem>

  <FAQItem question="What scopes are required for Outlook OAuth2 in Himalaya?">
    Use `https://outlook.office.com/IMAP.AccessAsUser.All` for reading email and `https://outlook.office.com/SMTP.Send` for sending. Both need to be registered as API permissions in your Azure app.
  </FAQItem>

  <FAQItem question="Why does Himalaya require PKCE for Outlook but not Gmail?">
    Microsoft requires PKCE (Proof Key for Code Exchange) for public clients like desktop apps. Google doesn't enforce it for desktop OAuth flows. Add `backend.auth.pkce = true` to your Outlook config block or you'll get a "cannot wait for redirection" error.
  </FAQItem>

  <FAQItem question="How do I fix the 'cannot wait for redirection' error in Himalaya?">
    This usually means PKCE is missing from your Outlook config. Add `backend.auth.pkce = true` to both the IMAP and SMTP auth sections. Also verify your redirect URI in Azure AD is set to `http://localhost`.
  </FAQItem>

  <FAQItem question="Can Himalaya OAuth2 work on a headless server without a browser?">
    Not directly. The initial authorization requires a browser for the consent screen. You can authorize on a local machine first and copy the tokens to the server, or use an OAuth2 proxy tool like `oauth2ms` to handle the flow separately.
  </FAQItem>

  <FAQItem question="How does Himalaya handle OAuth2 token refresh after expiry?">
    Himalaya stores the refresh token in your system keyring and automatically uses it to get new access tokens when they expire. If the refresh token itself is revoked or expires (Google's "Testing" apps expire tokens after 7 days), you'll need to re-authorize via the browser.
  </FAQItem>

  <FAQItem question="How do I store the OAuth2 client secret in the system keyring for Himalaya?">
    On Linux, use `secret-tool store --label="name" service keyring-name` and paste the secret when prompted. On macOS, Himalaya uses Keychain automatically. On Windows, it uses Credential Manager.
  </FAQItem>

  <FAQItem question="Can I use Himalaya OAuth2 inside an AI agent or automation pipeline?">
    It's difficult. OAuth2 requires interactive browser-based authorization that autonomous agents can't perform. For agent email, consider infrastructure like [LobsterMail](/) where the agent self-provisions an inbox with no human interaction required.
  </FAQItem>

  <FAQItem question="Is it possible to use Himalaya OAuth2 with a Google service account?">
    Himalaya doesn't natively support service accounts, but you can use a helper script to generate access tokens via the Google service account API and inject them into your keyring. This requires a Google Workspace account with domain-wide delegation enabled.
  </FAQItem>

  <FAQItem question="What Cargo feature flags enable OAuth2 when building Himalaya from source?">
    Enable the `oauth2` feature flag when building: `cargo install himalaya --features oauth2`. Without this flag, the OAuth2 configuration options won't be recognized.
  </FAQItem>

  <FAQItem question="What is the difference between configuring OAuth2 via TOML config versus CLI flags?">
    The TOML config file at `~/.config/himalaya/config.toml` is the standard approach and supports all OAuth2 options. CLI flags can override some settings per-command but don't cover the full OAuth2 configuration. Use the config file for setup and CLI flags for one-off overrides.
  </FAQItem>
</FAQ>
