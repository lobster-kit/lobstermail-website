You installed ClawHub, ran `clawhub search`, and got back a wall of 401 errors. Or maybe you ran `clawhub login` three days ago and now your token has quietly expired, so every command fails with a cryptic "Unauthorized" message and no further explanation.

Authentication is the first thing that has to work before you can install skills, publish to the registry, or do anything useful with the ClawHub CLI. Yet the official docs treat it as a one-liner you should already know. Here's everything you actually need to understand about `clawhub login`, `clawhub whoami`, and the three different ways to authenticate, including which method to pick for your specific setup.

## ClawHub authentication commands

Before we get into the details, here's a quick reference of every auth-related command in ClawHub:

- `clawhub login` opens a browser-based OAuth flow to authenticate interactively
- `clawhub login --token <token>` pastes an API token directly without a browser
- `clawhub whoami` verifies your current authenticated session by calling the API
- `clawhub logout` clears your locally stored credentials
- `clawhub -V` or `clawhub --cli-version` prints the installed CLI version (useful for debugging auth issues tied to outdated builds)

That's the full set. Everything else in ClawHub depends on one of these working correctly first.

## The browser flow: clawhub login

Running `clawhub login` with no arguments opens your default browser and walks you through an OAuth flow. You sign in with your OpenClaw account, approve the CLI's access request, and the browser redirects back with a token that gets saved locally.

This is the simplest path if you're working on your own machine. You run it once, the token persists, and every subsequent `clawhub` command uses it automatically.

```bash
clawhub login
The CLI will print a URL and attempt to open your browser. If the browser doesn't open (common on remote machines or WSL), copy the URL manually.

After successful authentication, your token is stored at `~/.config/clawhub/credentials` on Linux/macOS. On Windows, it lives under `%APPDATA%\clawhub\credentials`. The file is readable only by your user account, which is fine for development but worth knowing if you're debugging permission issues later.

One thing to note: the browser flow requires an active OpenClaw account. If you haven't signed up yet, the login page will let you create one during the flow. You don't need to visit the website separately first.

## Token-based login: clawhub login --token

If you can't open a browser (headless servers, containers, CI pipelines), pass a token directly:

```bash
clawhub login --token lm_sk_live_abc123yourtoken
This skips the OAuth flow entirely and writes the token to the same credentials file. The CLI doesn't validate the token at write time, so you won't get an error here even if the token is wrong. That validation happens the next time you run a command that hits the API.

Where do you get the token? From your OpenClaw dashboard under **Settings > API Tokens**. You can generate multiple tokens with different scopes if you need read-only access for some workflows and full write access for others.

For CI/CD pipelines, I'd recommend storing the token as a secret in your pipeline configuration (GitHub Actions secrets, GitLab CI variables, etc.) rather than hardcoding it. Then your pipeline step becomes:

```bash
clawhub login --token "$CLAWHUB_TOKEN"
```

This keeps the token out of your repository and logs.

## The environment variable: CLAWHUB_AUTH

The third option skips `clawhub login` entirely. Set the `CLAWHUB_AUTH` environment variable, and the CLI will use it for every request:

```bash
export CLAWHUB_AUTH="lm_sk_live_abc123yourtoken"
clawhub whoami
```

This takes priority over any stored credentials file. If both exist, the environment variable wins.

When should you use this instead of `clawhub login --token`? Two scenarios. First, when you want temporary authentication that disappears when the shell session ends. Second, when you're running multiple ClawHub accounts and need to switch between them without logging in and out repeatedly. You can set different `CLAWHUB_AUTH` values in different terminal windows.

The downside is that environment variables can leak into process listings and child processes. On shared servers, `clawhub login --token` with proper file permissions is more secure.

## Comparing all three auth methods

Here's when to use each approach:

| Method | Best for | Requires browser | Persists | Security |
|---|---|---|---|---|
| `clawhub login` | Local development on your own machine | Yes | Yes, to credentials file | Good (file permissions) |
| `clawhub login --token` | CI/CD, headless servers, containers | No | Yes, to credentials file | Good if token is stored as a secret |
| `CLAWHUB_AUTH` env var | Temporary sessions, multi-account switching | No | No (session only) | Lower on shared systems |

Pick the one that fits your environment. There's no functional difference in what you can do after authenticating. All three methods produce the same result: an API token attached to every CLI request.

## Verifying your session with clawhub whoami

Once you've logged in through any method, verify it worked:

```bash
clawhub whoami
```

This makes a live API call to `GET /api/v1/whoami` and returns your account details: username, email, account tier, and token scope. It's not reading from a local cache. If the API is down or your token has expired, this command will tell you.

A successful response looks like:

```
Logged in as: your-username
Email: you@example.com
Plan: free
Token scope: full
```

If something is wrong, you'll see a 401 Unauthorized error instead. More on that in a moment.

I run `clawhub whoami` as the first step in every CI pipeline that uses ClawHub. If auth is broken, I want to know before the pipeline spends five minutes installing skills and running tests. It's a one-second sanity check.

You can also parse the output programmatically. The CLI supports `--json` output on most commands:

```bash
clawhub whoami --json | jq '.username'
```

This is useful for automation scripts that need to branch based on the authenticated account or plan tier.

## Troubleshooting authentication errors

A 401 from `clawhub whoami` means one of four things, roughly in order of likelihood:

**Expired token.** Tokens issued through the browser flow have a default expiration. If it's been a while since you logged in, re-authenticate:

```bash
clawhub logout
clawhub login
```

**Invalid token.** If you used `clawhub login --token` and pasted the token incorrectly (extra whitespace, missing characters), the CLI saved it without complaint. Log out and re-enter it carefully.

**Revoked token.** Someone (or you) revoked the token from the OpenClaw dashboard. Generate a new one.

**Environment variable conflict.** If `CLAWHUB_AUTH` is set to a stale value, it overrides your credentials file. Check with:

```bash
echo $CLAWHUB_AUTH
```

If it's set and you don't want it, unset it:

```bash
unset CLAWHUB_AUTH
```

Beyond 401 errors, a common frustration is `clawhub login` hanging without opening a browser. This usually happens in environments without a display server (SSH sessions, Docker containers, WSL without browser integration). Switch to `clawhub login --token` in these cases.

## Token rotation and security practices

The ClawHub docs don't spell out token expiry timelines clearly, which is a gap. From testing, browser-flow tokens appear to last around 30 days before requiring re-authentication. API tokens generated from the dashboard don't expire unless you revoke them manually.

For long-running deployments, set a calendar reminder to rotate tokens monthly. If you're building automation that agents rely on, consider a wrapper script that runs `clawhub whoami` before each operation and re-authenticates if it fails:

```bash
if ! clawhub whoami &>/dev/null; then
  clawhub login --token "$CLAWHUB_TOKEN"
fi
```

This pattern keeps your automation resilient without manual intervention.

## How agents handle auth differently

If you're building an agent that uses ClawHub skills, the authentication story changes. Your agent isn't sitting at a terminal running `clawhub login` interactively. It needs headless auth from the start.

The typical pattern is: store a long-lived API token as part of the agent's configuration, set it via `CLAWHUB_AUTH` or `clawhub login --token` during the agent's startup sequence, and have the agent call `clawhub whoami` as a health check.

This is where agent-first infrastructure matters. An agent shouldn't need a human to open a browser and click "Authorize" every time a token expires. If your agent also needs email (for receiving verification codes, sending notifications, or subscribing to services), tools like [LobsterMail](/) let the agent provision its own inbox the same way it provisions its own ClawHub session: automatically, without human intervention.

The philosophy is the same in both cases. The agent handles its own credentials. The human sets up the initial token and gets out of the way.

## Quick start: zero to authenticated in 60 seconds

If you just want to get going:

```bash
# Install ClawHub CLI (if you haven't)
npm install -g clawhub

# Check the version
clawhub -V

# Log in (opens browser)
clawhub login

# Verify it worked
clawhub whoami

# Start using ClawHub
clawhub search email
clawhub install lobstermail-skill
```

That's it. Five commands and you're authenticated and installing skills. If anything fails, work through the troubleshooting section above.
