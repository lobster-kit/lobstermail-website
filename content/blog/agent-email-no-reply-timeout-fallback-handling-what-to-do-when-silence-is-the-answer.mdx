Your agent sends an email. A verification request, a follow-up to a lead, a support inquiry on behalf of a user. Then nothing. No reply. No bounce. Just silence.

This is the most common failure mode in agent email workflows, and it's the one most people forget to plan for. Bounce handling gets attention because it's loud. Timeout handling gets ignored because it's quiet. But an agent stuck waiting for a reply that never comes is just as broken as one getting 550 rejections. It's worse, actually, because nobody notices until the whole workflow has stalled.

Here's how to build timeout detection and fallback logic that keeps your agent moving when the other side goes quiet.

## How agent email no-reply timeout fallback works

1. **Set a timeout threshold.** Define how long your agent waits for a reply before declaring the conversation stalled (e.g., 30 minutes for verification codes, 48 hours for outreach).
2. **Detect the no-reply event.** Poll the inbox or listen for incoming mail. If no matching reply arrives within the threshold, fire a timeout event.
3. **Trigger the fallback action.** Depending on the workflow, retry with a different message, try an alternate contact method, or requeue the task.
4. **Escalate if the fallback fails.** If retries don't produce a reply, hand off to a human or move the conversation to a different channel.
5. **Notify the stakeholder.** Alert the person or system that owns this workflow. Include the full conversation context so they can pick up where the agent left off.
6. **Log everything for observability.** Record the timeout event, the fallback action taken, and the outcome. You'll need this data to tune your thresholds later.

That's the skeleton. Let's put some muscle on it.

## Timeout thresholds aren't one-size-fits-all

The right timeout depends on what your agent is waiting for. A verification code from an automated system should arrive in seconds. A reply from a human sales prospect might take days. Using the same threshold for both is like setting one alarm for "wake up" and "check on dinner in the oven."

Here's a practical starting framework:

| Scenario | Suggested timeout | Why |
|---|---|---|
| Verification codes | 5-10 minutes | Automated, should be near-instant |
| Transactional confirmations | 30-60 minutes | May pass through internal queues |
| Business outreach (first touch) | 48-72 hours | Humans are slow |
| Follow-up to existing thread | 24-48 hours | They've already engaged once |
| Support/billing inquiries | 4-8 hours (business hours) | SLA expectations |

The key insight: your agent should track these per-conversation, not globally. A single inbox might handle verification codes and outreach simultaneously. Each conversation needs its own timer.

If you're using [webhooks instead of polling](/blog/webhooks-vs-polling-email) to receive mail, timeout detection becomes simpler. Your webhook handler fires on every incoming message. If no webhook fires within the threshold, your scheduling layer triggers the fallback. With polling, you need to check the inbox at intervals and compare the latest message timestamps against your thresholds.

## Retry vs. fallback vs. escalation: they're different things

People use these terms interchangeably. They shouldn't.

**Retry** means sending the same (or similar) message again. Your agent re-sends the verification request, or nudges a prospect with "just following up." Retries assume the original message was lost, delayed, or overlooked.

**Fallback** means trying a different approach entirely. Instead of emailing again, your agent might try a different address, use a different channel, or restructure the message. Fallbacks assume the original approach isn't working.

**Escalation** means handing off to something with more authority or capability. Usually a human. Escalation assumes the agent has exhausted its options.

A good timeout handler chains these in order. Retry first (cheap, often works). Fallback second (more expensive, covers different failure modes). Escalate last (most expensive, but guaranteed to get human attention).

Here's what that looks like in practice:

```typescript
async function handleTimeout(conversation) {
  const attempts = conversation.retryCount || 0;

  if (attempts === 0) {
    // First timeout: gentle retry
    await conversation.inbox.send({
      to: conversation.recipient,
      subject: `Re: ${conversation.originalSubject}`,
      body: "Hi, just following up on my previous message. Let me know if you need anything."
    });
    conversation.retryCount = 1;
    conversation.resetTimeout();
    return;
  }

  if (attempts === 1) {
    // Second timeout: fallback with different framing
    await conversation.inbox.send({
      to: conversation.recipient,
      subject: conversation.originalSubject,
      body: conversation.fallbackMessage
    });
    conversation.retryCount = 2;
    conversation.resetTimeout();
    return;
  }

  // Third timeout: escalate to human
  await escalateToHuman(conversation);
}
Notice the conversation context travels with each step. When the human finally picks this up, they can see the full thread, every retry, and the reason for escalation. That context preservation is the difference between a useful handoff and a confusing one.

## False positives will burn you

Not every timeout is a real no-reply. Before your agent fires a fallback, consider what might have happened to that email.

**Greylisting** is the biggest offender. Some mail servers temporarily reject messages from unknown senders, expecting the sending server to retry after a delay. Standard MTAs handle this automatically, but if your agent interprets the delay as "no reply," it might escalate prematurely. This is a sending-side issue, not a receiving-side one. If you're seeing consistent delays to specific domains, check whether the receiving server uses greylisting before adjusting your timeout logic.

**Spam filtering** is the other culprit. Your agent's email might have landed in the recipient's spam folder. They never saw it. Your agent assumes they're ignoring it. The fix here is upstream: make sure your [email deliverability is solid](/blog/email-deliverability-ai-agents) before you invest too much in timeout handling. No amount of retry logic fixes an email that never reached the inbox.

**Threading mismatches** cause subtler problems. If the recipient replies but their email client generates a new thread (instead of replying to the existing one), your agent might not recognize the reply. It sees a new inbound email but doesn't connect it to the waiting conversation. Your matching logic needs to account for subject-line variations, stripped prefixes, and recipients who reply from a different address than the one you emailed.

## Observability: the part everyone skips

You can't tune what you can't measure. Every timeout event should produce a structured log entry with at least:

- Conversation ID
- Recipient domain
- Timeout threshold used
- Fallback action taken
- Outcome (reply received after retry, escalated, abandoned)
- Time elapsed from original send to resolution

After a few weeks of data, patterns emerge. You'll find that certain domains almost always trigger timeouts (maybe they're slow, maybe your mail is hitting their spam filter). You'll find that your verification code timeout is too aggressive or too lenient. You'll find that the second retry almost never works but the first one does 40% of the time.

This is the data you need to set intelligent, per-domain timeout thresholds. Instead of a flat "48 hours for all outreach," your agent can learn that replies from `bigcorp.com` take 3-5 business days while replies from `startup.io` come within hours. If you want to [test this logic without hitting real recipients](/blog/testing-agent-email-sandbox), a sandbox environment lets you simulate timeout scenarios and validate your fallback chains before they run in production.

## How agent-level timeouts differ from MTA retry logic

Traditional mail transfer agents (Exim, Postfix, Exchange) have their own retry mechanisms. When a message can't be delivered, the MTA queues it and retries on a schedule, typically for 4-5 days before generating a bounce. This handles *delivery* failures.

Agent-level timeout handling is different. Your message was delivered successfully. It reached the recipient's inbox. The MTA's job is done. But the recipient hasn't replied, and your agent's workflow depends on that reply.

These are two separate layers. MTA retries handle transport failures. Agent timeouts handle conversation failures. Confusing them leads to either duplicate sends (your agent retries a message the MTA is already retrying) or missed timeouts (your agent waits for the MTA's 5-day retry window when it should have escalated after 2 hours).

LobsterMail handles the transport layer so your agent doesn't have to think about MTA configuration, delivery retries, or queue management. Your agent only needs to care about the conversation layer: did I get the reply I was expecting, and what do I do if I didn't?

## A practical fallback hierarchy

If you're building this from scratch, here's the order I'd recommend:

1. **Wait.** Don't be trigger-happy. Give the recipient a reasonable window based on what you're waiting for.
2. **Retry once.** A simple follow-up, same thread. Short, polite, specific about what you need.
3. **Retry with variation.** Different subject line, different framing. Maybe the original email looked like spam.
4. **Try a different channel.** If you have a phone number or secondary email, use it.
5. **Escalate to a human with full context.** The conversation history, every retry attempt, the timeout logs. Don't make them start from zero.
6. **Close and archive.** If escalation gets no result after a defined period, mark the conversation as abandoned and move on.

Each step should be configurable per workflow. A password reset flow might jump straight from step 1 to step 5 (no point retrying a verification code that's already expired). A sales outreach flow might loop through steps 2-3 multiple times over weeks.

The goal isn't to guarantee a reply. Some emails simply won't get one. The goal is to make sure your agent recognizes when it's stuck and does something about it instead of waiting forever in a loop that nobody is monitoring.
