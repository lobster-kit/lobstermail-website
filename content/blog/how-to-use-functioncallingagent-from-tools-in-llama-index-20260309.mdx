Most LlamaIndex tutorials show you a chatbot answering questions about a PDF. That's fine for demos. But when your agent needs to *do things* (check an inbox, call an API, write to a database), you need `FunctionCallingAgent.from_tools`. It's the constructor that turns a language model into something that can actually take action.

I spent a few weeks building agent pipelines with LlamaIndex's different agent types. `FunctionCallingAgent.from_tools` is the one I keep reaching for. Here's why, and how to use it properly.

## What is FunctionCallingAgent in LlamaIndex?

`FunctionCallingAgent` is a LlamaIndex agent class that uses the native function-calling (tool-use) capabilities of modern LLMs. Instead of the ReAct-style prompting loop where the model generates text like "Thought: I should call X, Action: X", `FunctionCallingAgent` sends structured tool definitions to the LLM and receives structured tool-call responses back. The model picks which tool to call, with which arguments, using its built-in function-calling protocol.

The `.from_tools()` class method is the standard way to construct one. You hand it a list of tools, an LLM, and optional configuration. It gives you back an agent ready to chat and execute tools.

This matters because structured function calling is more reliable than text-based reasoning for tool selection. The model doesn't have to parse its own output. The tool name and arguments come back as typed data, not strings you hope are formatted correctly.

## How to use FunctionCallingAgent.from_tools in LlamaIndex

1. Install `llama-index` and your preferred LLM integration package
2. Import `FunctionCallingAgent` and `FunctionTool` from `llama_index`
3. Define your tools as Python functions, then wrap them with `FunctionTool.from_defaults()`
4. Initialize the agent with `FunctionCallingAgent.from_tools(tools, llm=your_llm, verbose=True)`
5. Call `agent.chat("your query")` to run the agent loop
6. The agent selects and calls tools automatically, then returns a final response

Here's what that looks like in practice:

```python
from llama_index.core.agent import FunctionCallingAgent
from llama_index.core.tools import FunctionTool
from llama_index.llms.openai import OpenAI

# Define a simple tool
def get_weather(city: str) -> str:
    """Get the current weather for a city."""
    return f"The weather in {city} is 72°F and sunny."

def send_notification(message: str) -> str:
    """Send a notification message."""
    return f"Notification sent: {message}"

# Wrap functions as tools
weather_tool = FunctionTool.from_defaults(fn=get_weather)
notify_tool = FunctionTool.from_defaults(fn=send_notification)

# Build the agent
llm = OpenAI(model="gpt-4o")
agent = FunctionCallingAgent.from_tools(
    [weather_tool, notify_tool],
    llm=llm,
    verbose=True,
    system_prompt="You are a helpful assistant that checks weather and sends alerts.",
)

# Run it
response = agent.chat("What's the weather in Montreal? If it's sunny, notify me.")
print(response)
The `verbose=True` flag is worth keeping on during development. It prints each tool call the agent makes, including the arguments and return values. When something breaks (and it will), this is how you figure out what happened.

## Which LLMs work with FunctionCallingAgent

Not every LLM supports structured function calling. `FunctionCallingAgent` requires a model that implements the function-calling protocol. As of early 2026, these work:

| LLM Provider | LlamaIndex Integration | Function Calling Support |
|---|---|---|
| OpenAI (GPT-4o, GPT-4, GPT-3.5) | `llama_index.llms.openai` | Yes, native |
| Anthropic (Claude 3.5, Claude 3) | `llama_index.llms.anthropic` | Yes, native |
| AWS Bedrock (Converse API) | `llama_index.llms.bedrock_converse` | Yes, with caveats |
| Mistral | `llama_index.llms.mistralai` | Yes |
| Google Gemini | `llama_index.llms.gemini` | Yes |
| Local/open-source (Ollama, vLLM) | Varies | Limited, model-dependent |

If you try to use `FunctionCallingAgent` with a model that doesn't support function calling, you'll get an error at initialization or a confusing failure when the agent tries to invoke a tool. For local models, `ReActAgent` is usually the better choice since it works with any text-generation model.

### The BedrockConverse caveat

There's a known issue (GitHub #17606) where `BedrockConverse` doesn't properly respect the `allow_parallel_tool_calls` flag. If you're using Bedrock and your agent is making tool calls one at a time when it should be batching them, this is likely why. The workaround is to set `allow_parallel_tool_calls=False` explicitly and accept sequential execution, or switch to the OpenAI backend for workflows where parallel calls matter.

## FunctionCallingAgent vs ReActAgent

This is the question everyone asks. Both are LlamaIndex agent types. Both can call tools. The difference is *how* they decide which tool to call.

**ReActAgent** uses a prompt-based reasoning loop. The LLM generates "Thought / Action / Observation" text, and LlamaIndex parses that text to extract the tool name and arguments. This works with any LLM (even ones that don't support function calling), but it's fragile. If the model formats its output slightly wrong, the parser fails. You end up writing retry logic and output-fixing prompts.

**FunctionCallingAgent** sends tool schemas directly to the LLM through its API. The model responds with structured JSON specifying the tool name and arguments. No text parsing needed. This is more reliable but requires an LLM that supports the protocol.

My rule of thumb: if your LLM supports function calling, use `FunctionCallingAgent`. If it doesn't, use `ReActAgent`. There's no advantage to using ReAct with a model that has native tool-use capabilities.

One more difference: `FunctionCallingAgent` supports parallel tool calls (where the model requests multiple tools in a single response). ReAct is inherently sequential since each reasoning step produces one action.

## Configuring tools and handling failures

The `FunctionTool.from_defaults()` wrapper is convenient but limited. For production use, you'll want to understand what's happening under the hood.

Each tool needs a name, a description, and a function signature. The description is what the LLM reads to decide whether to use the tool, so it matters more than you'd think. A vague description like "does stuff with email" will confuse the model. Be specific: "Checks the inbox at a given email address and returns the most recent 10 messages with subject lines and sender addresses."

```python
from llama_index.core.tools import FunctionTool

def check_inbox(email_address: str, limit: int = 10) -> str:
    """Check the inbox at the given email address.
    Returns the most recent messages with subject and sender."""
    # Your implementation here
    pass

inbox_tool = FunctionTool.from_defaults(
    fn=check_inbox,
    name="check_inbox",
    description="Check an email inbox and return recent messages with subjects and senders.",
)
### What happens when a tool raises an exception

If a tool function throws an exception during execution, `FunctionCallingAgent` catches it and feeds the error message back to the LLM as the tool's output. The agent then decides what to do: it might try a different tool, retry with different arguments, or tell the user it failed.

This is actually reasonable behavior for simple cases. But it means the agent can get stuck in a loop retrying a broken tool. The `max_function_calls` parameter (default: 10) acts as a circuit breaker. Once the agent hits that limit, it stops and returns whatever it has. For production workflows, I'd lower this to 5 or 6 and add your own retry logic outside the agent loop.

```python
agent = FunctionCallingAgent.from_tools(
    tools,
    llm=llm,
    max_function_calls=5,
    verbose=True,
)
```

## Wiring FunctionCallingAgent into an email workflow

This is where things get interesting. A `FunctionCallingAgent` with the right tools can handle multi-step processes that would normally require human supervision.

Consider an agent that needs to sign up for a service, receive a verification email, extract the confirmation code, and complete registration. That's four steps, at least two tools (one for interacting with the service, one for handling email), and a decision loop in between.

Here's a simplified version of what that looks like:

```python
from llama_index.core.agent import FunctionCallingAgent
from llama_index.core.tools import FunctionTool
from llama_index.llms.openai import OpenAI
from lobstermail import LobsterMail

lm = LobsterMail.create()

def create_email_inbox(name: str) -> str:
    """Create a new email inbox and return the address."""
    inbox = lm.create_smart_inbox(name=name)
    return inbox.address

def check_email(address: str) -> str:
    """Check for new emails at the given address. Returns subjects and bodies."""
    inbox = lm.get_inbox(address)
    emails = inbox.receive()
    return "\n".join([f"From: {e.sender}, Subject: {e.subject}, Body: {e.body}" for e in emails])

agent = FunctionCallingAgent.from_tools(
    [FunctionTool.from_defaults(fn=create_email_inbox),
     FunctionTool.from_defaults(fn=check_email)],
    llm=OpenAI(model="gpt-4o"),
    system_prompt="You help agents manage email workflows.",
    verbose=True,
)

response = agent.chat(
    "Create an inbox called 'signup-bot', then check it for a verification code."
)
```

The agent creates the inbox, checks for mail, and extracts whatever it finds. If a verification email hasn't arrived yet, it tells you. You could add a polling tool that waits and retries, or handle that outside the agent.

The point isn't that you *need* LlamaIndex to do this (you could write the same logic in a plain Python script). It's that `FunctionCallingAgent` handles the decision-making between steps. If the email contains a verification link instead of a code, the agent adapts. If there are multiple emails, it picks the right one.

## Debugging tips

When your agent isn't calling the right tools (or isn't calling any tools at all), check these things in order:

1. **Turn on `verbose=True`.** Always. You need to see what the model is requesting.
2. **Check your tool descriptions.** If two tools have similar descriptions, the model will pick the wrong one. Make descriptions distinct and specific.
3. **Verify your LLM supports function calling.** If you're using a local model through Ollama, it probably doesn't.
4. **Check `max_function_calls`.** If the agent seems to give up early, you might have hit the limit.
5. **Look at the tool arguments.** Sometimes the model calls the right tool with wrong arguments. Type hints on your function parameters help the model get this right.

If you're still stuck, switch to `ReActAgent` temporarily and compare the outputs. Sometimes seeing the model's explicit reasoning ("Thought: I should check the inbox first") reveals what's confusing it.

## Where to go from here

`FunctionCallingAgent.from_tools` is the right starting point for any LlamaIndex agent that needs to take real actions. Pair it with tools that do meaningful work, keep `max_function_calls` reasonable, and use verbose logging until you trust the pipeline.

If your agent needs email capabilities (receiving verification codes, sending outreach, monitoring inboxes), [LobsterMail](/) handles the infrastructure side. Your agent provisions its own inbox with a single call, and you wrap that in a `FunctionTool` for the agent to use. No manual configuration required.
