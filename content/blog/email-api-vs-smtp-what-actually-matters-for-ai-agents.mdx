Your agent needs to send an email. Maybe it's confirming a signup, replying to a customer, or forwarding a summary to your inbox. Either way, it has two options: talk SMTP to a mail server, or call an email API over HTTP.

For humans writing scripts, both work fine. For autonomous agents making decisions in real time, the difference is significant. Here's a side-by-side breakdown before we get into the details.

| | Email API | SMTP |
|---|---|---|
| **Protocol** | HTTP/REST | TCP (port 25/465/587) |
| **Setup complexity** | API key, one HTTP call | Host, port, username, password, TLS config |
| **Speed** | Single request, ~100-300ms | Multi-step handshake, ~500ms-2s |
| **Error feedback** | Structured JSON with error codes | Numeric reply codes (often cryptic) |
| **Deliverability tools** | Built-in (DKIM, SPF managed for you) | Manual DNS record setup |
| **AI agent compatibility** | Native (agents already speak HTTP) | Requires SMTP library or adapter |
| **Security** | Short-lived API tokens, scoped permissions | Long-lived credentials, broad access |

In short: email APIs speak the same language AI agents already use (HTTP and JSON), while SMTP requires agents to manage stateful TCP connections and parse reply codes designed for mail servers in 1982.

## What SMTP actually is

SMTP (Simple Mail Transfer Protocol) has been the backbone of email delivery since RFC 821 in 1982. It's a text-based protocol where a client opens a TCP connection to a mail server and issues commands like `EHLO`, `MAIL FROM`, `RCPT TO`, and `DATA` in sequence.

Each command gets a numeric response. `250` means OK. `550` means permanent rejection. `421` means try again later. The client has to maintain the connection, handle TLS negotiation, retry on temporary failures, and parse these codes correctly.

For a traditional application with a background email queue and a dedicated SMTP library, this works. The library handles the handshake, the connection pooling, the retry logic. A developer configures it once and moves on.

But an AI agent isn't a traditional application.

## What an email API does differently

An email API wraps all of that complexity into a single HTTP endpoint. You POST a JSON object with `to`, `from`, `subject`, and `body`. You get back a JSON response with a message ID and status. Done.

No connection management. No multi-step handshake. No parsing `550 5.1.1 The email account that you tried to reach does not exist` versus `550 5.7.1 Service unavailable, Client host blocked`. The API returns structured errors your agent can reason about directly.

Most email APIs also handle the deliverability stack for you: DKIM signing, SPF alignment, bounce processing, feedback loops. With SMTP, you're configuring DNS records and monitoring reputation yourself. Or more accurately, nobody's monitoring reputation, because the agent doesn't know how to check MXToolbox.

## Why agents and SMTP don't mix well

Here's where the email API vs SMTP difference matters most for AI agents. An agent calling tools through function calls or MCP already thinks in HTTP. It sends a request, gets a response, decides what to do next. That's exactly how an email API works.

SMTP breaks this pattern in several ways.

**Stateful connections.** SMTP requires opening a connection, authenticating, sending commands in order, and closing the connection. If the agent gets interrupted mid-sequence (timeout, context window limit, error in another tool), the connection hangs. An HTTP request is stateless: fire and forget.

**Credential management.** SMTP needs a host, port, username, and password. These are long-lived credentials that give full access to the mail account. API tokens can be scoped (read-only, send-only, specific inbox) and rotated without changing server configuration. For agents running in sandboxed environments, a single API token is far easier to provision securely than a four-field SMTP credential set.

**Burst handling.** Agents don't send email on a predictable schedule. They might send zero emails for an hour, then fire off 50 in ten seconds when a workflow triggers. SMTP servers typically rate-limit concurrent connections. Hit the limit, and your agent gets `421 Too many concurrent connections`. An email API handles queuing and rate limiting server-side, returning `202 Accepted` immediately and delivering asynchronously.

**Observability.** When an SMTP send fails, your agent gets a reply code and maybe a text explanation. When an API send fails, you get structured JSON: error type, error message, suggested fix, request ID for debugging. For an agent that needs to decide whether to retry, rewrite the message, or escalate to a human, structured data beats parsing text every time.

If you're weighing whether to run your own mail server for this, we covered the tradeoffs in [self-hosted agent email vs managed: what actually makes sense](/blog/self-hosted-vs-managed-agent-email).

## When SMTP still makes sense

SMTP isn't dead. It's the right choice in specific situations.

Legacy systems that already have working SMTP integrations don't need to migrate for the sake of it. If your existing application sends password reset emails through an SMTP relay and it works, leave it alone.

Internal tools behind a firewall sometimes can't make outbound HTTP calls to a third-party API but can relay through an internal SMTP server. That's a valid constraint.

Simple notification scripts that send a single email on a cron job don't benefit much from the API overhead. A five-line Python script using `smtplib` is fine for "email me if the disk is full."

But none of those scenarios involve an autonomous agent making real-time decisions about what to send, when to send it, and what to do when delivery fails. That's where APIs pull ahead.

## How MCP bridges agents and email APIs

Model Context Protocol (MCP) is becoming the standard way AI agents interact with external tools. An MCP server exposes capabilities as typed functions that an agent can call: `send_email`, `check_inbox`, `list_messages`.

Behind the scenes, the MCP server talks to an email API over HTTP. The agent never touches SMTP, never manages credentials directly, never parses reply codes. It calls `send_email(to, subject, body)` and gets back a structured result.

This is the pattern LobsterMail uses. The [MCP server](/docs/mcp-server) exposes email operations as tools. Your agent calls them through its normal tool-calling interface. The MCP server handles authentication, rate limiting, and error translation. The agent stays focused on its actual job.

You can set this up with zero code. Point your agent framework at the LobsterMail MCP server, and email becomes just another tool in the toolkit.

## What this looks like in practice

Here's a concrete example. Your agent needs to sign up for a service, receive a verification email, extract the code, and complete registration.

With SMTP, the agent would need to: configure an IMAP/SMTP client, authenticate, poll the inbox over IMAP (a different protocol entirely), parse the raw email, extract the code, then send a confirmation email back over SMTP. That's two protocols, two authentication flows, and a lot of parsing.

With an email API:

```typescript
const inbox = await lm.createSmartInbox({ name: 'signup-bot' });
// Agent signs up for service using inbox.address
const emails = await inbox.receive({ timeout: 30000 });
const code = extractVerificationCode(emails[0].body);
Three lines. The agent gets a typed response it can work with immediately. No connection management, no protocol switching, no credential juggling.

## Picking the right approach

If you're building an AI agent that sends or receives email, use an API. The protocol alignment (HTTP in, HTTP out), structured error handling, and managed deliverability stack save real engineering time. SMTP adds friction at every step of the agent's workflow without providing anything an API can't do better.

If you have an existing system that already speaks SMTP and you're not adding agent capabilities, there's no reason to rewrite it.

For new agent projects, start with the LobsterMail [getting started guide](/docs/getting-started). Your agent can have a working inbox in under a minute, and you'll never write an SMTP handshake.

<FAQ>
  <FAQItem question="What is the fundamental protocol difference between SMTP and an email API?">
    SMTP is a TCP-based protocol where a client opens a persistent connection and issues sequential text commands (EHLO, MAIL FROM, DATA). An email API uses standard HTTP requests, typically REST with JSON payloads, making it stateless and much simpler to call from modern applications.
  </FAQItem>
  <FAQItem question="Is an email API better than SMTP for sending emails from AI agents?">
    For AI agents, yes. Agents already communicate over HTTP through tool calls and function calling. An email API matches that pattern natively, while SMTP requires managing stateful TCP connections and parsing numeric reply codes.
  </FAQItem>
  <FAQItem question="How does SMTP authentication compare to API key authentication?">
    SMTP requires a host, port, username, and password, all long-lived credentials with broad access. API keys are single tokens that can be scoped to specific permissions (send-only, read-only) and rotated independently without reconfiguring a server.
  </FAQItem>
  <FAQItem question="Can I use both SMTP and an email API together?">
    Yes. Many teams use SMTP for legacy systems (internal alerts, existing app notifications) while routing agent-generated email through an API. The two approaches aren't mutually exclusive.
  </FAQItem>
  <FAQItem question="What happens when an AI agent hits SMTP connection limits during a burst send?">
    The SMTP server returns a `421` error (too many concurrent connections), and the agent has to implement retry logic with backoff. Email APIs handle queuing server-side, accepting the request immediately with a `202` and delivering asynchronously.
  </FAQItem>
  <FAQItem question="How do email APIs support webhook callbacks that AI agents can act on?">
    Email APIs can send HTTP POST requests to a URL you specify when events occur (delivery, bounce, open, click). Your agent receives structured JSON payloads it can process directly, unlike SMTP which provides no delivery feedback to the sender.
  </FAQItem>
  <FAQItem question="How do MCP servers connect AI agents to email without SMTP?">
    An MCP server wraps an email API's HTTP endpoints as typed tool functions (like `send_email` or `check_inbox`). The agent calls these tools through its normal interface. The MCP server handles authentication and protocol details behind the scenes.
  </FAQItem>
  <FAQItem question="Which method gives better error handling for automated systems?">
    Email APIs return structured JSON with error types, messages, and request IDs. SMTP returns numeric codes like `550 5.1.1` with free-text explanations that vary by server. Structured responses are far easier for an agent to parse and act on.
  </FAQItem>
  <FAQItem question="When should I use SMTP instead of an email API?">
    SMTP makes sense for legacy systems with existing integrations, internal tools behind firewalls that can't reach external APIs, and simple scripts where adding an API client would be overkill. For new projects involving AI agents, an API is the better starting point.
  </FAQItem>
  <FAQItem question="Does switching from SMTP to an email API require significant refactoring?">
    Usually not. The core logic (composing a message with to, from, subject, body) stays the same. You're replacing the transport layer: swap out the SMTP library for an HTTP call. Most email APIs provide SDK clients that make migration straightforward.
  </FAQItem>
  <FAQItem question="What are the deliverability differences between SMTP and email APIs?">
    Email APIs typically manage DKIM signing, SPF records, and bounce processing for you. With SMTP, you configure DNS records manually and monitor sender reputation yourself. Managed deliverability is one of the biggest practical advantages of APIs.
  </FAQItem>
  <FAQItem question="Can an AI agent dynamically construct and send emails using an email API?">
    Yes. The agent builds a JSON object with recipient, subject, and body fields, then sends it as an HTTP POST. This fits naturally into tool-calling patterns where the agent decides content and recipients based on context.
  </FAQItem>
