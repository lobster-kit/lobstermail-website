Two agents walk into a conversation. One sends a message. The other replies. Simple enough, right?

Not really. When AI agents communicate over email, every message needs to land in the right thread, reference the right context, and arrive in an order that makes sense to both sides. Humans handle this intuitively. We scan subject lines, skim quoted text, and mentally reconstruct the conversation. Agents don't have that luxury. They need structured metadata to know which message belongs where, what came before it, and what they're actually responding to.

That's where email threading protocols come in. And if you're building multi-agent systems in 2026, understanding how they work isn't optional.

## Why agents talk over email in the first place

There's a growing ecosystem of protocols designed for agent-to-agent communication. Google's A2A (Agent-to-Agent Protocol), Anthropic's MCP, IBM's ACP. These are purpose-built for structured agent collaboration, and they're worth paying attention to.

But email has something none of them have: universality.

Every service on the internet already supports email. Every SaaS product sends notifications over it. Every human has an inbox. When your agent needs to interact with a system that wasn't designed for agent-to-agent protocols (which is most systems), email is the fallback that always works.

More than that, email is asynchronous by design. Agents don't need to be online at the same time. Messages queue up and wait. For workflows that span hours or days (think: procurement approvals, legal review chains, multi-step verification), this matters more than real-time speed.

The catch is that email wasn't designed for machines. It was designed for people in the 1970s. Threading, in particular, is a mess of competing standards and inconsistent implementations.

## How email threading actually works

Email threading relies on three headers defined in RFC 5322 and refined in RFC 2822:

```
Message-ID: <unique-id@sender-domain.com>
In-Reply-To: <message-id-of-parent@sender-domain.com>
References: <first-message-id> <second-message-id> <parent-message-id>
```

**Message-ID** is a globally unique identifier assigned to every email. Think of it as the primary key.

**In-Reply-To** contains the Message-ID of the email you're replying to. It points to the immediate parent in the thread.

**References** contains the full chain of Message-IDs from the root of the conversation to the immediate parent. It's the thread's genealogy.

When an agent receives a reply, it can reconstruct the entire conversation tree by walking the References header. When it sends a reply, it needs to set both In-Reply-To (pointing to the message it's responding to) and References (appending that message's ID to the existing chain).

Get this wrong and the thread breaks. The reply shows up as a new conversation. Context is lost. The receiving agent has no idea what the message is about.

## Where threading breaks down for agents

Human email clients are forgiving. Gmail, for instance, will group messages by subject line even if the threading headers are missing or malformed. Outlook uses its own proprietary `Thread-Index` header. Apple Mail falls back to subject matching when References is empty.

Agents can't rely on these heuristics. Here are the common failure modes:

**Missing Message-ID.** Some SMTP libraries auto-generate a Message-ID, others don't. If your agent's outbound email has no Message-ID, the receiving agent can't reference it in a reply. The thread is broken before it starts.

**Truncated References.** RFC 4021 says the References header should contain the complete chain. But some mail servers truncate it when it gets too long. For agent workflows with dozens of back-and-forth exchanges, this means older context silently disappears.

**Subject line mutations.** Forwarding prepends "Fwd:", replying prepends "Re:". Some servers localize these prefixes ("AW:" in German, "SV:" in Swedish). If your agent matches threads by subject line as a fallback, it needs to strip all of these variants.

**Cross-platform inconsistency.** An agent sending through one email provider might receive replies processed by a completely different provider. Each provider handles edge cases differently. There's no guarantee that threading metadata survives the round trip intact.

## Building reliable agent-to-agent threads

If you're designing a system where agents communicate over email, here are the patterns that actually hold up:

**Always set all three headers.** Don't rely on the email library to handle it. Explicitly set Message-ID, In-Reply-To, and References on every outbound message. Generate Message-IDs using a format like `timestamp.random@yourdomain.com` to guarantee uniqueness.

**Store threads locally.** Don't depend on the email server to maintain thread state. Keep a local mapping of thread ID to message chain. When a new email arrives, match it by In-Reply-To first, then fall back to References, then subject line as a last resort.

**Normalize subjects aggressively.** Strip "Re:", "Fwd:", "RE:", "FW:", and localized variants before comparing. Lowercase everything. Trim whitespace. Two agents sending "Re: Re: FWD: Invoice #4821" and "invoice #4821" should end up in the same thread.

**Set a thread depth limit.** If your agents are going back and forth 50 times in one thread, something is wrong with the workflow design. Cap thread depth at a reasonable number (10-15 exchanges) and start a new thread with a summary of the previous one.

**Use structured content in the body.** Plain text or HTML email bodies are ambiguous. If both agents are yours, consider embedding a small JSON block at the top or bottom of the email body with structured data (task ID, status, required action). The human-readable portion can still exist for debugging, but the machine-readable block is what the receiving agent actually parses.

```typescript
const structuredBlock = {
  threadId: "task-7821",
  action: "approve",
  payload: { amount: 2400, vendor: "Acme Corp" },
};

const body = `
<!-- agent-data: ${JSON.stringify(structuredBlock)} -->

Hi, please approve the attached invoice for $2,400 from Acme Corp.
`;
```

## When to use email vs. purpose-built protocols

Email threading works well when your agents need to interact with external systems, human stakeholders, or services that only support email. It's also a good fit for workflows where auditability matters, since every email is a timestamped record that both parties can reference.

For internal agent-to-agent communication where you control both sides, purpose-built protocols like A2A or ACP will give you better type safety, lower latency, and richer semantics. You don't need to worry about subject line normalization when the protocol handles message routing natively.

The practical answer for most teams in 2026: use both. Purpose-built protocols for the fast, structured internal stuff. Email for everything that touches the outside world.

## What comes next

The line between email protocols and agent communication protocols is blurring. Projects like AgenticMail are already combining traditional email infrastructure with agent-native features like SSE streaming and automatic reply threading. Google's A2A protocol explicitly compares itself to "email protocols for AI agents."

My bet: within a year, we'll see hybrid systems where an agent receives an A2A message, translates it into a threaded email for a human approver, gets the reply, and routes the decision back through A2A to the requesting agent. Email won't be replaced. It'll be the bridge layer.

The agents that handle this well will be the ones with solid threading fundamentals. The ones that don't will be the ones sending context-free messages into the void, wondering why nothing works.
