You installed ClawHub, ran your first command, and got hit with a `401 Unauthorized` error. Or maybe `clawhub whoami` printed nothing useful and you're not sure if you're even logged in.

Authentication is the first real step after installing the OpenClaw CLI, and it trips people up more often than it should. The docs cover the commands, but they don't explain *when* to use which method, what happens under the hood, or how to handle the common failures.

This is a practical walkthrough of every authentication path ClawHub supports: browser login, token-based login, environment variables, and session verification. By the end, you'll know which method fits your setup and how to troubleshoot when things go sideways.

## What is ClawHub and why does authentication matter?

ClawHub is the skill directory and CLI tool for OpenClaw. It's how you discover, install, manage, and update skills that give AI agents new capabilities. Think of it as a package manager, but for agent behaviors instead of code libraries.

Every interaction with the ClawHub registry (searching, installing, updating skills) requires a verified identity. Without authentication, the CLI can't pull skills from the registry, push updates, or even confirm your account exists. The `clawhub login` command handles that handshake, and `clawhub whoami` lets you verify it worked.

## How to log in to ClawHub CLI

There are three ways to authenticate, each suited to a different scenario:

1. Run `clawhub login` to open the browser-based auth flow (interactive use).
2. Run `clawhub login --token <token>` for token-based auth (automated or headless environments).
3. Set the `CLAWHUB_AUTH` environment variable for manual, session-level authentication.

Let's walk through each one.

### Browser-based login

The default flow. Run `clawhub login` in your terminal and the CLI opens your default browser to the OpenClaw authentication page. You sign in (or create an account), approve the CLI's access request, and the browser redirects back. The CLI catches the callback, stores your token locally, and you're done.

```bash
clawhub login
This is the right choice when you're working on your own machine, have a browser available, and want the simplest path forward. The token gets persisted to disk so you don't need to re-authenticate every session.

After logging in, verify it worked:

```bash
clawhub whoami
This prints your authenticated user handle. If you see your username, you're good. If you see an error, something went wrong during the login flow (more on that below).

### Token-based login

If you're running ClawHub on a remote server, inside a container, or anywhere without a browser, the interactive flow won't work. Token-based login solves this.

First, generate an API token from your OpenClaw account settings (on the web). Then pass it directly:

```bash
clawhub login --token <your-api-token>
```

The CLI stores the token the same way it would after a browser login. From this point forward, all commands authenticate using that token. No browser needed.

This is the method you want for CI/CD pipelines, server-side agent deployments, and any non-interactive environment. It's also useful if the browser flow fails for whatever reason and you just need to get past authentication quickly.

### Environment variable auth

The third option is setting `CLAWHUB_AUTH` as an environment variable:

```bash
export CLAWHUB_AUTH="your-auth-here"
```

This takes precedence over any stored token. It's useful for temporary sessions, testing with different accounts, or environments where you don't want tokens written to disk at all. The downside is it only lasts for the current shell session (unless you add it to your shell profile).

## Side-by-side comparison of auth methods

| Method | Best for | Requires browser? | Persists to disk? |
|---|---|---|---|
| `clawhub login` | Local development | Yes | Yes |
| `clawhub login --token` | CI/CD, servers, containers | No | Yes |
| `CLAWHUB_AUTH` env var | Temporary sessions, testing | No | No |

If you're building agents that need to interact with ClawHub programmatically, token-based auth or the environment variable approach is almost always the right call. Browser auth is for humans sitting at a terminal.

## What clawhub whoami actually does

The `clawhub whoami` command is a quick health check. It sends your stored credentials to the ClawHub API and returns your authenticated identity. That's it. No side effects, no state changes. Just a confirmation that your token is valid and the API recognizes you.

```bash
clawhub whoami
# Output: your-username
```

Under the hood, it hits an authentication verification endpoint with your stored Bearer token. If the token is valid, you get your handle back. If it's expired, revoked, or malformed, you get a `401 Unauthorized` error.

I run `clawhub whoami` after every login, after restarting terminals, and before debugging any "permission denied" issue. It takes one second and eliminates the most common variable: "am I even authenticated right now?"

## Troubleshooting authentication errors

### 401 Unauthorized from clawhub whoami

This is the most common issue. It means one of three things:

**Your token expired or was revoked.** Tokens don't last forever. If you generated a token weeks ago and it's stopped working, it may have hit its expiration window. Fix: re-authenticate with `clawhub login` or generate a fresh token.

**You never logged in.** Running `clawhub whoami` before any login attempt will always return 401. The CLI doesn't have credentials to send. Fix: run `clawhub login` first.

**Your stored token is corrupted.** Rare, but it happens. If you manually edited the token file or had a partial write during login, the stored credentials might be broken. Fix: log out and log back in.

```bash
clawhub logout
clawhub login
clawhub whoami
```

That three-command sequence fixes the majority of auth problems. Logout clears the stored token, login creates a fresh one, and whoami confirms it works.

### Browser login flow hangs or fails

Sometimes `clawhub login` opens the browser but the callback never completes. This can happen if a firewall blocks the local redirect, if you're using a remote terminal over SSH, or if your browser is configured to block redirects.

Two workarounds:

1. Switch to token-based auth: `clawhub login --token <token>`
2. Copy the auth URL from the terminal manually, paste it in a browser, complete the flow, and copy the token from the success page.

If you're on a headless server, don't even try the browser flow. Go straight to `--token`.

### Multiple accounts or team environments

If you work across multiple OpenClaw accounts, the environment variable approach keeps things clean. Instead of logging in and out repeatedly, set `CLAWHUB_AUTH` to the token for whichever account you need:

```bash
CLAWHUB_AUTH="token-for-project-a" clawhub whoami
CLAWHUB_AUTH="token-for-project-b" clawhub whoami
```

This avoids overwriting your persisted token and lets you switch contexts per-command.

## Token storage and security

When you authenticate via `clawhub login` or `clawhub login --token`, the CLI writes your Bearer token to a local config file. The exact location depends on your OS, but it follows standard config directory conventions.

A few things to keep in mind:

- Don't commit this file to version control. Add the config path to your `.gitignore`.
- In shared infrastructure (multi-agent setups, shared servers), each agent or user should have its own token. Sharing tokens means you can't revoke access for one agent without breaking the others.
- For automated deployments, prefer the `CLAWHUB_AUTH` environment variable or inject the token via your secrets manager. Writing tokens to disk in ephemeral environments is unnecessary risk.
- Rotate tokens periodically. If you suspect a token has been exposed, revoke it from your OpenClaw account settings and generate a new one immediately.

## Other useful CLI commands

While you're getting comfortable with ClawHub, a few other commands pair well with the auth flow:

```bash
clawhub -V              # Print CLI version
clawhub search <query>  # Search for skills in the registry
clawhub install <slug>  # Install a skill locally
clawhub list            # See what you have installed
clawhub update --all    # Update all installed skills
```

All of these require valid authentication. If any of them return a 401, circle back to `clawhub whoami` and fix your session first.

## Where email fits into agent authentication workflows

If your agent uses ClawHub skills to perform tasks that involve email (signing up for services, receiving verification codes, sending notifications), the auth layer matters more than you might think. A broken ClawHub session means the agent can't install or update the skills it depends on, and the whole workflow stalls.

For agents that need email as part of their skill set, [LobsterMail](/) handles inbox provisioning without requiring manual setup. The agent creates its own email address, receives messages, and extracts what it needs. Pairing that with a properly authenticated ClawHub session means the agent can self-provision both its tools and its communication channel.
