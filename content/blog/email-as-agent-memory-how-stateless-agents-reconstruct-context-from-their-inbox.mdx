Your agent finishes a task, shuts down, and forgets everything. Every conversation, every decision, every piece of context it gathered over the last twelve hours — gone. The next time it spins up, it starts from zero.

This is the statelessness problem. LLMs don't retain memory between invocations. They process whatever lands in their context window, generate a response, and move on. No hidden state persists. No internal notebook carries over. If you want continuity, you have to build it yourself.

Most teams reach for vector databases, Redis caches, or custom file stores. Those work. But there's a memory substrate your agent might already be producing without you realizing it: email.

## What is email as agent memory?

Email as agent memory is the practice of using a structured email thread (its message bodies, threading headers, timestamps, and participant metadata) as the persistent memory layer for a stateless AI agent. Because LLMs reset between invocations, the email record acts as an externalized, append-only context log that the agent reconstructs into working state at the start of each new session. No separate memory store required.

The idea isn't theoretical. If your agent handles customer conversations, vendor coordination, or service signups through email, it's already generating a structured history of everything it has done. Every sent message, every reply, every forwarded attachment is a memory event written to durable storage. The inbox *is* the memory.

## How agents reconstruct context from email threads

When a stateless agent wakes up, it needs to answer one question: what was I doing? With email, the reconstruction process follows a predictable path.

The agent pulls its recent threads. Each email carries metadata that most memory systems would kill for: `In-Reply-To` and `References` headers that map exact conversation lineage, timestamps that establish ordering, `From` and `To` fields that identify every participant, and subject lines that label the task. This isn't unstructured text dumped into a vector store. It's already organized.

From there, the agent loads the relevant thread into its context window, ordered by timestamp. The conversation history tells it who it was talking to, what was agreed on, what's pending, and what the last action was. Context reconstructed. No embedding search. No similarity scoring. Just reading the mail.

We explored pieces of this in [the agent communication stack](/blog/agent-communication-stack). Email sits at the base layer partly because it creates this kind of structured, persistent record that other channels don't.

## Email vs. vector databases as a memory backend

Vector databases are powerful. They let agents search semantically across thousands of memories using embedding similarity. For broad knowledge retrieval ("what do I know about this customer's preferences?"), they're the right tool.

But for episodic memory ("what happened in this specific conversation?"), email has real advantages.

First, email is append-only and immutable. Once a message is sent, it exists as a permanent record. You can't accidentally overwrite a memory. You can't corrupt it with a bad upsert. The history is the history.

Second, email threading is deterministic. You don't need a relevance threshold to decide which memories to retrieve. The thread *is* the relevant context. `References` headers give you the exact chain. No hallucinated connections, no missed edges.

Third, email is already being produced. If your agent communicates externally (and as we covered in [what is agent email](/blog/what-is-agent-email), most useful agents do), it's generating memory events as a side effect of doing its job. There's zero additional infrastructure to provision.

Where email falls short is breadth. A vector database can surface connections across hundreds of unrelated interactions. Email memory is conversation-scoped. For agents that need to recall patterns across many threads, you'd want both: email for episodic context within a task, a broader store for cross-task knowledge.

## Multi-agent context sharing through threads

Here's where email memory gets interesting. When multiple agents share a thread, they share context automatically.

Your research agent emails findings to your writing agent. The writing agent drafts a response and copies your review agent. Every agent on that thread can reconstruct the full task history by reading the same emails. No centralized memory broker. No shared vector store that all three agents need credentials for. The email thread *is* the shared memory, and any agent with access to it can reconstruct the full picture.

This maps directly to what we described in [the future of agent email](/blog/future-of-agent-email): communication history becoming the agent's knowledge graph. The thread isn't just coordination. It's the canonical record of what happened, who did what, and what's left to do.

<Callout type="info">
  The asynchronous nature of email makes it particularly good for multi-agent memory. Agents don't need to be running simultaneously to share context. Agent B can reconstruct everything Agent A did by reading the thread when it wakes up, hours or days later.
</Callout>

## What happens when threads exceed the context window

Real email threads get long. A vendor negotiation might span forty messages over two weeks. A customer support case might have dozens of replies with attachments. Loading the full thread into a 128K token window works for most cases. For the outliers, agents need a truncation strategy.

The practical approach: load the most recent messages in full, summarize older messages into a condensed block, and always preserve the first message (which sets the task context) and the last few messages (which carry the current state). The email metadata (participants, timestamps, subject) takes almost no tokens and should never be truncated. It's the skeleton that holds everything together.

This is the same tradeoff every memory system makes. Vector databases have retrieval limits. File-based memory has read budgets. Email threads have length. The difference is that email gives you natural breakpoints (individual messages) and structural metadata (headers) that make intelligent truncation straightforward.

## When your agent crashes

If an agent with a Redis memory store crashes and the cache is cold, reconstruction depends on what was persisted and how. If an agent with email memory crashes, reconstruction is trivial: read the inbox. Every outbound message the agent sent and every inbound message it received is sitting there, ordered, threaded, and intact. Full state recovery from email history is possible for any task where the agent's work was conducted through email.

The limit is obvious: actions the agent took *outside* email (API calls, file edits, database writes) aren't captured in the email record. Email memory covers the communication layer, not the execution layer. For agents whose primary job is communication (customer support, vendor coordination, scheduling), the coverage is nearly complete. For agents that mostly execute code or manipulate data, email memory is a complement, not a replacement.

The inbox is already a memory store. Your agent just needs to read it.
