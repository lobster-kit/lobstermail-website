You've seen the import line everywhere:

```python
from llama_index.core.agent import FunctionCallingAgent
It shows up in tutorials, GitHub issues, and half-finished Medium posts. But most of those examples stop at "hello world" and leave you guessing about how to actually build something useful with it. This guide covers the full picture: what `FunctionCallingAgent` is, how it compares to the alternatives in LlamaIndex, and how to wire real tools into it, including giving your agent the ability to send and read email on its own.

## What FunctionCallingAgent actually does

LlamaIndex offers two main agent types. `ReActAgent` uses a reasoning loop where the LLM thinks step-by-step and decides which tool to call based on text-based chain-of-thought. `FunctionCallingAgent` skips that loop entirely. It relies on the LLM's native function-calling API (the same mechanism behind OpenAI's tool_calls, Anthropic's tool_use, and similar features in other providers) to select and invoke tools directly.

The practical difference: `FunctionCallingAgent` is faster and more predictable for structured tasks. The LLM sees your tools as typed function signatures and picks the right one with the right arguments in a single inference call. No intermediate reasoning tokens. No parsing text output to figure out which tool was intended.

The tradeoff is flexibility. `ReActAgent` can reason through ambiguous situations where the right tool isn't obvious. `FunctionCallingAgent` works best when your tools are well-defined and the task maps cleanly onto them.

If your agent needs to check an inbox, extract a code, and reply to a message, that's three clear function calls. `FunctionCallingAgent` handles that well. If your agent needs to decide *whether* to reply based on vague contextual clues, `ReActAgent` might be the better fit.

## How to set up FunctionCallingAgent in LlamaIndex

Here's the step-by-step setup, from install to running your first query.

1. **Install the packages.** You need the core library and at least one LLM integration.

```bash
pip install llama-index-core llama-index-llms-openai
2. **Import FunctionCallingAgent and FunctionTool.**

```python
from llama_index.core.agent import FunctionCallingAgent
from llama_index.core.tools import FunctionTool
```

3. **Define your tools as plain Python functions.** Each function becomes a tool the agent can call.

```python
def multiply(a: int, b: int) -> int:
    """Multiply two numbers together."""
    return a * b

def add(a: int, b: int) -> int:
    """Add two numbers together."""
    return a + b

multiply_tool = FunctionTool.from_defaults(fn=multiply)
add_tool = FunctionTool.from_defaults(fn=add)
```

4. **Initialize the LLM.** Any model that supports native function calling works here.

```python
from llama_index.llms.openai import OpenAI

llm = OpenAI(model="gpt-4o")
```

5. **Instantiate the agent with `from_tools()`.**

```python
agent = FunctionCallingAgent.from_tools(
    [multiply_tool, add_tool],
    llm=llm,
    verbose=True,
)
```

6. **Run a query.**

```python
response = agent.chat("What is 7 times 8, plus 3?")
print(response)
```

The agent will call `multiply(7, 8)` first, then `add(56, 3)`, and return `59`. Each tool invocation shows up in the verbose output so you can trace exactly what happened.

## FunctionCallingAgent vs FunctionCallingAgentWorker + AgentRunner

You'll also see this pattern in LlamaIndex codebases:

```python
from llama_index.core.agent import FunctionCallingAgentWorker, AgentRunner

worker = FunctionCallingAgentWorker.from_tools(
    [multiply_tool, add_tool],
    llm=llm,
)
agent = AgentRunner(worker)
```

This does the same thing as `FunctionCallingAgent.from_tools()`, but splits the agent into two layers. The `AgentRunner` manages conversation state, memory, and execution flow. The `FunctionCallingAgentWorker` handles the actual tool-calling logic.

When should you use which?

Use `FunctionCallingAgent` (the simple version) for standalone agents where you don't need custom step-level control. Use the `AgentWorker + AgentRunner` pattern when you want to customize how steps execute, add middleware between tool calls, or compose agents inside a multi-agent system like `llama-agents`.

For most projects, start with `FunctionCallingAgent`. You can always refactor to the worker pattern later if you need more control.

## Wiring in real tools: email as an example

Math functions make good demos. But the reason you're building an agent is to do real work. Let's wire in email as a tool so your agent can send messages and check its inbox as part of a workflow.

Here's where most people hit a wall. Setting up email for an agent usually means provisioning SMTP credentials, configuring DNS records, and managing authentication tokens. That's a lot of manual setup for something that should be a tool call.

LobsterMail handles this differently. Your agent provisions its own inbox with a single function call. No human signup. No DNS configuration. The agent gets an email address and starts sending and receiving immediately.

Here's what an email tool looks like inside a `FunctionCallingAgent` pipeline:

```python
from llama_index.core.agent import FunctionCallingAgent
from llama_index.core.tools import FunctionTool
from llama_index.llms.openai import OpenAI
from lobstermail import LobsterMail

# Agent provisions its own inbox
lm = await LobsterMail.create()
inbox = await lm.createSmartInbox(name="research-agent")

def send_email(to: str, subject: str, body: str) -> str:
    """Send an email from the agent's inbox."""
    inbox.send(to=to, subject=subject, body=body)
    return f"Email sent to {to}"

def check_inbox() -> str:
    """Check for new emails in the agent's inbox."""
    emails = inbox.receive()
    if not emails:
        return "No new emails."
    return "\n".join(
        f"From: {e.sender}, Subject: {e.subject}" for e in emails
    )

send_tool = FunctionTool.from_defaults(fn=send_email)
check_tool = FunctionTool.from_defaults(fn=check_inbox)

agent = FunctionCallingAgent.from_tools(
    [send_tool, check_tool],
    llm=OpenAI(model="gpt-4o"),
    system_prompt="You are a research agent. You can send and check emails.",
    verbose=True,
)

response = agent.chat("Check my inbox for any new messages")
```

The agent now has email as a native capability. It can check for incoming messages, extract information, compose replies, and send outbound mail. All through the same function-calling interface it uses for any other tool.

## Adding memory to your agent

By default, `FunctionCallingAgent` keeps conversation history for the duration of a session but forgets everything when the process ends. For longer-running agents, you'll want persistent memory.

LlamaIndex supports pluggable memory backends. The simplest approach is the built-in chat memory buffer:

```python
from llama_index.core.memory import ChatMemoryBuffer

memory = ChatMemoryBuffer.from_defaults(token_limit=4096)

agent = FunctionCallingAgent.from_tools(
    [send_tool, check_tool],
    llm=llm,
    memory=memory,
    verbose=True,
)
```

For persistence across sessions, integrations like Mem0 and custom memory classes let you store conversation history in a database. The `AgentRunner` pattern gives you more control over how memory is loaded and saved between steps if you need that level of customization.

## Using local models via Ollama

Not every project can send data to OpenAI. If you're running Llama 3 or another local model through Ollama, you can use it with `FunctionCallingAgent`, but there's a catch.

```python
from llama_index.llms.ollama import Ollama

llm = Ollama(model="llama3", request_timeout=120)
```

The catch: not all local models support native function calling. Llama 3 does (through Ollama's function-calling support), but many smaller models don't. If your model doesn't support it, you'll get errors about missing tool_calls in the response. In that case, fall back to `ReActAgent`, which works with any model that can generate text.

A quick test: if `llm.metadata.is_function_calling_model` returns `True`, you're good to go with `FunctionCallingAgent`.

## Debugging tool calls in production

When things go wrong in a `FunctionCallingAgent` pipeline, the issue is usually one of three things:

**The LLM picked the wrong tool.** This happens when tool descriptions are ambiguous. Make your docstrings specific. "Send an email to a recipient" is better than "Handle email." If two tools have overlapping descriptions, the LLM will guess.

**The LLM passed bad arguments.** Type hints matter. If your function expects an `int` and the LLM passes a string, the call fails silently or throws a confusing error. Use strict type annotations and validate inputs inside your tool functions.

**The tool itself failed.** Network errors, auth failures, rate limits. Wrap your tool functions in try/except blocks and return meaningful error messages. The agent will see the error message and can often recover by retrying or adjusting its approach.

Setting `verbose=True` is non-negotiable during development. It shows you every tool selection, every argument, and every return value. In production, pipe that same information to your logging system.

## Where to go from here

If you're building an agent that needs to interact with the real world (sending emails, calling APIs, querying databases), `FunctionCallingAgent` gives you a clean, predictable interface for wiring those capabilities in. Start with the simple `from_tools()` pattern, add tools as you need them, and upgrade to the `AgentWorker + AgentRunner` split when your architecture demands it.

For email specifically, LobsterMail's free tier gives you an inbox your agent can provision on its own. No configuration required. [Check out the getting started guide](/docs/getting-started) to have your agent sending its first email in under a minute.
