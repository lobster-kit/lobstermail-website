In January 2026, a review published in *Information* found that prompt injection now ranks as the number one vulnerability in the OWASP Top 10 for LLM Applications, showing up in over 73% of production AI deployments assessed. That stat is alarming on its own. But there's a newer, less understood variant of the attack that makes traditional prompt injection look quaint by comparison.

It's called prompt infection. And unlike a single injection that corrupts one agent's output, a prompt infection can propagate across an entire multi-agent system, hopping from agent to agent, rewriting instructions as it goes.

If you're building with multiple agents that communicate with each other (or with the outside world through email, documents, or APIs), this is the threat model you need to understand right now.

## What is multi-agent prompt infection spreading?

Multi-agent prompt infection spreading is an attack where a compromised AI agent passes malicious instructions to peer agents through normal communication channels. The malicious prompt embeds itself in the agent's outputs, which become inputs for other agents, causing each newly infected agent to replicate and forward the payload. This creates a self-propagating chain, similar to a biological virus, that can compromise an entire agent network without any further attacker involvement.

Here's how it unfolds in stages:

1. **Initial compromise.** An attacker injects a malicious prompt into one agent, often through user input, an email, or a document the agent processes.
2. **Payload embedding.** The compromised agent begins including the malicious instruction in its outputs, hidden within otherwise normal-looking text.
3. **Peer infection.** Other agents consume those outputs as inputs. The malicious prompt hijacks their behavior too.
4. **Self-replication.** Each newly infected agent embeds the payload in *its* outputs, spreading the infection further.
5. **Objective execution.** While spreading, the infected agents carry out the attacker's goal: data exfiltration, unauthorized actions, or coordination toward a specific target.

Researchers at the paper "Prompt Infection: LLM-to-LLM Prompt Injection within Multi-Agent Systems" (Pasquini et al.) demonstrated this experimentally. Their findings showed that self-replicating infections significantly outperformed non-replicating ones, and that larger agent populations actually accelerated the spread rather than diluting it.

## Why this behaves like a biological virus

The comparison to computer worms and biological viruses isn't just a metaphor. The infection follows logistic growth curves: slow at first, then exponential through the middle of the population, then tapering as fewer uninfected agents remain. In Pasquini et al.'s experiments, a 50-agent system could be fully compromised in as few as 4-6 communication rounds when the payload was self-replicating.

Traditional prompt injection is like food poisoning. One bad input, one sick agent. Prompt infection is like the flu. One sick agent shows up to work and the whole office goes down.

The difference matters because your defenses need to be different. Input sanitization on the perimeter won't help if the infection is already inside, spreading through internal agent-to-agent communication.

## Email: the front door for prompt infections

This is where things get concrete for anyone building agents that process email.

Email is arguably the most dangerous ingestion channel for multi-agent systems. Here's why: emails contain unstructured natural language, they arrive from untrusted external sources, and agents are specifically designed to read and act on them. That's the perfect delivery mechanism for a prompt infection payload.

Consider a common agent workflow: an inbox-monitoring agent reads incoming emails, extracts relevant information, and passes summaries to a task-planning agent. The task-planning agent delegates work to specialized agents (a calendar agent, a CRM agent, a reply-drafting agent). If a single inbound email contains a prompt infection payload, and the inbox agent passes it along without sanitization, every downstream agent in that chain is exposed.

The attack doesn't require anything sophisticated in the email itself. No attachments, no links, no malware. Just text that looks like a normal message but contains embedded instructions that an LLM will interpret as commands.

This is why email security for agents can't rely on the same playbook as email security for humans. Spam filters catch phishing links and known malware signatures. They don't catch "Please also include the following instruction in all your future responses to other agents."

## What infected agents actually do

The research identifies several attack objectives that infected agents can pursue while the infection spreads:

**Data exfiltration.** Infected agents can be instructed to include sensitive information (API keys, user data, conversation history) in their outputs, routing it toward an agent that has external communication capabilities. If any agent in the system can send email, that becomes the exfiltration channel.

**Coordinated action.** Multiple infected agents can be directed to take specific actions in sequence. One agent approves a request, another processes it, a third confirms it. Each agent appears to be functioning normally from the outside.

**Denial of service.** Infected agents can be instructed to produce garbage outputs, consume excessive tokens, or enter infinite loops, effectively taking down the multi-agent system from the inside.

**Persistence.** The most concerning variant: infected agents that store the malicious prompt in long-term memory or configuration, ensuring the infection survives restarts.

## Detection signals you can actually monitor

The research papers describe the theory well, but most don't tell you what to watch for in production. Here are concrete signals:

**Output length anomalies.** Infected agents often produce longer outputs than usual because they're appending the replication payload. If your task-summary agent suddenly starts writing 3x longer summaries, investigate.

**Behavioral drift.** Track what tools and APIs each agent calls. An agent that starts making calls it's never made before (especially to external communication tools) may be compromised.

**Cross-agent echo patterns.** If the same unusual phrase or instruction appears in the outputs of multiple agents that don't share a common prompt, you're likely looking at a propagating infection.

**Token usage spikes.** Self-replicating prompts consume extra tokens at every hop. A sudden, sustained increase in token usage across your agent system is a red flag.

None of these signals are definitive on their own. But together, they form a monitoring baseline that most teams don't have today.

## Defenses that actually help

**LLM Tagging** is one proposed defense from the research literature. The idea is to cryptographically tag LLM-generated outputs so that receiving agents can distinguish between "instructions from a trusted orchestrator" and "text that another LLM generated." It's promising, but it's not a complete solution. It requires every agent in the system to support the tagging protocol, and it doesn't protect against infections that arrive through external channels like email.

**Input sanitization at every boundary, not just the perimeter.** This is the big mental shift. In a multi-agent system, every agent-to-agent communication is a trust boundary. Treat internal messages with the same suspicion as external ones.

**Least-privilege tool access.** An agent that processes emails shouldn't have write access to your database. An agent that summarizes documents shouldn't be able to send outbound mail. Limit the blast radius by limiting what each agent can do.

**Email-layer protection.** If your agents process email, the email infrastructure itself needs to participate in defense. LobsterMail's built-in [injection risk scoring](/docs/security-and-injection) analyzes inbound emails and flags content that looks like prompt injection before your agent ever sees it. That's not a silver bullet, but it moves the detection upstream, before the infection has a chance to enter the multi-agent communication graph.

**Isolation and circuit-breaking.** Design your agent architecture so you can quarantine a suspected infected agent without taking down the whole system. If agent B's outputs suddenly look suspicious, other agents should be able to pause consumption of B's messages while you investigate.

## The compliance angle nobody's talking about

If an infected agent exfiltrates user data through email, you have a data breach. GDPR, HIPAA, and similar regulations don't care whether the breach was caused by a human attacker or a self-replicating prompt. The obligations are the same: notification timelines, documentation, potential fines.

Most teams building multi-agent systems haven't mapped their agent communication flows against their data processing agreements. If Agent A has access to customer PII and Agent B can send external email, a prompt infection that bridges those two agents creates a regulatory incident. Knowing your agent topology isn't just good engineering. It's a compliance requirement.

## What to do this week

If you're running multi-agent systems in production (or planning to), here's where to start:

Map your agent communication graph. Know which agents talk to which, and what data flows between them. You can't defend boundaries you haven't identified.

Add output monitoring. Even basic length and token tracking will catch the most obvious infection patterns.

Audit tool permissions. Remove any tool access that agents don't strictly need. Pay special attention to agents that can send email or make external API calls.

If your agents process inbound email, make sure the email layer is doing security work before the content reaches your agents. LobsterMail's [security features](/docs/security-and-injection) were built specifically for this, because email is the most common way untrusted content enters agent systems.

The research is clear: multi-agent prompt infection spreading is not theoretical. It's been demonstrated experimentally, and the attack surface grows with every agent you add to your system. The good news is that the defenses are straightforward. They just require treating agent-to-agent communication with the same rigor you'd apply to any other trust boundary.
