Email is the tool people reach for first when they give Claude something real to do. Not web search. Not code execution. Email. Almost every workflow that touches the outside world runs through it: verification codes, customer replies, vendor quotes, coordination between agents running on different frameworks. Once Claude can send and receive messages, it can participate in the actual internet.

The standard setup for this involves OAuth flows, SMTP credentials, and enough configuration that you lose an afternoon before the agent sends its first message. With LobsterMail, the agent provisions its own inbox on first run. You define the tools, hand them to Claude, and the agent handles the rest. No human account creation. No API keys to track down.

Here's the full setup, from tool definitions through a working agent loop.

## How Claude tool use works

Tool use lets you describe functions that Claude can call during a conversation. You provide a list of tools, each with a name, a description of what it does, and the parameters it expects. Claude reads those descriptions and decides when to call each tool based on the task at hand.

The interaction follows a loop. You send a message to Claude with your tool list. Claude responds with a `tool_use` block naming the tool and its inputs. Your code executes the tool and returns a `tool_result`. Claude reads the result and either calls another tool or finishes its response. Your application lives in that middle execution step. That's where LobsterMail runs.

The key thing that makes this work well: Claude sequences tool calls itself. Tell it to "create an inbox and then send a welcome email," and it will call `create_inbox` first, read the address from the result, then call `send_email` with that address as the sender. You don't need to hardcode the order.

## Defining the email tools

Three tools handle most agent email workflows: creating an inbox, reading messages, and sending from that inbox.

```typescript
import Anthropic from "@anthropic-ai/sdk";

const emailTools: Anthropic.Tool[] = [
  {
    name: "create_inbox",
    description:
      "Creates a new email inbox and returns its address. Call this before reading or sending.",
    input_schema: {
      type: "object",
      properties: {
        name: {
          type: "string",
          description: "A readable name for the inbox, like 'customer-support'",
        },
      },
      required: ["name"],
    },
  },
  {
    name: "read_emails",
    description: "Returns recent emails delivered to the inbox.",
    input_schema: {
      type: "object",
      properties: {
        address: {
          type: "string",
          description: "The email address returned by create_inbox",
        },
        limit: {
          type: "number",
          description: "Max emails to return. Default 10.",
        },
      },
      required: ["address"],
    },
  },
  {
    name: "send_email",
    description: "Sends an email from the agent's inbox.",
    input_schema: {
      type: "object",
      properties: {
        from: {
          type: "string",
          description: "The sender address (from create_inbox)",
        },
        to: {
          type: "string",
          description: "Recipient email address",
        },
        subject: {
          type: "string",
          description: "Email subject",
        },
        body: {
          type: "string",
          description: "Plain text email body",
        },
      },
      required: ["from", "to", "subject", "body"],
    },
  },
];
```

These go in the `tools` array on every API call. Claude reads the `description` fields to know when to use each one. Write them as plain instructions ("call this before reading or sending"), not abstract keyword strings. Clear descriptions are the single biggest factor in reliable tool selection.

## Wiring in LobsterMail

```bash
npm install @lobsterkit/lobstermail @anthropic-ai/sdk
```

The tool execution function runs when Claude decides to call one of your tools. `LobsterMail.create()` handles account provisioning on first run: it signs up for a free account and stores the token locally at `~/.lobstermail/token`. No browser, no CAPTCHA, no manual step. The [agent self-signup explainer](/blog/agent-self-signup-explained) walks through the full provisioning flow.

```typescript
import { LobsterMail } from "@lobsterkit/lobstermail";

let lm: LobsterMail | null = null;

async function executeTool(
  name: string,
  input: Record<string, unknown>
): Promise<string> {
  if (!lm) lm = await LobsterMail.create();

  if (name === "create_inbox") {
    const inbox = await lm.createSmartInbox({ name: input.name as string });
    return `Inbox created: ${inbox.address}`;
  }

  if (name === "read_emails") {
    const inbox = lm.inbox(input.address as string);
    const emails = await inbox.receive({ limit: (input.limit as number) ?? 10 });
    if (emails.length === 0) return "No emails found.";
    return emails
      .map((e) => `From: ${e.from}\nSubject: ${e.subject}\n\n${e.preview}`)
      .join("\n---\n");
  }

  if (name === "send_email") {
    const inbox = lm.inbox(input.from as string);
    await inbox.send({
      to: input.to as string,
      subject: input.subject as string,
      body: input.body as string,
    });
    return `Email sent to ${input.to}`;
  }

  return `Unknown tool: ${name}`;
}
```

`createSmartInbox` takes a name and generates a clean address from it. If `customer-support@lobstermail.ai` is already taken, it tries variations until it finds a free one. You get a stable, human-readable address without writing any collision logic yourself.

## The tool loop

This is the standard Claude tool use pattern with email handlers plugged in:

```typescript
const client = new Anthropic();

async function runEmailAgent(task: string): Promise<string> {
  const messages: Anthropic.MessageParam[] = [
    { role: "user", content: task },
  ];

  while (true) {
    const response = await client.messages.create({
      model: "claude-sonnet-4-5-20250514",
      max_tokens: 4096,
      tools: emailTools,
      messages,
    });

    if (response.stop_reason === "end_turn") {
      const text = response.content.find((b) => b.type === "text");
      return text?.text ?? "";
    }

    const toolUseBlocks = response.content.filter(
      (b) => b.type === "tool_use"
    );
    if (toolUseBlocks.length === 0) break;

    const toolResults: Anthropic.ToolResultBlockParam[] = [];

    for (const block of toolUseBlocks) {
      if (block.type !== "tool_use") continue;
      const result = await executeTool(
        block.name,
        block.input as Record<string, unknown>
      );
      toolResults.push({
        type: "tool_result",
        tool_use_id: block.id,
        content: result,
      });
    }

    messages.push({ role: "assistant", content: response.content });
    messages.push({ role: "user", content: toolResults });
  }

  return "";
}
```

Run it with a task:

```typescript
const result = await runEmailAgent(
  "Create an inbox called 'signup-agent', then send a welcome message to hello@example.com."
);
console.log(result);
```

Claude calls `create_inbox`, gets back an address, then calls `send_email` using that address as the sender. No hardcoded values, no config file. It sequences the calls itself based on what the task description requires.

## What becomes possible

That's the foundation. Once email is a real tool in Claude's toolkit, a few patterns open up.

**Verification code loops.** Your agent creates an inbox, uses that address to sign up for a third-party service, then polls `read_emails` until the confirmation code lands. The full signup flow runs autonomously in a single conversation turn. The [SDK quickstart](/blog/lobstermail-sdk-quickstart-give-your-agent-an-email-in-5-minutes) includes a working verification example you can copy.

**Cross-agent coordination.** Two agents on separate frameworks can pass structured messages by email with no shared infrastructure between them, just addresses. Agent A sends a JSON payload to Agent B's LobsterMail address. Agent B polls and acts. It's slower than a direct API call, but it works across any runtime or language. See [what agents do with email](/blog/what-agents-do-with-email) for more on this pattern.

**Customer-facing support agents.** Give your support agent a persistent inbox. It receives tickets, drafts replies, sends them. Because `createSmartInbox` resolves naming collisions automatically, the address stays stable across restarts with no manual re-provisioning. We walk through building a full support agent in our [support agent guide](/blog/build-support-agent-email).

The frame worth holding onto: Claude isn't connecting to a preconfigured email account. It's acquiring email as a capability at runtime, based on what the task requires. That distinction matters when you start building workflows where the agent decides what tools it needs.

  LobsterMail returns injection risk scores on received emails. ## Before you ship

A few things that matter in production but are easy to skip in prototypes.

Set `LOBSTERMAIL_API_KEY` in your environment rather than relying on the local token file. The SDK reads it automatically, and it's simpler to manage when your agent runs in a container or serverless function.

Decide on inbox lifetime upfront. Calling `createSmartInbox` with the same name is idempotent: you get back the same address each time. Good for long-running agents. For throwaway verification flows, `createInbox()` with no name gives you a random `lobster-xxxx@lobstermail.ai` address you can discard.

If you need real-time delivery instead of polling, LobsterMail supports [webhooks](/blog/webhooks-vs-polling-email). When an email arrives, it POSTs to your URL immediately. Better suited to event-driven agents than a polling loop.

Free tier covers 1,000 emails a month with no credit card. That's enough for most builds. For higher volume, the Builder tier is $9/month. See [how pricing works](/blog/lobstermail-pricing-explained) for the full breakdown.
