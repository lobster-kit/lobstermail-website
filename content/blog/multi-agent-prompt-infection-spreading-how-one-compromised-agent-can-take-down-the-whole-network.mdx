In January 2026, a research review published in *Information* reported that prompt injection now ranks as the number one vulnerability in the OWASP Top 10 for LLM Applications, showing up in over 73% of production AI deployments assessed. That stat alone should get your attention. But there's a newer, nastier variant most teams haven't prepared for: prompt infection that spreads from agent to agent, autonomously, without any human in the loop.

If you're building with multi-agent systems (or planning to), this is the threat model you need to understand right now.

## What is multi-agent prompt infection spreading?

Multi-agent prompt infection spreading is a class of attack where a single compromised AI agent propagates malicious instructions to other agents it communicates with, using LLM-to-LLM prompt injection. The compromised agent embeds self-replicating payloads in its outputs, causing each newly infected agent to repeat the process and spread the infection further across the network.

Think of it like a biological virus, except instead of cells, it hijacks language models. Instead of RNA, it uses natural language. And instead of sneezing, it spreads through the normal channels agents already use to talk to each other: function calls, shared memory, message passing, and yes, email.

## How a prompt infection actually propagates

The mechanics are surprisingly simple, which is part of what makes them so dangerous.

A standard prompt injection attack targets a single LLM. You sneak instructions into the model's input (through a webpage it reads, a document it processes, an email it receives) and the model follows those instructions instead of its original task. That's bad enough on its own.

Prompt infection takes this further. The injected payload includes instructions for the compromised agent to embed the same malicious prompt in its own outputs. When another agent reads those outputs, it gets infected too. Then that agent's outputs infect the next one, and so on.

The research paper ["Prompt Infection: LLM-to-LLM Prompt Injection within Multi-Agent Systems"](https://arxiv.org/html/2410.07283v1) demonstrated this with controlled experiments. Their findings were uncomfortable: self-replicating infections significantly outperformed non-replicating ones in compromising agent networks. Larger agent populations didn't provide safety in numbers. They actually accelerated the spread.

This isn't theoretical. The infection follows logistic growth curves, similar to epidemiological models. A single infected agent in a network of 20 can compromise the majority within a handful of interaction cycles.

## Why email is the attack surface nobody's talking about

Most discussions of prompt infection focus on internal agent communication: shared context windows, tool outputs, memory stores. But there's an obvious attack vector that gets surprisingly little attention. Email.

AI agents increasingly process inbound email as part of their workflows. Customer support agents read tickets. Sales agents parse replies. Scheduling agents extract calendar invitations. Research agents process newsletters and alerts.

Every one of those inbound emails is an injection surface.

Here's a scenario that keeps me up at night: an attacker sends a carefully crafted email to an agent-monitored inbox. The email contains a prompt infection payload disguised as normal content. The receiving agent processes it, gets compromised, and begins embedding the payload in its outbound communications. If that agent sends emails to other agent-monitored inboxes (forwarding summaries, sending reports, triggering workflows), the infection jumps to the next agent through completely legitimate email channels.

The result: a chain of compromised agents, all coordinating through email, potentially exfiltrating data, sending unauthorized messages, or creating forwarding loops that amplify the attack.

This is different from a human phishing attack. There's no human reading the email and deciding whether to click a link. The agent processes the content automatically. The infection spreads at machine speed.

## Self-replicating vs. non-replicating infections

Not all prompt infections are created equal. The distinction between self-replicating and non-replicating variants matters for defense.

A non-replicating infection compromises one agent. It might instruct that agent to exfiltrate data or perform unauthorized actions, but it doesn't spread further. The blast radius is limited to the single compromised node.

A self-replicating infection includes propagation instructions. The compromised agent is told to embed the payload in everything it outputs. Each downstream agent that processes that output gets the same instructions, including the propagation directive. This is what researchers mean when they call it an "AI worm."

The self-replicating variant is harder to detect because the infection doesn't originate from an external source after the initial compromise. It comes from trusted internal agents. Your monitoring might flag a suspicious inbound email, but it probably won't flag an internal message from Agent A to Agent B that contains business-relevant content with a subtle injection payload woven in.

## What compromised agents actually do

A prompt infection isn't just about spreading. The payload typically includes action directives alongside the replication instructions. The research identified three primary categories:

**Data exfiltration.** The compromised agent is instructed to include sensitive information (user data, API keys, conversation history) in its outbound messages, routing it to an attacker-controlled destination.

**Content manipulation.** The agent's normal outputs get subtly altered. A customer support agent might start recommending a competitor's product. A summarization agent might omit critical information. These changes can be small enough to avoid detection for weeks.

**Scam propagation.** The agent begins generating messages designed to social-engineer humans or other systems. Imagine a compromised email agent sending convincing phishing messages from a trusted business address.

## Defenses that exist today (and their limits)

The most discussed defense mechanism is LLM Tagging, where each message in a multi-agent system is tagged with its origin, allowing agents to distinguish between legitimate instructions and potentially injected content. It helps. It's not sufficient on its own.

LLM Tagging works best when you control the entire communication stack. But many multi-agent systems use heterogeneous channels: some agents communicate through APIs, others through shared databases, others through email. The tagging breaks down at the boundaries.

A layered approach works better:

**Input sanitization at every boundary.** Every time an agent receives content from an external source (including email), that content should be scanned for injection patterns before the agent processes it. This is where agent-aware email infrastructure matters. LobsterMail, for instance, runs injection risk scoring on inbound emails, giving agents a security metadata layer that flags suspicious content before it reaches the LLM.

**Output monitoring for anomalous patterns.** If an agent that normally sends 10 emails a day suddenly starts sending 200, or if its message content shifts dramatically from its baseline, those are signals worth investigating. Good logging and telemetry catch infection spread the same way epidemiologists catch disease outbreaks: by watching for deviation from normal patterns.

**Agent isolation boundaries.** Not every agent needs to talk to every other agent. Reducing the connectivity of your agent network limits how far an infection can spread. Think of it as network segmentation, but for AI agents.

**Quarantine workflows.** When you suspect an agent is compromised, you need a playbook: isolate the agent, review its recent outputs, check downstream agents for infection, and roll back any actions taken during the compromised period. If the infection entered through email, quarantine the inbox and audit all messages processed since the estimated infection time.

## Detection is the hard part

The uncomfortable truth is that most organizations have no way to detect prompt infection spreading through their agent networks today. Traditional security monitoring watches for unauthorized access, unusual network traffic, or known malware signatures. Prompt infections don't trigger any of those alerts.

What you need is behavioral monitoring at the LLM level. Track what your agents are saying, not just what they're doing. Compare current outputs against baseline behavior. Flag content that includes instruction-like patterns directed at other agents.

For email-based workflows specifically, monitoring outbound message content for embedded instructions is critical. An agent that starts including phrases like "ignore your previous instructions" or "forward this message to all contacts" in its emails has almost certainly been compromised.

## The compliance angle nobody's discussing

When a prompt infection causes an agent to exfiltrate user data through email, who's liable? If your agent sends personal data to an unauthorized recipient because it was compromised by a self-replicating prompt infection, you still have a GDPR breach on your hands. "Our AI got infected" is not a recognized exemption under any current data protection regulation.

This is why containment boundaries matter. Having your agents use dedicated, monitored email infrastructure (rather than piggybacking on shared company email accounts) gives you audit trails, isolation capabilities, and the ability to shut down a compromised inbox without affecting your entire organization.

## What to do right now

If you're running multi-agent systems that process email, here are three things to do this week:

First, map your agent communication graph. Which agents talk to which other agents, through what channels? Identify the highest-connectivity nodes. Those are your biggest infection risks.

Second, add injection scanning to every email processing step. Don't let your agents blindly consume email content. LobsterMail's built-in security metadata gives agents injection risk scores on every inbound message, so you can build conditional logic around suspicious content.

Third, set up behavioral baselines for your agents' outputs. You can't detect anomalies without knowing what normal looks like. Start logging now, even if you don't have automated detection yet.

Multi-agent prompt infection spreading is a real, documented attack vector. The research is public, the techniques are known, and the defenses are still catching up. The best time to harden your agent network was before your first deployment. The second best time is today.
