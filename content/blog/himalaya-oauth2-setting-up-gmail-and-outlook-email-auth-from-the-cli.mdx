Himalaya is a terminal-based email client that speaks IMAP and SMTP, outputs JSON, and fits neatly into scripts. It's one of the better tools for managing email from a command line. But the moment you try to hook it up to Gmail or Outlook with OAuth2, the setup gets dense. Fast.

The docs show fragments. A TOML block here, a keyring command there. If you've never registered an OAuth2 application with Google Cloud or Azure AD, you're left jumping between three different documentation sites while trying to figure out which scopes you actually need.

This guide walks through the full Himalaya OAuth2 configuration for both Gmail and Outlook, from application registration to working email. I'll also cover where this whole approach breaks down (headless servers, automated agents, CI/CD pipelines) and what alternatives exist.

## What Himalaya OAuth2 actually does

OAuth2 in Himalaya replaces your email password with token-based authentication. Instead of storing a Gmail password in plaintext, you register an OAuth2 application, authorize it through a browser flow, and Himalaya stores the resulting access and refresh tokens in your system keyring.

Himalaya supports two OAuth2 methods:

- **xoauth2**: The method Gmail uses. It sends a base64-encoded token alongside your email address during IMAP/SMTP authentication.
- **oauthbearer**: The newer standard, used by some providers. Sends the token in a structured bearer format.

Gmail requires `xoauth2`. Outlook supports both but defaults to `xoauth2` in most configurations. If you're unsure which to pick, start with `xoauth2`.

## How to configure Himalaya OAuth2 for Gmail

Here's the full process, from zero to a working inbox:

1. **Create a Google Cloud project** at [console.cloud.google.com](https://console.cloud.google.com). Enable the Gmail API under "APIs & Services."
2. **Create OAuth2 credentials.** Under "Credentials," create an OAuth client ID of type "Desktop app." Copy the client ID and client secret.
3. **Store the client secret in your system keyring** using your OS keyring tool or Himalaya's built-in keyring integration.
4. **Add the OAuth2 config to your `himalaya.toml`** with `backend.auth.type = "oauth2"` and `backend.auth.method = "xoauth2"`, plus the required scopes.
5. **Run `himalaya folder list`** to trigger the browser authorization flow. Himalaya opens a local redirect server, you approve access in the browser, and tokens are stored in the keyring.
6. **Verify it works** by listing your folders or sending a test email.

Here's the minimal TOML configuration for Gmail:

```toml
[accounts.gmail]
email = "you@gmail.com"
backend.type = "imap"
backend.host = "imap.gmail.com"
backend.port = 993
backend.login = "you@gmail.com"
backend.auth.type = "oauth2"
backend.auth.method = "xoauth2"
backend.auth.client-id = "YOUR_CLIENT_ID.apps.googleusercontent.com"
backend.auth.client-secret.keyring = "gmail-oauth2-client-secret"
backend.auth.access-token.keyring = "gmail-oauth2-access-token"
backend.auth.refresh-token.keyring = "gmail-oauth2-refresh-token"
backend.auth.auth-url = "https://accounts.google.com/o/oauth2/v2/auth"
backend.auth.token-url = "https://oauth2.googleapis.com/token"
backend.auth.scopes = ["https://mail.google.com/"]

message.send.backend.type = "smtp"
message.send.backend.host = "smtp.gmail.com"
message.send.backend.port = 465
message.send.backend.login = "you@gmail.com"
message.send.backend.auth.type = "oauth2"
message.send.backend.auth.method = "xoauth2"
message.send.backend.auth.client-id = "YOUR_CLIENT_ID.apps.googleusercontent.com"
message.send.backend.auth.client-secret.keyring = "gmail-oauth2-client-secret"
message.send.backend.auth.access-token.keyring = "gmail-oauth2-access-token"
message.send.backend.auth.refresh-token.keyring = "gmail-oauth2-refresh-token"
message.send.backend.auth.auth-url = "https://accounts.google.com/o/oauth2/v2/auth"
message.send.backend.auth.token-url = "https://oauth2.googleapis.com/token"
message.send.backend.auth.scopes = ["https://mail.google.com/"]
The `https://mail.google.com/` scope grants full IMAP and SMTP access. Google offers narrower scopes, but Himalaya needs read and send permissions to function as a full email client.

## How to configure Himalaya OAuth2 for Outlook

Outlook's setup follows the same pattern, but you register the app through Azure AD instead of Google Cloud.

1. Go to [portal.azure.com](https://portal.azure.com) and navigate to "App registrations."
2. Register a new application. Set the redirect URI to `http://localhost` (Himalaya handles the port).
3. Under "Certificates & secrets," create a new client secret. Copy it immediately; Azure won't show it again.
4. Note the required scopes: `https://outlook.office.com/IMAP.AccessAsUser.All` for reading and `https://outlook.office.com/SMTP.Send` for sending.
5. Outlook requires PKCE (Proof Key for Code Exchange). Add `backend.auth.pkce = true` to your TOML config.

```toml
[accounts.outlook]
email = "you@outlook.com"
backend.type = "imap"
backend.host = "outlook.office365.com"
backend.port = 993
backend.login = "you@outlook.com"
backend.auth.type = "oauth2"
backend.auth.client-id = "YOUR_AZURE_CLIENT_ID"
backend.auth.client-secret.keyring = "outlook-oauth2-client-secret"
backend.auth.access-token.keyring = "outlook-oauth2-access-token"
backend.auth.refresh-token.keyring = "outlook-oauth2-refresh-token"
backend.auth.auth-url = "https://login.microsoftonline.com/common/oauth2/v2.0/authorize"
backend.auth.token-url = "https://login.microsoftonline.com/common/oauth2/v2.0/token"
backend.auth.pkce = true
backend.auth.scopes = ["https://outlook.office.com/IMAP.AccessAsUser.All"]

message.send.backend.type = "smtp"
message.send.backend.host = "smtp-mail.outlook.com"
message.send.backend.port = 587
message.send.backend.login = "you@outlook.com"
message.send.backend.auth.type = "oauth2"
message.send.backend.auth.client-id = "YOUR_AZURE_CLIENT_ID"
message.send.backend.auth.client-secret.keyring = "outlook-oauth2-client-secret"
message.send.backend.auth.access-token.keyring = "outlook-oauth2-access-token"
message.send.backend.auth.refresh-token.keyring = "outlook-oauth2-refresh-token"
message.send.backend.auth.auth-url = "https://login.microsoftonline.com/common/oauth2/v2.0/authorize"
message.send.backend.auth.token-url = "https://login.microsoftonline.com/common/oauth2/v2.0/token"
message.send.backend.auth.pkce = true
message.send.backend.auth.scopes = ["https://outlook.office.com/SMTP.Send"]
The big difference: Outlook requires PKCE and uses separate scopes for IMAP and SMTP. Gmail bundles both under one scope and doesn't require PKCE.

## OAuth2 provider comparison at a glance

| Provider | Auth method | PKCE required | IMAP scope | SMTP scope | Token lifetime |
|---|---|---|---|---|---|
| Gmail | xoauth2 | No | `https://mail.google.com/` | Same as IMAP | ~1 hour (auto-refresh) |
| Outlook | xoauth2 | Yes | `IMAP.AccessAsUser.All` | `SMTP.Send` | ~1 hour (auto-refresh) |
| Yahoo | xoauth2 | No | `mail-w` | Same as IMAP | ~1 hour |

Himalaya handles token refresh automatically once the initial authorization is complete. When the access token expires, the SDK uses the stored refresh token to get a new one without prompting you again. This works well on desktop machines where the keyring stays accessible.

## Fixing the "cannot wait for redirection" error

This is the most common Himalaya OAuth2 failure. You run a command, Himalaya tries to start a local HTTP server to catch the OAuth2 redirect, and it fails with something like `cannot wait for redirection`.

The usual causes:

- **Port conflict.** Another process is using the port Himalaya needs for the redirect. Check with `lsof -i :1143` (or whatever port the error mentions) and kill the conflicting process.
- **Redirect URI mismatch.** The redirect URI in your OAuth2 app registration doesn't match what Himalaya is using. For Google, set it to `http://localhost`. For Azure, same thing. Don't include a port number in the registration; Himalaya picks an available port.
- **Firewall blocking localhost.** Some corporate environments or container setups block local loopback connections. If you're behind a strict firewall, you may need to allowlist the connection.
- **Running in a headless environment.** This is the fundamental problem. Himalaya's OAuth2 flow requires a browser. No browser, no token. (More on this below.)

## The headless problem: OAuth2 without a browser

Here's where Himalaya OAuth2 hits a wall. The authorization flow requires opening a browser, logging in, clicking "Allow," and catching the redirect. On a laptop, that's fine. On a server, inside a Docker container, or in a CI/CD pipeline, there's no browser to open.

Some workarounds exist:

- **Authorize on your laptop first, then copy the keyring tokens to the server.** This works but it's fragile. Tokens expire, and when the refresh token eventually rotates or gets revoked, you're stuck SSH-ing into the server to re-auth manually.
- **Use a headless browser like Puppeteer to automate the consent flow.** This is brittle and violates most providers' terms of service.
- **Pre-populate the keyring with tokens from another OAuth2 client.** Possible, but you're duct-taping around a flow that was designed for humans sitting in front of screens.

None of these are great answers. OAuth2 was built for interactive use. It assumes a human is present to grant consent. That assumption falls apart the moment you try to run email in an automated pipeline.

## When OAuth2 isn't the right tool

If you're building an AI agent that needs to send and receive email programmatically, the Himalaya + OAuth2 approach introduces a human dependency at the worst possible point: initial setup and token renewal. Your agent can't open a browser. It can't click "Allow access." It needs email to work on the first call, without a human in the loop.

This is exactly why we built [LobsterMail](/) with a different model. Instead of wrapping OAuth2, your agent provisions its own inbox through an API call. No browser flow, no client secrets, no token refresh lifecycle to manage.

```typescript
import { LobsterMail } from '@lobsterkit/lobstermail';

const lm = await LobsterMail.create();
const inbox = await lm.createSmartInbox({ name: 'My Agent' });
// inbox.address → my-agent@lobstermail.ai
```

That's the entire setup. No Google Cloud console, no Azure AD registration, no TOML files. The agent handles it.

Himalaya is a good tool for humans who want CLI email. For agents that need autonomous email, the OAuth2 dependency is a design mismatch. If you're wiring email into an automated workflow and don't want to deal with token management, [LobsterMail's getting started guide](/docs/getting-started) covers the full setup in about two minutes.

  </FAQItem>

  <FAQItem question="How do I register an OAuth2 application in Azure AD for Outlook with Himalaya?">
    In the Azure portal, go to "App registrations," create a new app with a redirect URI of `http://localhost`, and create a client secret. Note the client ID and secret for your Himalaya config. You'll also need to enable PKCE.
  </FAQItem>

  <FAQItem question="What scopes are required for Himalaya OAuth2 with Gmail?">
    Gmail uses a single scope: `https://mail.google.com/`. This covers both IMAP read access and SMTP send access.
  </FAQItem>

  <FAQItem question="What scopes are required for Himalaya OAuth2 with Outlook?">
    Outlook uses separate scopes: `https://outlook.office.com/IMAP.AccessAsUser.All` for reading email and `https://outlook.office.com/SMTP.Send` for sending.
  </FAQItem>

  <FAQItem question="Does Himalaya require PKCE for OAuth2?">
    It depends on the provider. Outlook/Office365 requires PKCE (set `backend.auth.pkce = true`). Gmail does not require it. You can enable PKCE for Gmail if you want, but it's not necessary.
  </FAQItem>

  <FAQItem question="How does Himalaya store OAuth2 tokens?">
    Himalaya stores access tokens and refresh tokens in your operating system's keyring (Keychain on macOS, Secret Service on Linux, Credential Manager on Windows). Each token gets a keyring entry name you define in the TOML config.
  </FAQItem>

  <FAQItem question="What does the 'cannot wait for redirection' error mean in Himalaya?">
    This error means Himalaya couldn't start a local HTTP server to catch the OAuth2 redirect callback. Common causes: a port conflict with another process, a redirect URI mismatch in your OAuth2 app registration, or running in a headless environment with no browser available.
  </FAQItem>

  <FAQItem question="Can Himalaya handle OAuth2 token refresh automatically?">
    Yes. Once the initial browser-based authorization is complete, Himalaya uses the stored refresh token to get new access tokens automatically when they expire (typically after one hour). No re-authentication needed unless the refresh token itself is revoked.
  </FAQItem>

  <FAQItem question="How do I use Himalaya OAuth2 in a headless server environment?">
    There's no clean solution. The OAuth2 flow requires a browser for initial authorization. The common workaround is to authorize on a machine with a browser, then copy the keyring tokens to the server. This breaks when tokens need manual renewal.
  </FAQItem>

  <FAQItem question="Can I use Himalaya OAuth2 inside an AI agent or automated pipeline?">
    Technically yes, but it's fragile. The initial OAuth2 authorization requires human interaction (browser consent). Token refresh is automatic, but if the refresh token expires or gets revoked, the pipeline stalls until a human re-authorizes. For fully autonomous agent email, [LobsterMail](/) removes the OAuth2 dependency entirely.
  </FAQItem>

  <FAQItem question="What's the difference between the old himalaya-repl TOML format and the current format?">
    The old format used flat keys like `imap.host` and `imap.auth.type`. The current Himalaya CLI uses nested `backend.*` keys like `backend.type = "imap"` and `backend.auth.type = "oauth2"`. If you're migrating from himalaya-repl, you'll need to restructure your TOML file.
  </FAQItem>

  <FAQItem question="Is Himalaya CLI secure for email authentication?">
    Yes, for interactive use. OAuth2 tokens are stored in the system keyring rather than in plaintext config files, and Himalaya never stores your email password. The security concern is more about the operational environment: keyring access on shared servers, token handling in containers, and the difficulty of rotating credentials in automated setups.
  </FAQItem>

  <FAQItem question="Can I use Himalaya OAuth2 with a custom or self-hosted email provider?">
    Yes, if your provider supports OAuth2 with standard IMAP/SMTP. You'll need to set the `auth-url` and `token-url` to your provider's OAuth2 endpoints and configure the appropriate scopes. Not all self-hosted solutions (like Dovecot) support OAuth2 out of the box.
  </FAQItem>
</FAQ>
