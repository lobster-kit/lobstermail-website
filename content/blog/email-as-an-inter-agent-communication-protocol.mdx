Every few weeks, a new inter-agent communication protocol drops. MCP. ACP. A2A. ANP. Each one promises to be the standard that finally lets AI agents talk to each other without duct tape. I've been reading the specs, following the GitHub threads, trying the SDKs. Some of these are genuinely good work.

But here's what keeps bugging me: we already have a protocol that handles asynchronous, cross-boundary, persistent messaging between independent parties. It's been running since 1982. It's called email.

I'm not saying SMTP replaces everything MCP does. That would be silly. But when you look at what agents actually need for inter-agent communication, email covers a surprising amount of the surface area, and the protocol comparison conversations keep ignoring it entirely.

## The protocols, compared

An inter-agent communication protocol is a standardized way for AI agents to exchange messages, share context, and coordinate tasks across different systems or frameworks. Here's how the current options stack up:

| Protocol | Primary use case | Communication style | Transport layer | Email support |
|----------|-----------------|---------------------|-----------------|---------------|
| MCP | Tool and context access | Synchronous | JSON-RPC over stdio/HTTP | No |
| ACP | Multi-agent coordination | Async and sync | HTTP/REST | No |
| A2A | Cross-framework agent interop | Async with callbacks | HTTP with Agent Cards | No |
| ANP | Decentralized agent networking | Async | HTTP with DID auth | No |
| SMTP/IMAP | Async message passing | Asynchronous | SMTP (send), IMAP (receive) | Native |

Notice the pattern? Every new protocol builds its own transport. None of them use the transport that already reaches every organization, every domain, every inbox on the internet.

## What MCP and ACP actually solve

Let me be fair to these protocols before I make the case for email.

**MCP (Model Context Protocol)** is Anthropic's standard for giving agents access to tools and data sources. Think of it as the way an agent reads your internal wiki or calls your database. It's synchronous, tightly coupled, and designed for agents operating within a single trust boundary. AWS recently published a [detailed exploration](https://aws.amazon.com/blogs/opensource/open-protocols-for-agent-interoperability-part-1-inter-agent-communication-on-mcp/) of how MCP can enable inter-agent communication, but it requires both agents to share the same MCP server infrastructure.

**ACP (Agent Communication Protocol)** handles multi-agent coordination with both sync and async patterns. IBM's been pushing it as the way for agents to collaborate on complex tasks. It's more flexible than MCP for agent-to-agent work, supporting long-running tasks and structured message semantics.

**A2A (Agent-to-Agent Protocol)** from Google focuses on cross-framework interop. Your LangChain agent talks to your CrewAI agent through standardized "Agent Cards" that describe capabilities. It's the closest thing to a universal directory for agents.

**ANP (Agent Network Protocol)** takes a decentralized approach with DID-based authentication. It's early, but the vision is a peer-to-peer network where agents discover and communicate without a central registry.

These all solve real problems. But they share a limitation: they only work when both sides have adopted the same protocol. And they require infrastructure that someone has to set up and maintain.

## Why email works as an inter-agent transport

Email has properties that no new protocol has managed to replicate:

**Universal addressing.** Every agent that has an email address can be reached by every other agent. No shared infrastructure required. No protocol negotiation. No Agent Cards. Just an address. We covered this in depth in [the agent communication stack: email, chat, voice, and what's next](/blog/agent-communication-stack).

**Store-and-forward queuing.** If the receiving agent is offline, the message waits. SMTP was literally designed for this. The "offline agent discovery" problem that ACP and ANP are solving with complex retry logic and capability registries? Email's had that solved since before most of us were born. Messages queue at the receiving mail server until the agent polls or gets a webhook notification.

**Built-in audit trails.** Every email has headers showing the full delivery path: who sent it, when, through which servers, and whether the signatures checked out. For enterprise multi-agent deployments where compliance matters, this is not a small thing. Try getting that level of auditability from a WebSocket connection.

**Async by default.** Inter-agent communication is almost always asynchronous. Agent A sends a request. Agent B processes it whenever it's ready. Agent B responds. This is email's native mode. You don't need to bolt on callback URLs or polling mechanisms. The protocol is the queue.

**Human-in-the-loop for free.** Here's something none of the new protocols handle well: what happens when an agent needs to escalate to a human? With email, the human is already in the channel. An agent can CC a person, forward a thread, or hand off a conversation without switching protocols. We explored this pattern in [multi-agent email: when agents need to talk to each other](/blog/multi-agent-email-coordination).

## The deliverability problem nobody talks about

If email is so great for inter-agent messaging, why isn't everyone using it?

Because deliverability for automated senders is genuinely hard. When agents send high volumes of email, spam filters get suspicious. Domain authentication (SPF, DKIM, DMARC) has to be configured correctly or messages vanish into spam folders. IP reputation matters. Content scanning can flag structured data as suspicious.

This is the real engineering challenge, and it's the reason most teams reach for a new protocol instead of using email. Setting up proper email infrastructure for agents means managing DNS records, warming IP addresses, monitoring bounce rates, and handling feedback loops. It's not impossible. It's just tedious enough that building a custom WebSocket feels easier.

## When email isn't the right choice

I'd be dishonest if I claimed email replaces all inter-agent protocols. It doesn't.

**Real-time coordination:** If two agents need sub-second response times (think: a coding agent calling a search agent mid-generation), email is too slow. MCP or direct function calls make more sense here.

**Tool invocation:** MCP exists because agents need to call functions, not just send messages. Email is a messaging protocol, not an RPC framework. You wouldn't send an email to query a database.

**Structured capability discovery:** A2A's Agent Cards let agents programmatically discover what another agent can do. Email has no equivalent. You'd need to layer that on top (which is entirely possible, but it's extra work).

**High-frequency messaging:** If agents are exchanging hundreds of messages per second, email's overhead is prohibitive. Use a message broker.

The sweet spot for email as an inter-agent communication protocol is cross-boundary, async, low-to-medium frequency messaging where persistence and auditability matter more than speed.

## The hybrid approach

The most practical architecture I've seen combines protocols by layer:

- **MCP** for internal tool access (agent talks to its own tools)
- **A2A or ACP** for real-time agent coordination within a trusted network
- **Email** for cross-boundary messaging, human escalation, and audit trails

Your agent uses MCP to access its knowledge base, A2A to coordinate with other agents in your stack, and email to communicate with agents (or humans) outside your trust boundary. Each protocol plays to its strengths.

For the email layer, your agent needs to receive messages reliably. That means either [webhooks or polling](/blog/webhooks-vs-polling-email), depending on your latency requirements. Webhooks give you near-real-time delivery. Polling is simpler to set up and works well when a few seconds of delay doesn't matter.

## What comes next

I think we'll see email formalized as an inter-agent transport within the next year. Someone will write an RFC for structured agent-to-agent messages over SMTP, with standardized headers for capability declaration, message typing, and conversation threading. The building blocks are all there. MIME types can carry structured payloads. Email headers are extensible. Threading already works.

The missing piece has been infrastructure that treats agents as first-class email users, not second-class citizens piggy-backing on human Gmail accounts. That's changing.
