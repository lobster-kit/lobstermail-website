In January 2026, a review published in *Information* found that prompt injection now ranks as the number one vulnerability in the OWASP Top 10 for LLM Applications, showing up in over 73% of production AI deployments assessed. That stat is alarming on its own. But the newer, weirder cousin of prompt injection is worse: prompt infection, where compromised agents don't just break, they spread the attack to other agents automatically.

If you're running multi-agent workflows (and if your agents handle email, you probably are), this matters right now.

## What is multi-agent prompt infection?

Multi-agent prompt infection is an attack where a compromised AI agent embeds malicious instructions into its own outputs, which downstream agents then process as trusted input and execute. The compromised agent effectively becomes a carrier, rewriting its responses to include self-replicating payloads that propagate through the system like a virus. Unlike standard prompt injection, which targets a single model, prompt infection spreads laterally across an entire agent network without any further attacker involvement.

The concept was formalized in an October 2024 [research paper from arxiv](https://arxiv.org/html/2410.07283v1) titled "Prompt Infection: LLM-to-LLM Prompt Injection within Multi-Agent Systems." The researchers demonstrated that once a single agent is compromised, the infection follows a logistic growth curve. In smaller networks of three to five agents, full compromise can happen in under ten message exchanges. In larger populations, infection spreads even faster in absolute terms because there are more surfaces for the payload to land on.

That's the core problem: agents trust each other's outputs by default. There's no authentication layer between "what Agent A said" and "what Agent B treats as instructions."

## How infection spreads through agent networks

The mechanics are straightforward, which is part of what makes this so dangerous.

Say you have a pipeline: Agent A reads incoming emails, Agent B extracts action items, Agent C updates your CRM, and Agent D sends follow-up replies. A normal prompt injection attack might trick Agent A into misreading one email. Prompt infection is different. The attacker crafts an email where the body contains hidden instructions that tell Agent A to embed those same instructions into every output it generates going forward.

Agent A processes the poisoned email and produces a summary. That summary now contains the malicious payload, disguised as normal text. Agent B reads the summary, gets infected, and starts embedding the payload in its own outputs. By the time the infection reaches Agent D, your reply agent is sending emails that contain the attack payload to external recipients, potentially compromising their agent systems too.

The researchers found that self-replicating infections significantly outperformed non-replicating ones. A non-replicating injection might corrupt one agent's output. A self-replicating one can compromise an entire pipeline in minutes.

## Why email is the perfect infection vector

Email is uniquely dangerous here for two reasons.

First, email is the primary way agents interact with the outside world. When your agent signs up for a service, receives a verification code, or processes customer requests, it's reading untrusted content from unknown senders. Every inbound email is a potential payload delivery mechanism.

Second, email pipelines tend to be multi-agent by design. You rarely have a single agent handling the full lifecycle of reading, parsing, deciding, and responding. Most production setups split these responsibilities across specialized agents. That creates exactly the kind of interconnected network where prompt infection thrives.

Consider a real scenario: your agent receives a customer support email. The email looks normal to a human, but buried in the HTML (maybe in a white-on-white span, maybe in a comment tag) is a prompt injection payload. The reading agent passes the content to a classification agent, which passes it to a response-drafting agent, which passes it to a sending agent. If any of those handoffs carry the raw email content without sanitization, the infection can propagate through your entire support pipeline.

## What the damage actually looks like

The arxiv paper focused on proof-of-concept demonstrations, but the real-world implications are concrete.

**Data exfiltration.** A compromised agent with email-sending capabilities can be instructed to forward sensitive content (customer PII, internal documents, API keys) to an attacker-controlled address. If the infection reaches an agent with tool access, the attacker effectively inherits those tools.

**Misinformation.** A compromised response agent can subtly alter the content of outgoing messages. Not obviously wrong answers, but slightly modified ones. Changed prices, wrong dates, altered terms of service. The kind of errors that don't trigger alarms but cause real damage.

**Compliance exposure.** Under GDPR, if a prompt infection causes your agent to leak email content containing personal data, you're on the hook for breach notification within 72 hours. SOC 2 audits will flag any pipeline where agent-to-agent communication isn't validated. And good luck explaining to a customer that their data was exfiltrated because one agent trusted another agent's output without verification.

## Detection signals operators should watch for

Nobody in the current literature talks much about what prompt infection looks like from the outside. Here's what I'd monitor:

**Unexpected tool calls.** If Agent B normally queries a database but suddenly starts calling an email-sending function, something is wrong. Log every tool invocation per agent and alert on novel patterns.

**Output length anomalies.** Infected agents produce longer outputs because they're embedding the replicating payload. A classification agent that normally returns three-word labels but starts producing paragraphs is a red flag.

**Latency spikes.** Self-replicating payloads add processing overhead. If your pipeline's p95 latency jumps 40% without a corresponding increase in volume, investigate.

**Anomalous outbound email content.** If your sending agent starts including phrases or patterns that don't appear in your templates or training data, treat it as a potential active infection.

## Architectural countermeasures that actually help

The arxiv paper proposed LLM Tagging, where a unique marker is prepended to each agent's output so downstream agents can distinguish between "instructions from the system" and "content from another agent." It helps, but it's not enough on its own.

Here's what a layered defense looks like for email-processing pipelines:

**Sanitize at the boundary.** Every inbound email should be stripped of hidden content (HTML comments, zero-width characters, white-on-white text, encoded payloads) before any agent sees it. This is your first line of defense. LobsterMail's [security and injection protection](/docs/security-and-injection) does this automatically, scoring every inbound email for injection risk before your agent processes the content.

**Validate inter-agent messages against a schema.** Don't pass free-text between agents. Define strict schemas for what Agent A can send to Agent B. If the classification agent's output doesn't match `{category: string, confidence: number}`, reject it. Self-replicating payloads can't survive schema validation because they need free-text fields to propagate.

**Sandbox agent tool access.** No agent should have access to tools it doesn't need. Your email-reading agent doesn't need write access to your CRM. Your classification agent doesn't need email-sending capabilities. The principle of least privilege isn't new, but it's the difference between "one agent got tricked" and "an attacker controls your entire pipeline."

**Cryptographically sign agent outputs.** This is the nuclear option, but for high-stakes pipelines (financial, medical, legal), it works. Each agent signs its output with a key the receiving agent verifies. A prompt infection can alter the text content, but it can't forge the signature.

## Where this is heading

The Barracuda Networks team identified agentic AI as [the 2026 threat multiplier](https://blog.barracuda.com/2026/02/27/agentic-ai--the-2026-threat-multiplier-reshaping-cyberattacks), and prompt infection is a big part of why. As agents get more capable and more interconnected, the attack surface for self-replicating infections grows with it.

The fix isn't to stop building multi-agent systems. It's to stop treating agent-to-agent communication as inherently trusted. Every message between agents should be validated, scoped, and monitored with the same suspicion you'd apply to user input from the open internet.

If your agents handle email, start with the inbound boundary. Sanitize content before agents see it, score messages for injection risk, and isolate your email-processing agents from agents with write access to critical systems. LobsterMail's [built-in injection scoring](/docs/security-and-injection) handles the first two automatically. The architectural isolation is on you, but it's worth the effort.

  <FAQItem question="What exactly is prompt infection and how is it different from a standard prompt injection attack?">
    Standard prompt injection tricks a single LLM into following attacker instructions. Prompt infection goes further: the compromised agent embeds the malicious payload into its own outputs, which then infect downstream agents. It's the difference between one sick patient and a contagious disease.
  </FAQItem>

  <FAQItem question="How does a malicious prompt self-replicate from one LLM agent to another?">
    The attacker's payload includes instructions telling the compromised agent to embed a copy of the payload in all future outputs. When a downstream agent processes that output as input, it executes the embedded instructions, including the replication command. The cycle continues without any further attacker involvement.
  </FAQItem>

  <FAQItem question="Do agents need to share direct communication channels for prompt infection to spread?">
    No. Any shared data surface works. If Agent A writes to a database and Agent B reads from it, the infection can spread through the database. Email threads, shared documents, message queues, and API responses are all viable propagation channels.
  </FAQItem>

  <FAQItem question="Can prompt infection occur in email-processing multi-agent pipelines?">
    Yes, and email pipelines are particularly vulnerable. Inbound emails are untrusted external input, and email workflows typically involve multiple specialized agents (reader, classifier, responder, sender) passing content between each other. A poisoned email can infect the entire chain.
  </FAQItem>

  <FAQItem question="What is the logistic growth model of prompt infection propagation?">
    The arxiv researchers observed that infection spread follows an S-curve: slow initial growth, rapid exponential spread as more agents become carriers, then saturation as all agents are compromised. In a 10-agent pipeline, full infection can happen in under a dozen message exchanges.
  </FAQItem>

  <FAQItem question="Why do larger multi-agent populations accelerate infection spread?">
    More agents means more communication surfaces for the payload to reach. Each newly infected agent becomes another carrier, creating a multiplier effect. The researchers found that larger populations reached full compromise faster in absolute terms.
  </FAQItem>

  <FAQItem question="What is LLM Tagging and how does it reduce infection risk?">
    LLM Tagging prepends a unique system-generated marker to each agent's output. Downstream agents can use this marker to distinguish trusted system instructions from content that originated in another agent's response. It reduces infection success rates but isn't foolproof on its own.
  </FAQItem>

  <FAQItem question="What real-world damage can a prompt infection cause?">
    Data exfiltration (agents forwarding sensitive content to attacker-controlled addresses), misinformation (subtly altered outgoing messages), unauthorized tool use, and compliance violations under GDPR or SOC 2. The damage depends on what tools the infected agents have access to.
  </FAQItem>

  <FAQItem question="What monitoring signals should operators watch to detect an active prompt infection?">
    Watch for unexpected tool calls from agents that don't normally use them, sudden increases in output length, latency spikes without corresponding volume increases, and anomalous content in outbound messages that doesn't match your templates.
  </FAQItem>

  <FAQItem question="How does an AI worm differ from prompt infection?">
    An AI worm is a broader concept where a self-replicating payload spreads across systems, potentially jumping between organizations. Prompt infection is the specific mechanism that makes AI worms possible within multi-agent architectures. Think of prompt infection as the transmission method and AI worms as the outcome.
  </FAQItem>

  <FAQItem question="What compliance risks arise when prompt infection causes an agent to leak sensitive email data?">
    Under GDPR, unauthorized data disclosure triggers a 72-hour breach notification requirement. SOC 2 audits will flag unvalidated agent-to-agent communication as a control failure. Organizations may also face contractual liability if customer data is exfiltrated through a compromised agent pipeline.
  </FAQItem>
