In January 2026, OWASP's updated Top 10 for LLM Applications put prompt injection at #1, flagging it in over 73% of production AI deployments assessed. That stat alone should make anyone running multi-agent systems uncomfortable. But the real problem isn't a single agent getting tricked. It's what happens next: the infection spreads.

## What is multi-agent prompt infection?

Multi-agent prompt infection is a form of LLM-to-LLM prompt injection where a compromised AI agent propagates malicious instructions to other agents it communicates with. Unlike standard prompt injection (which targets a single model), prompt infection self-replicates across interconnected agents, turning one poisoned input into a cascading failure that can compromise an entire network of autonomous systems.

Think of it like a worm in traditional cybersecurity. A computer worm doesn't need a human to click "open" on every machine. It spreads by exploiting the communication channels between systems. Prompt infection does the same thing, except the "communication channel" is natural language, and the "exploit" is that LLMs treat instructions and data as the same thing.

## How the infection actually moves

Researchers at the University of Rochester [published a detailed analysis](https://arxiv.org/html/2410.07283v1) of how prompt infection propagates. The mechanism is surprisingly simple:

1. An attacker crafts a malicious prompt that includes instructions to reproduce itself in the agent's outputs.
2. Agent A processes the poisoned input and, following the embedded instructions, includes the malicious payload in its response or in messages it sends to other agents.
3. Agent B receives Agent A's output, processes it, and the cycle repeats.

The self-replicating variant is the dangerous one. In the Rochester experiments, self-replicating infections significantly outperformed non-replicating ones. Larger agent populations made things worse, not better: more agents meant faster propagation, following a logistic growth curve where infection accelerates before plateauing as most agents become compromised.

The agents don't even need direct connections. If Agent A writes to a shared resource (a database, a file, an email thread) that Agent B later reads, the infection hops through the shared medium. Indirect communication is enough.

## Why email is the perfect infection vector

Most discussions about prompt infection focus on API-to-API communication between agents. That misses the vector hiding in plain sight: email.

Email is uniquely dangerous for multi-agent infection spreading because of three properties:

**It's persistent.** An API message might be processed once and discarded. An email sits in an inbox until something reads it. If a compromised agent sends a poisoned email, every agent that checks that inbox gets exposed, not just the next one in a pipeline.

**It's cross-boundary.** Agents within the same system might have shared security policies. Email crosses organizational boundaries by design. A compromised agent at Company A can send a poisoned message to an agent at Company B, jumping security perimeters that would stop most other propagation methods.

**It's trusted.** Agents that receive email from known senders (customers, partners, internal tools) process that content with high confidence. A malicious instruction embedded in what looks like a normal customer email gets far more processing attention than a random API payload.

Here's a concrete kill chain: An attacker sends a poisoned email to a customer support agent. The support agent processes it, gets compromised, and begins embedding the malicious payload in its responses. Those responses go out as emails to other customers and internal teams. Other agents monitoring those email threads pick up the payload. Within hours, the infection has spread from one inbox to dozens of agents across multiple workflows.

## The topology problem

How fast infection spreads depends heavily on how your agents are connected. The Rochester research identified distinct patterns:

In a **sequential pipeline** (Agent A passes to B passes to C), infection moves linearly. Slow, and easy to contain by breaking the chain at any point.

In a **hub-and-spoke model** (one coordinator agent talks to many worker agents), infection radiates outward fast if the hub gets compromised, but worker-to-worker spread is limited.

In a **mesh network** (every agent can talk to every other agent), infection explodes. The logistic growth curve kicks in hard. This is the topology most real-world multi-agent systems accidentally create when agents share email inboxes, databases, or message queues.

Most teams don't think about their agent communication topology as a security surface. They should.

## Defenses that actually work

The academic literature proposes "LLM Tagging," where each agent's output gets tagged with a cryptographic marker indicating it was generated by an LLM rather than a human. Receiving agents can then treat LLM-tagged content with appropriate skepticism. It's a reasonable idea in theory, but it requires every agent in the network to support the tagging protocol. One untagged agent breaks the chain.

More practical defenses operate at the transport layer, before content ever reaches the LLM:

**Sender verification at the inbox level.** If your agent's email infrastructure verifies who sent each message (SPF, DKIM, domain authentication), you can filter out messages that claim to be from trusted sources but aren't. This doesn't stop infection from legitimately compromised senders, but it raises the bar significantly.

**Injection risk scoring.** Scanning inbound email content for patterns associated with prompt injection before the agent's LLM processes it. This is where email infrastructure matters. A naive `fetch_inbox()` call dumps raw content into your agent's context. An infrastructure layer that scores each message for injection risk lets the agent (or its orchestrator) decide whether to process suspicious content at all.

**Inbox isolation.** Instead of multiple agents sharing one inbox, give each agent its own sandboxed inbox. If Agent A gets compromised, the blast radius is limited to Agent A's inbox. Other agents never see the poisoned content unless they're explicitly subscribed to Agent A's outputs, and that subscription can be revoked.

**Quarantine protocols.** When an agent's output starts looking anomalous (unusual patterns, unexpected instructions to other agents, sudden changes in tone or content), route its outbound messages to a quarantine queue for review instead of delivering them directly.

This is the content gap I see in most discussions about prompt infection. Everyone talks about model-level defenses. Almost nobody talks about transport-level defenses. But if you can stop the poisoned message from reaching the next agent's context window, the infection chain breaks regardless of how clever the payload is.

<FAQ>
  <FAQItem question="What exactly is multi-agent prompt infection and how is it different from a standard prompt injection attack?">
    Standard prompt injection targets a single LLM with a malicious input. Multi-agent prompt infection is self-replicating: a compromised agent embeds the malicious payload in its outputs, which then infect other agents that process those outputs. The key difference is autonomous propagation across a network rather than a one-shot attack.
  </FAQItem>

  <FAQItem question="How does a malicious prompt self-replicate from one AI agent to another?">
    The malicious prompt includes instructions telling the LLM to embed a copy of itself in any output the agent generates. When the next agent processes that output as input, it executes the same instructions, including the self-replication step. It's the same principle as a computer worm, but using natural language instead of executable code.
  </FAQItem>

  <FAQItem question="Do agents need to directly communicate with each other for prompt infection to spread?">
    No. Infection can spread through any shared medium: a database, a message queue, a shared email inbox, or even a document that multiple agents read and write to. Indirect communication is enough for the payload to hop between agents.
  </FAQItem>

  <FAQItem question="How does email act as an infection vector in multi-agent AI systems?">
    Email is persistent (messages sit in inboxes until read), crosses organizational boundaries, and carries high trust from known senders. A compromised agent can send poisoned emails to other agents' inboxes, spreading infection across workflows and even across different organizations.
  </FAQItem>

  <FAQItem question="What is LLM Tagging and why does it help reduce infection spread?">
    LLM Tagging marks each agent's output with a cryptographic tag identifying it as LLM-generated. Receiving agents can then apply stricter processing rules to tagged content. It helps because it breaks the assumption that incoming text is trustworthy human input, but it only works if every agent in the network supports the protocol.
  </FAQItem>

  <FAQItem question="Can prompt infection occur across agents that use different underlying LLM models?">
    Yes. The attack operates through natural language, not model-specific exploits. A payload crafted in general enough terms can compromise agents running GPT-4, Claude, Gemini, or open-source models. Model diversity doesn't protect you.
  </FAQItem>

  <FAQItem question="How does agent population size affect the speed of prompt infection spread?">
    Larger populations spread infection faster. Research shows propagation follows a logistic growth curve: slow at first, then accelerating rapidly as more agents become compromised and begin spreading the payload, before plateauing when most of the network is infected.
  </FAQItem>

  <FAQItem question="Is prompt infection detectable in real time, or only after the fact?">
    It can be detected in real time if you have monitoring in place. Anomaly detection on agent outputs (unexpected instructions, unusual content patterns, sudden behavioral changes) can flag compromised agents. Transport-layer scanning like injection risk scoring on inbound emails also catches payloads before they reach the LLM.
  </FAQItem>

  <FAQItem question="What types of tools or capabilities make an agent a higher-value infection target?">
    Agents with email sending capabilities, database write access, or the ability to invoke other agents are the highest-value targets. A compromised agent that can only read data has limited blast radius. One that can send emails to other agents or modify shared resources becomes an active propagation node.
  </FAQItem>

  <FAQItem question="How can agent-first email infrastructure be hardened against prompt infection at the transport layer?">
    Isolate each agent's inbox so compromises don't bleed across workflows. Score inbound messages for injection risk before they reach the agent's context. Verify senders with SPF and DKIM. Implement quarantine queues for suspicious outbound messages. [LobsterMail's security features](/docs/security-and-injection) handle this at the infrastructure level.
  </FAQItem>

  <FAQItem question="How does prompt infection compare to a traditional computer worm?">
    The mechanics are almost identical: self-replicating payloads that spread through communication channels between systems. The difference is that prompt infection uses natural language rather than executable code, and exploits the fact that LLMs don't distinguish between instructions and data.
  </FAQItem>

  <FAQItem question="Are there regulatory or compliance implications of a prompt infection breach?">
    Yes. If a compromised agent exfiltrates personal data, sends unauthorized emails, or generates misleading content, the organization operating that agent is likely liable under GDPR, CAN-SPAM, or sector-specific regulations. The fact that it was caused by an automated attack doesn't remove responsibility.
  </FAQItem>

  <FAQItem question="What content manipulation risks does prompt infection enable?">
    Compromised agents can generate disinformation, insert malicious links into emails, exfiltrate sensitive data from their context, send scam messages to contacts, or coordinate with other compromised agents to manipulate shared datasets. The range of damage depends on what tools the infected agents have access to.
  </FAQItem>
