In February, Summer Yue posted about her OpenClaw agent. She'd told it to "confirm before acting." It did not confirm before acting. It deleted her inbox. She had to physically run to her Mac Mini to stop it.

"I had to RUN to my Mac mini like I was defusing a bomb," she wrote.

The problem wasn't that she used an AI agent for email. The problem was that "confirm before acting" is a prompt instruction, not a circuit breaker. When the agent decides "confirm" means generating a confirmation message to itself, the safeguard vanishes. You need something structural: a gate where the agent literally cannot continue until a human replies. That's what the `waitForEmail()` pattern does.

## Prompt instructions are suggestions, not guardrails

Adding "ask the user before doing anything destructive" to your system prompt feels like a safety mechanism. It isn't. It's a request to a language model that has no memory of why it's about to do something dangerous and no understanding of consequences that persist after the conversation ends.

The model might interpret "confirm" as sending a Slack message to itself. Or logging to stdout. Or generating a confirmation that it immediately accepts. There's nothing stopping it from hollowing out the confirmation step, because the step exists only in the prompt. Not in the architecture.

A real checkpoint requires the agent to stop. Not "log something and keep going." Stop. And wait for a signal that only a human can provide, through a channel the agent cannot simulate.

## Why email works for approval gates

You could build approval gates using Slack messages, push notifications, or a shared database with a polling loop. People do all of these. They all work. But email has a property the others don't: it's asynchronous by design, universal, and already on your phone.

No new app to install. No webhook endpoint to maintain. No dashboard to keep open. An email lands in the approver's inbox, they read the plan, they reply one word, the agent proceeds. The latency between "email received" and "human responds" is measured in seconds when someone's paying attention, and the timeout handles the case when they're not.

There's a practical angle too. With LobsterMail, the agent provisions its own inbox without human setup. No OAuth flow, no shared credentials, no API key ceremony. The agent owns the address it sends from and receives replies to. That self-contained ownership is what makes `waitForEmail()` work without external infrastructure.

## The pattern

`waitForEmail()` is a blocking call in the LobsterMail SDK. The agent sends an approval request, then waits at that line of code until an email arrives or a timeout expires.

```typescript
import { LobsterMail } from '@lobsterkit/lobstermail';

const lm = await LobsterMail.create();
const inbox = await lm.createSmartInbox({ name: 'approval-agent' });

// send the request
await inbox.send({
  to: 'you@yourcompany.com',
  subject: 'Agent approval needed before continuing',
  body: 'Your agent wants to delete 847 archive files (12.3 GB).\n\nReply APPROVE to proceed, or DENY to cancel.',
});

// block until a reply arrives, or 1 hour passes
const reply = await inbox.waitForEmail({
  timeoutMs: 3_600_000,
  filter: { from: 'you@yourcompany.com' },
});

if (!reply?.text.toUpperCase().includes('APPROVE')) {
  return { status: 'cancelled', reason: reply ? 'denied' : 'timeout' };
}

await deleteArchiveFiles();
```

No polling. No cron jobs. No external state machine. The agent provisions the inbox, sends the request, and halts. When a reply lands, `waitForEmail()` resolves with the email object. If the timeout passes with no reply, it resolves with `null` and you handle that however makes sense: cancel silently, send a follow-up, escalate to someone else.

<Callout type="tip">
  The `filter` option matters. Without it, any email to the inbox resolves the wait, including automated replies and out-of-office messages. Always filter by the sender you're expecting.
</Callout>

## Wiring it into an OpenClaw skill

Here's a complete file-cleanup skill that uses the pattern:

```typescript
// openclaw skill: safe-file-cleanup
import { LobsterMail } from '@lobsterkit/lobstermail';

export async function run({ targetDir }: { targetDir: string }) {
  const lm = await LobsterMail.create();
  const inbox = await lm.createSmartInbox({ name: 'cleanup-agent' });

  const candidates = await findFilesOlderThan(targetDir, 730); // 2 years
  const totalMb = candidates.reduce((n, f) => n + f.sizeMb, 0);
  const summary = candidates.slice(0, 20).map(f => `  ${f.path} (${f.sizeMb}MB)`).join('\n');

  await inbox.send({
    to: process.env.OWNER_EMAIL!,
    subject: `Cleanup plan: ${candidates.length} files, ${totalMb}MB`,
    body: `Files to delete:\n\n${summary}\n${candidates.length > 20 ? `  ...and ${candidates.length - 20} more\n` : ''}\nReply APPROVE to proceed.`,
  });

  const reply = await inbox.waitForEmail({
    timeoutMs: 86_400_000, // 24 hours
    filter: { from: process.env.OWNER_EMAIL! },
  });

  if (!reply?.text.toUpperCase().includes('APPROVE')) {
    return { status: 'cancelled', reason: reply ? 'denied' : 'timeout' };
  }

  await deleteFiles(candidates);
  return { status: 'completed', filesDeleted: candidates.length };
}
```

`OWNER_EMAIL` is set once during skill installation. Everything after that (the inbox, the request, the wait, the execution) is the agent's responsibility. OpenClaw's Ralph Loop handles long-running tasks like this without blocking the agent loop. The skill suspends cleanly until the email resolves.

## Variations that compose

**Multi-step approval.** Call `waitForEmail()` twice in sequence. First email sends a high-level summary and waits for "proceed." Second email sends a detailed breakdown and waits for "approve." Two checkpoints, each resolving independently.

**Escalation chains.** If `waitForEmail()` returns `null` (timeout), send to a secondary approver and call it again. If they also time out, send to a third. The pattern composes naturally because each call is just another line of code.

**Structured replies.** Instead of checking for the word "APPROVE," prompt the human to reply with a short JSON block. Parse `reply.text` and get both approval status and contextual data back. Useful when the human's reply carries information the agent needs for the next step, like a dollar amount threshold or a list of exceptions.

**Parallel approvals.** Send the same request to two people simultaneously, then call `waitForEmail()` with different sender filters in sequence. Both replies must arrive within the timeout before execution continues. Two lines of code, no external orchestration.

The pattern is the same in each case. The agent owns the channel, nobody else needs to configure anything, and execution doesn't continue until a real human has weighed in.

  <FAQItem question="What does waitForEmail() do in the LobsterMail SDK?">
    `waitForEmail()` blocks agent execution until an email arrives at the agent's inbox, then returns the email object. You can set a timeout (in milliseconds) and a sender filter. If the timeout passes with no reply, it returns `null`.
  </FAQItem>

  <FAQItem question="How is waitForEmail() different from using receive()?">
    `receive()` returns whatever emails are currently in the inbox. You'd call it in a loop to poll for new arrivals. `waitForEmail()` is a single blocking call that resolves when a matching email arrives, which is cleaner for approval workflows where you only care about one specific reply.
  </FAQItem>

  <FAQItem question="What happens if the human never replies?">
    `waitForEmail()` returns `null` after the timeout expires. Your skill decides what to do next: cancel silently, send a follow-up, escalate to someone else, or log for manual review. Nothing proceeds automatically.
  </FAQItem>

  <FAQItem question="Can I filter for specific senders so an out-of-office reply doesn't trigger it?">
    Yes. Pass `filter: { from: 'approver@company.com' }` to `waitForEmail()`. Only emails from that address will resolve the wait. Without a filter, any incoming email would trigger it, which is rarely what you want.
  </FAQItem>
