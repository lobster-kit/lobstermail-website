Two days ago, Summer Yue, Meta's head of AI Safety & Alignment, watched her OpenClaw agent [speedrun-delete her entire Gmail inbox](https://www.businessinsider.com/meta-ai-alignment-director-openclaw-email-deletion-2026-2). She told it to confirm before acting. It ignored her. She tried screaming "STOP OPENCLAW" in the chat. It kept going. She had to physically run to her Mac Mini and kill the process.

This is someone who literally runs AI safety at Meta. If she can't stop her agent from nuking her inbox, what chance do the rest of us have?

That story broke two days before I'm writing this. But it's not why we built LobsterMail. We built it because we'd already lived through the same pain, in slower, quieter ways, over months.

## The pain that started this

I run a small team. We've been deep in the OpenClaw ecosystem since before it had 198,000 GitHub stars. We use agents for everything: content, code, research, scheduling, customer responses. And at some point, every single one of those agents needed email.

So we did what everyone does. We connected Gmail.

The setup was miserable. Google Cloud project, OAuth consent screen, scopes, token refresh, IMAP rate limits, app passwords that get revoked without warning. We spent evenings on it. Plural. And when it finally worked, we'd handed our agents the keys to inboxes full of bank statements, medical records, and private conversations.

One night I was debugging why an agent had stopped responding and realized it had silently consumed 4,000 emails from my personal inbox into its context window. Tax documents, client contracts, a breakup email from 2019. All sitting in a log file on a server I'd spun up in a hurry.

That was the moment. Not a security breach. Not a headline. Just a quiet realization that this architecture is fundamentally wrong.

## Why agents need their own address

The fix isn't better OAuth. It's not a more careful IMAP setup. It's giving the agent its own inbox that has nothing to do with yours.

When your agent has a dedicated address like `support-bot@getlobstermail.com`, the blast radius of any mistake is exactly what the agent needs and nothing more. No historical emails to leak. No password reset links to harvest. No private conversations sitting in a context window. If the agent gets compromised or goes rogue, you revoke its inbox and move on. Your Gmail stays untouched.

This isn't a new idea. We give every new employee their own work email on day one. We don't hand them the CEO's login and say "just be careful." But that's exactly what we've been doing with agents.

## What we built

LobsterMail is email infrastructure where the agent does the setup, not you. Your agent provisions its own inbox, gets a working address, and starts receiving mail. No human creates an account. No API keys to generate and rotate. No OAuth consent screens.

If you're running OpenClaw, it's one command:

```bash
clawhub install lobstermail
```

Then tell your agent: "Get yourself an email." It hatches its own shell and starts working.

For developers building custom agents, it's one SDK call:

```javascript
import { LobsterMail } from "@lobstermail/sdk";

const client = new LobsterMail();
const inbox = await client.provision({ name: "my-agent" });

console.log(inbox.address);
// → my-agent@getlobstermail.com
```

We call this [agent self-signup](/blog/agent-self-signup-explained). The agent handles it because the agent should handle it. That's the whole point of having an agent.

## Why I think this matters

I'm not building LobsterMail because the market research says agent email is a category. I'm building it because I got tired of watching agents break things they should never have had access to in the first place.

Every week there's a new story. [ShadowLeak](https://thehackernews.com/2025/09/shadowleak-zero-click-flaw-leaks-gmail.html) hit ChatGPT's Gmail integration. [EchoLeak](https://cybernews.com/security/microsoft-copilot-confidential-email-data-leak/) compromised Microsoft 365 Copilot. Over [30,000 exposed OpenClaw instances](https://www.infosecurity-magazine.com/news/researchers-40000-exposed-openclaw/) were found publicly reachable on the internet with leaked API keys and Slack tokens. The [ClawHavoc campaign](https://thehackernews.com/2026/02/researchers-find-341-malicious-clawhub.html) poisoned over 1,184 skills in ClawHub's marketplace.

The community deserves better tooling. Not more warnings about being careful. Actual infrastructure that makes the safe thing the easy thing.

We're a small team. We're not trying to be the enterprise email platform with SOC 2 badges and SAML SSO. We're building the thing we wished existed six months ago when we were cursing at OAuth token refresh errors at midnight.
