You installed ClawHub, ran `clawhub login`, got a browser tab, clicked "Authorize," and then... what? Did it work? Is the token stored? What happens when your agent needs to authenticate at 3 a.m. on a server with no browser?

ClawHub's auth system is straightforward once you see all the pieces together. But the docs scatter these details across multiple pages, and most guides skip the headless use case entirely. Here's everything in one place.

## ClawHub authentication commands explained

Before diving into specifics, here's a quick reference of every auth-related command in the ClawHub CLI:

1. `clawhub login` opens a browser-based OAuth flow and stores your token locally
2. `clawhub login --token <token>` accepts an API token directly, no browser needed
3. `clawhub whoami` calls the API and prints your authenticated user handle
4. `clawhub logout` clears all stored credentials from your machine
5. `CLAWHUB_AUTH` environment variable provides a manual fallback for non-interactive environments

Each of these serves a different purpose, and picking the right one depends on whether you're at your laptop, running a CI pipeline, or deploying a long-lived agent.

## Browser login: the default flow

Running `clawhub login` with no flags opens your default browser to ClawHub's OAuth consent page. You approve access, the browser redirects to a local callback URL, and the CLI captures the resulting token.

```bash
clawhub login
The CLI stores the token in a local config file (typically `~/.config/clawhub/credentials` or the platform equivalent). Every subsequent command reads from this file automatically.

This works well when you're sitting at your machine installing skills, browsing the registry, or testing things out. It's the path of least resistance for interactive use.

Two things to keep in mind. First, the browser flow requires a GUI environment. If you SSH into a remote server and run `clawhub login`, you'll get an error or a URL you need to copy-paste into a local browser. Second, the stored token doesn't expire on its own, but it can be revoked from your ClawHub account settings. If that happens, you'll need to re-authenticate.

## Token login: for servers and agents

When there's no browser available, pass a token directly:

```bash
clawhub login --token lm_ch_abc123yourtoken
This skips the OAuth dance and writes the token to the same local credentials file. The CLI treats it identically to a browser-acquired token from that point on.

Where do you get the token? Generate one from your ClawHub account dashboard under "API Tokens" or "Developer Settings." You can create multiple tokens with different labels, which makes it easier to track which token is used where.

For agents running on remote infrastructure, token login is the standard approach. Your agent provisioning script can include the token as part of setup:

```bash
# During agent deployment
clawhub login --token "$CLAWHUB_TOKEN"
clawhub install my-required-skill
```

One consideration for production: don't hardcode tokens in scripts or commit them to version control. Use secret managers, encrypted environment variables, or whatever your deployment platform provides for sensitive values.

## Checking your identity with clawhub whoami

After logging in, verify that authentication actually worked:

```bash
clawhub whoami
```

This sends a `GET` request to ClawHub's API and prints back your user handle, account ID, and sometimes your plan tier. If the token is valid, you'll see something like:

```
Authenticated as: @your-handle
Account: acc_abc123
```

If the token is missing, expired, or revoked, you'll get a `401 Unauthorized` error instead. This makes `clawhub whoami` useful as a health check in scripts:

```bash
if ! clawhub whoami > /dev/null 2>&1; then
  echo "ClawHub auth failed, re-authenticating..."
  clawhub login --token "$CLAWHUB_TOKEN"
fi
```

A 401 from `clawhub whoami` doesn't always mean your token is bad. It can also mean a network issue prevented the API call, or the ClawHub API is temporarily down. Check connectivity before rotating credentials.

## The CLAWHUB_AUTH environment variable

There's a third authentication path that bypasses the credentials file entirely. Setting the `CLAWHUB_AUTH` environment variable tells the CLI to use that value as the auth token for the current session:

```bash
export CLAWHUB_AUTH="lm_ch_abc123yourtoken"
clawhub whoami  # uses the env var, ignores stored credentials
```

This is particularly useful in CI/CD pipelines, Docker containers, and ephemeral environments where you don't want to persist state to disk. The env var takes precedence over any stored token, so you can override credentials without modifying local files.

For production agent deployments, `CLAWHUB_AUTH` combined with a secret manager is often the cleanest pattern. No `clawhub login` step, no credentials file, no state to manage:

```bash
# In your CI config or container entrypoint
export CLAWHUB_AUTH=$(vault read -field=token secret/clawhub)
clawhub install required-skill-v2
clawhub whoami  # confirms we're authenticated
```

Is it safe for production? Yes, with the same caveats as any secret in an environment variable. Don't log it, don't echo it to stdout, and be aware that child processes inherit environment variables by default. If you're running untrusted code in the same process tree, a secret manager with short-lived tokens is better than a long-lived env var.

## Logging out

```bash
clawhub logout
```

This deletes the stored credentials file. It does not revoke the token on the server side. If someone copied your token before you logged out, that token still works until you revoke it from your ClawHub account dashboard.

For a complete credential reset: run `clawhub logout` locally, then go to your account settings and revoke the token. If you're using `CLAWHUB_AUTH`, unsetting the env var is sufficient for the local machine, but again, revoke the token server-side if it may have been exposed.

## Headless agents and non-interactive auth

This is the gap most guides ignore. If your agent runs autonomously (no human to click "Authorize"), the browser flow is a non-starter. Here's what works:

**For short-lived tasks** (CI jobs, one-off scripts): set `CLAWHUB_AUTH` as an environment variable and move on. The token is injected at runtime, used for the job, and disappears when the container or process exits.

**For long-running agents**: use `clawhub login --token` during initial provisioning, then add a `clawhub whoami` health check that re-authenticates if the token is revoked. Store the backup token in a secret manager, not in the agent's code.

**For agent frameworks that manage their own auth** (like [OpenClaw skills](https://docs.openclaw.ai/tools/clawhub)): the framework typically handles token injection. Check your framework's docs for where to configure the ClawHub token. You usually don't need to call `clawhub login` at all.

The pattern that causes the most trouble is provisioning an agent with `clawhub login` (browser flow) on your laptop, then deploying the agent to a server. The credentials file stays on your laptop. The server has no token. The agent fails silently the first time it tries to install or update a skill. Always use token-based auth for deployed agents.

## Troubleshooting common auth errors

**401 Unauthorized from `clawhub whoami`**: Your token is missing, expired, or revoked. Re-authenticate with `clawhub login --token` or check that `CLAWHUB_AUTH` is set correctly. Also verify network connectivity to the ClawHub API.

**"Browser not found" during `clawhub login`**: You're in a headless environment. Switch to `clawhub login --token <token>` or set `CLAWHUB_AUTH`.

**Token works locally but fails in CI**: The most common cause is a missing or misconfigured secret. Print the length of the token (not the token itself) to verify it's being injected: `echo ${#CLAWHUB_AUTH}`.

**"Permission denied" errors after successful auth**: Authentication and authorization are separate. Your token is valid, but your account may not have access to the resource you're requesting. Check your ClawHub plan and permissions.

## Token rotation for production agents

ClawHub tokens don't currently have automatic expiry schedules, which means they work until revoked. For production agents, set a manual rotation cadence. Quarterly is reasonable for most use cases.

The rotation process: generate a new token in your account dashboard, update the secret in your deployment pipeline or secret manager, verify with `clawhub whoami`, then revoke the old token. Doing it in this order avoids downtime.

If your agent uses LobsterMail for email alongside ClawHub for skill management, you're already familiar with this pattern. LobsterMail tokens follow the same generate-rotate-revoke cycle, and [the SDK handles re-authentication automatically](/docs/getting-started) when you provide a new token.
