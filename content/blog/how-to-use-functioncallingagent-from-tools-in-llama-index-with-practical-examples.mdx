If you've been building agents with LlamaIndex, you've probably run into two flavors of agent: the `ReActAgent` that reasons step-by-step in text, and the `FunctionCallingAgent` that delegates tool execution to the LLM's native function-calling API. The second one is faster, more reliable, and less prone to hallucinating tool names. But the docs don't always make the practical setup obvious.

This is a walkthrough of `FunctionCallingAgent.from_tools` in `llama_index`, from defining your first tool to wiring up real external API calls (including giving your agent its own email inbox). I'll cover which LLMs actually support it, how parallel tool calls work (and where they don't), and what to do when your agent silently ignores the tools you registered.

## What is FunctionCallingAgent in LlamaIndex?

`FunctionCallingAgent` is a LlamaIndex agent class that uses the native function-calling capabilities built into LLMs like GPT-4, Claude, and Gemini. Instead of prompting the model to output a text-based action (like ReAct does), it passes your tool definitions as structured function schemas. The model returns a structured JSON object specifying which function to call and with what arguments.

This matters because structured function calls are deterministic in format. The model isn't guessing at syntax. It's filling in a schema. That means fewer parsing failures, fewer retries, and more predictable agent behavior in production.

`FunctionCallingAgent.from_tools` is the constructor method you'll use 90% of the time. It takes a list of tools, an LLM, and optional configuration, then returns a ready-to-run agent.

## How to set up FunctionCallingAgent.from_tools in LlamaIndex

Here's the process broken into concrete steps:

1. Install `llama-index` and your LLM provider package (e.g., `llama-index-llms-openai`).
2. Define your tools as Python functions decorated with `@tool` or wrapped in `FunctionTool.from_defaults()`.
3. Initialize your LLM backend (OpenAI, Anthropic, Bedrock, etc.).
4. Call `FunctionCallingAgent.from_tools(tools=[...], llm=llm)`.
5. Run your agent with `agent.chat("your prompt here")`.

A minimal working example:

```python
from llama_index.core.agent import FunctionCallingAgent
from llama_index.core.tools import FunctionTool
from llama_index.llms.openai import OpenAI

def get_weather(city: str) -> str:
    """Get the current weather for a city."""
    return f"The weather in {city} is 72°F and sunny."

weather_tool = FunctionTool.from_defaults(fn=get_weather)
llm = OpenAI(model="gpt-4o")

agent = FunctionCallingAgent.from_tools(
    tools=[weather_tool],
    llm=llm,
    verbose=True,
)

response = agent.chat("What's the weather in Montreal?")
print(response)
That's it. The agent receives the question, the LLM decides to call `get_weather` with `city="Montreal"`, gets the result back, and formulates a response. The `verbose=True` flag prints each tool call so you can see exactly what's happening.

## Which LLMs work with FunctionCallingAgent

Not every LLM supports native function calling. Here's the compatibility picture as of early 2026:

| Provider | Models | Parallel calls supported |
|---|---|---|
| OpenAI | GPT-4o, GPT-4-turbo, GPT-3.5-turbo | Yes |
| Anthropic | Claude 3.5 Sonnet, Claude 3 Opus/Haiku | Yes |
| Google | Gemini 1.5 Pro, Gemini 2.0 Flash | Yes |
| AWS Bedrock | Claude via Converse API | Partial (see note) |
| Open-source | Llama 3, Mistral (via Ollama) | No |

The Bedrock situation deserves a callout. There's a known issue (GitHub #17606) where `BedrockConverse` doesn't always respect the `allow_parallel_tool_calls=False` flag. If you're running on Bedrock and getting unexpected parallel execution, pin your calls to sequential mode at the application level rather than relying on the flag.

<Callout type="warning">
  Open-source models through Ollama technically work with FunctionCallingAgent, but tool-calling reliability varies significantly by model. Test thoroughly before deploying. Mistral's function calling is more consistent than Llama 3's in my experience.
</Callout>

## FunctionCallingAgent vs ReActAgent

The short version: use `FunctionCallingAgent` when your LLM supports native function calling. Use `ReActAgent` when it doesn't, or when you need the model to show its reasoning chain explicitly.

`ReActAgent` works by prompting the model to output text in a `Thought → Action → Observation` loop. The agent parses that text to extract tool calls. This is flexible (any model can do it) but fragile (the model might format the action wrong, misspell a tool name, or hallucinate parameters).

`FunctionCallingAgent` skips the text parsing entirely. The LLM receives tool schemas as part of the API call and returns structured JSON. The failure mode shifts from "the model wrote bad syntax" to "the model chose the wrong tool," which is a higher-level problem that's easier to debug.

One practical difference: ReAct agents give you a visible chain of thought. Function-calling agents don't (unless you enable verbose mode). If you need an audit trail of *why* the agent chose a particular tool, ReAct might still be worth the tradeoff.

## Adding tools that call external APIs

The weather example is fine for demos, but real agents need to hit real APIs. The pattern is the same: wrap your API call in a function, convert it to a `FunctionTool`, and pass it to the agent.

Here's an example where an agent can send emails using LobsterMail. The agent provisions its own inbox and sends a message, no human configuration required:

```python
from llama_index.core.agent import FunctionCallingAgent
from llama_index.core.tools import FunctionTool
from llama_index.llms.openai import OpenAI
import subprocess
import json

def send_email(to: str, subject: str, body: str) -> str:
    """Send an email to the specified address."""
    # Your agent's email-sending logic here
    # Using LobsterMail, the agent provisions its own inbox
    result = subprocess.run(
        ["node", "-e", f"""
        const {{ LobsterMail }} = require('@lobsterkit/lobstermail');
        (async () => {{
            const lm = await LobsterMail.create();
            const inbox = await lm.createSmartInbox({{ name: 'my-agent' }});
            await inbox.send({{ to: '{to}', subject: '{subject}', body: '{body}' }});
            console.log(JSON.stringify({{ sent: true, from: inbox.address }}));
        }})();
        """],
        capture_output=True, text=True
    )
    return result.stdout

def check_inbox() -> str:
    """Check for new emails in the agent's inbox."""
    result = subprocess.run(
        ["node", "-e", """
        const { LobsterMail } = require('@lobsterkit/lobstermail');
        (async () => {
            const lm = await LobsterMail.create();
            const inbox = await lm.createSmartInbox({ name: 'my-agent' });
            const emails = await inbox.receive();
            console.log(JSON.stringify(emails.map(e => ({
                from: e.from, subject: e.subject, preview: e.preview
            }))));
        })();
        """],
        capture_output=True, text=True
    )
    return result.stdout

send_tool = FunctionTool.from_defaults(fn=send_email)
check_tool = FunctionTool.from_defaults(fn=check_inbox)
llm = OpenAI(model="gpt-4o")

agent = FunctionCallingAgent.from_tools(
    tools=[send_tool, check_tool],
    llm=llm,
    system_prompt="You are a helpful assistant that can send and check emails.",
    verbose=True,
)

response = agent.chat("Send a summary of today's tasks to sam@example.com")
The agent decides when to call `send_email` based on the conversation. You don't need to wire up routing logic. The LLM handles tool selection.

<Callout type="tip">
  Notice the `system_prompt` parameter. This is how you give the agent context about its role and available capabilities. Keep it concise. Long system prompts eat into your context window without improving tool selection.
</Callout>

## Passing system prompts and conversation memory

`FunctionCallingAgent.from_tools` accepts a `system_prompt` string that gets prepended to every conversation. For memory, LlamaIndex provides `ChatMemoryBuffer`:

```python
from llama_index.core.memory import ChatMemoryBuffer

memory = ChatMemoryBuffer.from_defaults(token_limit=4096)

agent = FunctionCallingAgent.from_tools(
    tools=[weather_tool],
    llm=llm,
    memory=memory,
    system_prompt="You are a weather assistant for Canadian cities.",
)

# First turn
agent.chat("What's the weather in Toronto?")
# Second turn — the agent remembers the conversation
agent.chat("How about Vancouver?")
```

The memory buffer keeps a sliding window of conversation history. Set `token_limit` based on your model's context window and how many turns you expect.

## Debugging: why your agent isn't calling tools

Three common reasons an agent ignores your tools:

**The function docstring is missing or vague.** The LLM uses the docstring to decide when a tool is relevant. `"""Do stuff."""` won't help. Write something like `"""Get the current weather forecast for a given city name. Returns temperature in Fahrenheit and conditions."""`

**The parameter types aren't annotated.** LlamaIndex generates the JSON schema from your Python type hints. If your function signature is `def search(query)` instead of `def search(query: str)`, the schema will be incomplete and the model may skip it.

**The tool name collides or is too generic.** If you have two tools with similar names and descriptions, the model might pick the wrong one or neither. Make names distinct: `search_documents` vs. `search_web` instead of `search1` vs. `search2`.

Enable `verbose=True` and check the console output. You'll see exactly which tools the model considered and which it called. If the tools show up in the function schema but never get invoked, the issue is almost always in the docstring or the prompt context.

## Logging and observability

For production agents, `verbose=True` isn't enough. LlamaIndex supports callback handlers that let you trace every tool call:

```python
from llama_index.core.callbacks import CallbackManager, LlamaDebugHandler

debug_handler = LlamaDebugHandler()
callback_manager = CallbackManager([debug_handler])

agent = FunctionCallingAgent.from_tools(
    tools=[weather_tool],
    llm=llm,
    callback_manager=callback_manager,
)

agent.chat("Weather in Halifax?")

# Inspect the trace
for event in debug_handler.get_event_pairs():
    print(event)
```

You can also plug in OpenTelemetry-compatible handlers to ship traces to Datadog, Honeycomb, or whatever observability stack you're running.

## Where to go from here

If you're building agents that need to interact with the real world (send emails, call APIs, read databases), `FunctionCallingAgent.from_tools` is the right starting point in LlamaIndex. It's more reliable than ReAct for models that support it, and the setup is straightforward once you understand the tool-definition pattern.

For email specifically, giving your agent its own inbox removes a whole category of plumbing. Instead of configuring SMTP credentials and managing OAuth tokens, the agent provisions what it needs at runtime. If that sounds useful, [LobsterMail's getting-started guide](/docs/getting-started) walks through the setup in under five minutes.

  <FAQItem question="What is the difference between FunctionCallingAgent and OpenAIAgent in LlamaIndex?">
    `OpenAIAgent` was the original OpenAI-specific implementation. `FunctionCallingAgent` is the provider-agnostic version that works with any LLM supporting function calling. Use `FunctionCallingAgent` for new projects since it supports multiple backends.
  </FAQItem>

  <FAQItem question="Can FunctionCallingAgent.from_tools handle tools that make HTTP requests or call external APIs?">
    Yes. Your tool function can contain any Python code, including HTTP requests, database queries, or subprocess calls. The agent treats it as a black box and only sees the return value.
  </FAQItem>

  <FAQItem question="How do you pass system prompts or context to a FunctionCallingAgent?">
    Pass a `system_prompt` string to `FunctionCallingAgent.from_tools()`. This gets prepended to every conversation and helps the model understand when to use which tools.
  </FAQItem>

  <FAQItem question="How do you handle errors thrown by tools inside a FunctionCallingAgent?">
    Wrap your tool function body in a try/except and return the error as a string. The agent will see the error message and can decide to retry or take an alternative action. Don't let exceptions propagate unhandled or the entire agent loop will crash.
  </FAQItem>

  <FAQItem question="Can FunctionCallingAgent.from_tools be used with open-source LLMs like Llama 3 or Mistral?">
    It can, through providers like Ollama or vLLM that expose function-calling APIs. Reliability varies by model. Mistral tends to produce more consistent tool calls than Llama 3. Test with `verbose=True` before deploying.
  </FAQItem>

  <FAQItem question="How do you add memory or conversation history to a FunctionCallingAgent?">

  </FAQItem>

  <FAQItem question="How do I log or trace which tools were called during a FunctionCallingAgent run?">
    Use LlamaIndex's `CallbackManager` with a `LlamaDebugHandler` for local debugging. For production, plug in an OpenTelemetry-compatible handler to export traces to your observability platform.
  </FAQItem>

  <FAQItem question="Why is my LlamaIndex agent not calling tools correctly?">

  <FAQItem question="How do you chain multiple FunctionCallingAgents together in LlamaIndex?">
    Create separate agents with different tool sets, then wrap each as a `QueryEngineTool` that a parent agent can call. The parent agent routes queries to the appropriate sub-agent based on tool descriptions.
  </FAQItem>

  <FAQItem question="How do you test a FunctionCallingAgent locally before deploying to production?">
