Semantic Kernel's plugin model is one of the best parts of the framework. You decorate a method with `[KernelFunction]`, register the class, and the planner figures out when to call it. For database queries, calendar lookups, web searches, it all just works. The model reads your `[Description]` attributes and sequences the calls without you writing orchestration logic.

Email is where things get complicated.

## The Microsoft Graph wall

The official path for agent email in Semantic Kernel is the `Microsoft.SemanticKernel.Plugins.MsGraph` package. It wraps a Microsoft 365 mailbox as a set of kernel functions. If your agent lives inside a corporate tenant where an IT admin has already granted the required application permissions and your Azure AD app registration is fully configured, it works fine. That setup happens once and your agent uses it going forward.

But that setup requires a human. Specifically, it requires a human with admin privileges in the right Azure AD tenant. Application permissions for email access need admin consent from every tenant you want to touch. Your agent can't request this on its own. Someone has to click through the consent flow, configure the app registration, and hand over the credentials before the agent can do anything.

For an agent that needs to sign up for a SaaS trial, catch a verification email, or receive replies from external services, this is a dead end. The whole value of an autonomous agent is that it handles things itself. If a human has to pre-provision email access for every new workflow, you've just added a manual step to every deployment.

## A different model

LobsterMail flips the direction. Instead of granting an agent access to an existing human mailbox, the agent creates its own inbox from scratch. One HTTP call to provision an address. No admin consent, no OAuth dance, no Azure configuration. The agent gets a real `@lobstermail.ai` address and can start sending and receiving immediately.

The SDK is TypeScript-first, but from .NET the REST API maps cleanly onto `HttpClient`. Three endpoints cover everything:

1. `POST /v1/inboxes` provisions an inbox, returns an address and ID
2. `GET /v1/inboxes/{id}/emails` polls for incoming messages
3. `POST /v1/inboxes/{id}/send` sends from the provisioned inbox

That's the full surface area for a working email plugin.

## Building the plugin

Here's a complete Semantic Kernel plugin wrapping all three operations:

```csharp
using System.ComponentModel;
using System.Net.Http.Json;
using Microsoft.SemanticKernel;

public class LobsterMailPlugin
{
    private readonly HttpClient _http;
    private const string BaseUrl = "https://api.lobstermail.ai/v1";

    public LobsterMailPlugin(string apiKey)
    {
        _http = new HttpClient();
        _http.DefaultRequestHeaders.Add("Authorization", $"Bearer {apiKey}");
    }

    [KernelFunction]
    [Description("Creates a new email inbox. Returns the inbox address and ID.")]
    public async Task<string> CreateInboxAsync(
        [Description("A short name for the inbox, e.g. 'signup-agent'")] string name)
    {
        var response = await _http.PostAsJsonAsync($"{BaseUrl}/inboxes", new { name });
        response.EnsureSuccessStatusCode();
        var result = await response.Content.ReadFromJsonAsync<InboxResult>();
        return $"Inbox ready: {result!.Address} | id: {result.Id}";
    }

    [KernelFunction]
    [Description("Fetches unread emails from an inbox. Returns subject, sender, and preview.")]
    public async Task<string> ReceiveEmailsAsync(
        [Description("The inbox ID returned from CreateInbox")] string inboxId)
    {
        var response = await _http.GetAsync($"{BaseUrl}/inboxes/{inboxId}/emails");
        response.EnsureSuccessStatusCode();
        var result = await response.Content.ReadFromJsonAsync<EmailListResult>();

        if (result?.Emails is not { Count: > 0 })
            return "No emails yet.";

        return string.Join("\n---\n", result.Emails.Select(e =>
            $"From: {e.From}\nSubject: {e.Subject}\nPreview: {e.Preview}"));
    }

    [KernelFunction]
    [Description("Sends an email from an inbox to any recipient.")]
    public async Task<string> SendEmailAsync(
        [Description("The inbox ID to send from")] string inboxId,
        [Description("Recipient email address")] string to,
        [Description("Subject line")] string subject,
        [Description("Body in plain text")] string body)
    {
        var response = await _http.PostAsJsonAsync(
            $"{BaseUrl}/inboxes/{inboxId}/send",
            new { to, subject, body });
        response.EnsureSuccessStatusCode();
        return $"Email sent to {to}.";
    }

    private record InboxResult(string Id, string Address);
    private record Email(string From, string Subject, string Preview);
    private record EmailListResult(List<Email> Emails);
}
```

A note on the `[Description]` attributes: the planner uses them to decide what each parameter means and how to populate it from context. If you write vague descriptions, the model guesses poorly. Be specific about what each argument expects and where it comes from.

## Registering it with the kernel

```csharp
var builder = Kernel.CreateBuilder();
builder.AddOpenAIChatCompletion(
    "gpt-4o",
    Environment.GetEnvironmentVariable("OPENAI_API_KEY")!);

var kernel = builder.Build();

var apiKey = Environment.GetEnvironmentVariable("LOBSTERMAIL_API_KEY")!;
kernel.ImportPluginFromObject(new LobsterMailPlugin(apiKey), "Email");
```

With auto function invocation enabled, the model can now call `Email-CreateInbox`, `Email-ReceiveEmails`, and `Email-SendEmail` as part of any plan. The planner reads the function descriptions, understands the data dependencies, and works out the order itself. Swap `AddOpenAIChatCompletion` for another connector if you're using Claude or a different model. The plugin doesn't care which model drives the planner.

## A real use case

Here's a prompt that exercises the full plugin end-to-end:

```
Sign up for a free trial at https://example-saas.com.
Use the email you create. When the verification email arrives,
extract the confirmation code and complete the signup.
```

The agent calls `CreateInbox`, gets an address, submits it on the signup form, polls `ReceiveEmails` until the verification message shows up, extracts the code, and finishes the flow. No human involvement. The inbox is provisioned for the task and can be discarded after, or kept around for ongoing threads.

<Callout type="tip">
  For agents that handle ongoing threads, store the inbox ID somewhere durable after creation and skip `CreateInbox` on subsequent runs. The inbox persists until you explicitly delete it.
</Callout>

This pattern covers the most common production use cases: service signups, confirmation codes, reply monitoring, and external system coordination.

## Security: scanning inbound email

When your agent reads email from the outside world, it's consuming untrusted input. Someone could send a message with text designed to hijack your agent's instructions. LobsterMail scores every inbound email for injection risk across multiple categories and returns that metadata alongside the message content.

Check the risk score before passing email bodies into your agent's context. The [security docs](/docs/security-and-injection) cover how to use injection scores in your pipeline. If you're building agents that process external email regularly, [the prompt injection in email guide](/blog/what-is-prompt-injection-in-email) goes deeper on the threat model.

## Going further: multi-agent handoffs

One plugin gives a single agent email capability. But the real power shows up when multiple Semantic Kernel agents each have their own inboxes and coordinate through email. An intake agent triages incoming messages, forwards relevant ones to a specialist agent, and the specialist replies directly.

We covered this pattern in detail in [Semantic Kernel multi-agent orchestration with email](/blog/semantic-kernel-multi-agent-orchestration-email-as-the-async-handoff-layer). If you're building pipelines where agents need async, durable handoffs that survive process restarts, email is the transport layer that makes it reliable.

<FAQ>
  <FAQItem question="Do I need an Azure account to use LobsterMail with Semantic Kernel?">
    No. LobsterMail has no dependency on Azure, Microsoft 365, or any tenant configuration. You register at lobstermail.ai, get an API key, and call the REST API with `HttpClient`.
  </FAQItem>

  <FAQItem question="Is there an official .NET or C# SDK for LobsterMail?">
    Not yet. The primary SDK is TypeScript/JavaScript via the `@lobsterkit/lobstermail` npm package. From .NET, use the REST API directly as shown in this guide. The API surface is small enough that a thin `HttpClient` wrapper covers everything.
  </FAQItem>

  <FAQItem question="What's the difference between this approach and the Microsoft Graph email plugin?">
    The MS Graph plugin accesses an existing Microsoft 365 mailbox and requires admin consent plus Azure AD configuration. LobsterMail creates new inboxes on demand with no pre-existing account, no admin, no tenant setup. They solve different problems for different use cases.
  </FAQItem>

  <FAQItem question="Can I use LobsterMail with other .NET AI frameworks besides Semantic Kernel?">
    Yes. The REST API works with any HTTP client in any language. Whether you're using Semantic Kernel, a custom agent loop, or a console app with raw `HttpClient`, the integration is identical.
  </FAQItem>

  <FAQItem question="How should I handle email polling without overloading the API?">
    Add a delay between `ReceiveEmails` calls. Two to five seconds works for most verification flows. For real-time delivery without polling, LobsterMail supports webhooks that push notifications to your endpoint. See the [docs](/docs/getting-started) for [webhook](/glossary/webhook) setup.
  </FAQItem>

  <FAQItem question="Can I use a custom domain instead of @lobstermail.ai?">
    Yes. Custom domain support is available on the Builder plan. Your agent's inbox becomes something like `agent@yourcompany.com` instead of the default lobstermail.ai address.
  </FAQItem>

  <FAQItem question="Is the free tier enough for development and testing?">
    For most development workflows, yes. 1,000 emails per month covers a lot of agent test runs. When you need higher volume or multiple inboxes, the Builder plan at $9/month is the next step.
  </FAQItem>

  <FAQItem question="Can multiple Semantic Kernel agents share one inbox?">
    They can, but managing concurrency gets tricky fast. Two agents might try to act on the same email. In practice, giving each agent its own inbox (one per task or per agent role) avoids race conditions entirely and makes debugging simpler.
  </FAQItem>

  <FAQItem question="How does LobsterMail handle prompt injection in email content?">
    Every inbound email is scored for injection risk across multiple categories (boundary manipulation, [system prompt](/glossary/system-prompt) override, data exfiltration, role hijacking, tool invocation, encoding tricks). The risk metadata comes back with the email so you can filter before your agent processes the body. See the [security guide](/docs/security-and-injection) for implementation details.
  </FAQItem>

  <FAQItem question="What happens if the inbox name I request is already taken?">
    If you use the TypeScript SDK's `createSmartInbox`, it resolves collisions automatically. From the REST API, if the name is taken, the API returns an available variation. You can also skip the name entirely and get a random address with guaranteed availability.
  </FAQItem>

  <FAQItem question="Can my agent receive and process email attachments?">
    Yes. The email objects returned by the API include attachment metadata, and you can fetch attachment content by ID. File size limits vary by plan tier.
  </FAQItem>

  <FAQItem question="Does this plugin work with Semantic Kernel's auto function calling?">
    Yes. Once you register the plugin with `ImportPluginFromObject`, auto function calling lets the planner invoke `Email-CreateInbox`, `Email-ReceiveEmails`, and `Email-SendEmail` as part of any plan without manual orchestration.
  </FAQItem>
