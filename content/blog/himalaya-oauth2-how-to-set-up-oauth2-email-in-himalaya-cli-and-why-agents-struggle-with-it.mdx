Himalaya is one of the better CLI email clients out there. It speaks IMAP and SMTP, outputs JSON for scripting, and works well in automated pipelines. But the moment you try to connect it to Gmail or Outlook using OAuth2, things get complicated fast.

The OAuth2 setup in Himalaya involves registering an app with your email provider, configuring TOML blocks with client IDs and secret references, storing tokens in your system keyring, and handling a browser-based redirect flow. For a human sitting at a laptop, it's 15-20 minutes of work. For an AI agent running headless on a server? It's a wall.

I've walked through this setup multiple times, and I want to lay out exactly how it works, where the pain points are, and what your options look like if you need an agent to handle email without a human clicking through consent screens.

## How to configure OAuth2 in Himalaya

Here's the full process, start to finish:

1. **Register an OAuth2 app** with your email provider (Google Cloud Console for Gmail, Azure App Registrations for Outlook).
2. **Install Himalaya with the OAuth2 feature flag** enabled. If you're building from source, you need `cargo install himalaya --features oauth2`. Pre-built binaries may or may not include it.
3. **Add the TOML configuration block** to your `~/.config/himalaya/config.toml` with `client-id`, `client-secret`, `auth-url`, `token-url`, and the correct scopes for your provider.
4. **Store your client secret in the system keyring** using the `.keyring` suffix in the config (e.g., `backend.auth.client-secret.keyring = "gmail-oauth2-client-secret"`).
5. **Run the authorization flow** by executing any Himalaya command. It will open a browser, prompt you to sign in, and capture the redirect token.

Here's a minimal Gmail TOML config block:

```toml
[accounts.gmail]
email = "you@gmail.com"
backend.type = "imap"
backend.host = "imap.gmail.com"
backend.port = 993
backend.login = "you@gmail.com"
backend.auth.type = "oauth2"
backend.auth.client-id = "your-client-id.apps.googleusercontent.com"
backend.auth.client-secret.keyring = "gmail-oauth2-client-secret"
backend.auth.access-token.keyring = "gmail-oauth2-access-token"
backend.auth.refresh-token.keyring = "gmail-oauth2-refresh-token"
backend.auth.auth-url = "https://accounts.google.com/o/oauth2/v2/auth"
backend.auth.token-url = "https://oauth2.googleapis.com/token"
backend.auth.scopes = ["https://mail.google.com/"]

message.send.backend.type = "smtp"
message.send.backend.host = "smtp.gmail.com"
message.send.backend.port = 465
message.send.backend.login = "you@gmail.com"
message.send.backend.auth.type = "oauth2"
message.send.backend.auth.client-id = "your-client-id.apps.googleusercontent.com"
message.send.backend.auth.client-secret.keyring = "gmail-oauth2-client-secret"
message.send.backend.auth.access-token.keyring = "gmail-oauth2-access-token"
message.send.backend.auth.refresh-token.keyring = "gmail-oauth2-refresh-token"
message.send.backend.auth.auth-url = "https://accounts.google.com/o/oauth2/v2/auth"
message.send.backend.auth.token-url = "https://oauth2.googleapis.com/token"
message.send.backend.auth.scopes = ["https://mail.google.com/"]
Yes, you need to duplicate the OAuth2 block for both IMAP and SMTP in the same account. The `https://mail.google.com/` scope covers both read and send for Gmail. Outlook uses different scopes, which I'll cover next.

## Gmail vs Outlook: the configuration differences

The two providers handle OAuth2 differently, and the differences trip people up.

**Gmail** uses the `xoauth2` method by default. You register your app in Google Cloud Console, enable the Gmail API, create OAuth2 credentials with a redirect URI (usually `http://localhost`), and download the client ID and secret. The scopes are straightforward: `https://mail.google.com/` gives you full IMAP and SMTP access.

**Outlook** uses a different auth URL structure (Microsoft identity platform) and requires PKCE (Proof Key for Code Exchange). In Himalaya, you enable this with `backend.auth.pkce = true`. The scopes also differ:

```toml
[accounts.outlook]
email = "you@outlook.com"
backend.type = "imap"
backend.host = "outlook.office365.com"
backend.port = 993
backend.login = "you@outlook.com"
backend.auth.type = "oauth2"
backend.auth.client-id = "your-azure-client-id"
backend.auth.client-secret.keyring = "outlook-oauth2-client-secret"
backend.auth.access-token.keyring = "outlook-oauth2-access-token"
backend.auth.refresh-token.keyring = "outlook-oauth2-refresh-token"
backend.auth.auth-url = "https://login.microsoftonline.com/common/oauth2/v2.0/authorize"
backend.auth.token-url = "https://login.microsoftonline.com/common/oauth2/v2.0/token"
backend.auth.pkce = true
backend.auth.scopes = ["https://outlook.office.com/IMAP.AccessAsUser.All"]
For Outlook SMTP, the scope changes to `https://outlook.office.com/SMTP.Send`. You'll need both if you want to read and send.

One gotcha: Microsoft's Azure app registration requires you to add the redirect URI explicitly under "Mobile and desktop applications," not "Web." If you pick the wrong platform type, you'll get a redirect mismatch error that's not obvious to debug.

## The "cannot wait for redirection" problem

This is the most common error people hit with Himalaya OAuth2. During the authorization flow, Himalaya starts a local HTTP server and opens your browser to the provider's consent screen. After you approve, the provider redirects back to `localhost` with an authorization code.

If anything interrupts that redirect (firewall blocking localhost, browser on a different machine, running in an SSH session, or the local server timing out), you get the infamous "cannot wait for redirection" error.

Fixes, depending on your situation:

- **Running over SSH?** Forward port 9090 (or whatever Himalaya uses) with `ssh -L 9090:localhost:9090 user@server`, then open the auth URL in your local browser.
- **Firewall issues?** Make sure nothing is blocking the loopback port Himalaya binds to.
- **WSL users?** The browser might not be able to open automatically. Copy the URL manually.
- **Headless server with no browser at all?** This is where things get genuinely difficult.

## The headless problem (and why agents can't do OAuth2)

OAuth2 was designed for humans. The entire flow assumes someone is sitting at a browser, reading a consent screen, clicking "Allow," and getting redirected. That's the security model. It's intentional.

For an AI agent running in a container or on a headless server, this is a dead end. There's no browser. There's no human to click consent. The agent can't complete the OAuth2 redirect flow on its own.

You can work around it by manually obtaining tokens and injecting them into the keyring. Run the OAuth2 flow once on your local machine, grab the access and refresh tokens, then store them on the server:

```bash
secret-tool store --label="gmail-oauth2-access-token" service himalaya
secret-tool store --label="gmail-oauth2-refresh-token" service himalaya
```

Himalaya *does* automatically refresh expired access tokens using the stored refresh token, so you won't need to redo this every hour. But refresh tokens themselves expire (Google's last 6 months if the app is in "testing" mode, indefinitely if published). When the refresh token dies, a human has to re-authorize.

For a personal project, this is manageable. For an agent that needs to provision its own email and operate autonomously, it's a maintenance burden that never goes away.

## An alternative for agents that need email

The OAuth2 dance exists because Gmail and Outlook weren't designed for agents. They were designed for humans using email clients. The security model is correct for that use case, but it creates real friction when your agent needs to send a verification email or receive a confirmation code.

This is exactly why we built [LobsterMail](https://lobstermail.ai). Instead of configuring OAuth2 credentials and managing token refresh cycles, your agent provisions its own inbox with one SDK call:

```typescript
import { LobsterMail } from '@lobsterkit/lobstermail';

const lm = await LobsterMail.create();
const inbox = await lm.createSmartInbox({ name: 'My Agent' });
// inbox.address → my-agent@lobstermail.ai
```

No OAuth2 redirect. No browser. No human. No token rotation. The agent handles everything itself, and the inbox is ready in under a second.

LobsterMail's free tier gives you 1,000 emails per month with no credit card required. If you're already using Himalaya for personal email and it's working well, keep using it. But if you're building an agent that needs to self-provision email on a headless server, fighting OAuth2 redirect flows is the wrong battle.

## When Himalaya OAuth2 is the right choice

To be clear, Himalaya is excellent for what it does. If you're a developer who wants a scriptable CLI email client that talks to your existing Gmail or Outlook account, OAuth2 is the correct auth method. It's more secure than app passwords, it scopes access properly, and once configured, it works reliably.

Use Himalaya with OAuth2 when:

- You're a human reading your own email from the terminal
- You need to access an existing Gmail/Outlook mailbox programmatically
- You can run the initial OAuth2 flow in a browser
- You have a process for re-authorizing when refresh tokens expire

Consider a different approach when:

- Your agent needs to create new email addresses on its own
- You're running headless with no browser access
- Token maintenance is a liability in your workflow
- You need email as infrastructure, not email as a client

The right tool depends on what you're actually building. Himalaya solves the "I want to script my existing email" problem well. The agent email problem is a different shape entirely.

<FAQ>
  <FAQItem question="What OAuth2 authentication methods does Himalaya support?">
    Himalaya supports `xoauth2` (used by Gmail) and `oauthbearer` methods. Gmail defaults to xoauth2, while some providers may use oauthbearer. You set this in the `backend.auth.type = "oauth2"` config block, and Himalaya negotiates the method with the server.
  </FAQItem>

  <FAQItem question="How do I register an OAuth2 app in Google Cloud Console for Himalaya?">
    Go to [console.cloud.google.com](https://console.cloud.google.com), create a project, enable the Gmail API, then go to Credentials and create an OAuth 2.0 Client ID. Set the application type to "Desktop app" and add `http://localhost` as a redirect URI. Copy the client ID and client secret into your Himalaya config.
  </FAQItem>

  <FAQItem question="How do I register an OAuth2 app in Microsoft Azure for Himalaya?">
    Go to Azure Portal > App Registrations > New Registration. Under "Redirect URI," choose "Mobile and desktop applications" (not Web) and add `http://localhost`. Under API Permissions, add `IMAP.AccessAsUser.All` and `SMTP.Send` from the Office 365 Exchange Online API.
  </FAQItem>

  <FAQItem question="What Cargo feature flags do I need to enable for Himalaya OAuth2?">
    Build with `cargo install himalaya --features oauth2`. Without this flag, OAuth2 support is not compiled into the binary. Pre-built release binaries may or may not include it; check the release notes for your version.
  </FAQItem>

  <FAQItem question="How does Himalaya store OAuth2 tokens securely?">
    Himalaya uses your system keyring (GNOME Keyring, macOS Keychain, or Windows Credential Manager) to store access tokens, refresh tokens, and client secrets. You reference them in the TOML config with the `.keyring` suffix, like `backend.auth.client-secret.keyring = "my-secret-name"`.
  </FAQItem>

  <FAQItem question="What scopes are required for Gmail OAuth2 in Himalaya?">
    The scope `https://mail.google.com/` grants full IMAP and SMTP access for Gmail. This single scope covers both reading and sending email. You can use more restrictive scopes like `https://www.googleapis.com/auth/gmail.readonly` if you only need read access.
  </FAQItem>

  <FAQItem question="What scopes are required for Outlook OAuth2 in Himalaya?">
    For IMAP, use `https://outlook.office.com/IMAP.AccessAsUser.All`. For SMTP, use `https://outlook.office.com/SMTP.Send`. You need both scopes if you want to read and send email from the same Outlook account.
  </FAQItem>

  <FAQItem question="Why does Himalaya show 'cannot wait for redirection' during OAuth2 setup?">
    This error occurs when Himalaya's local HTTP server can't capture the OAuth2 redirect. Common causes: running over SSH without port forwarding, firewall blocking localhost, or using WSL where the browser can't open automatically. Try forwarding the port with `ssh -L` or copying the auth URL to a local browser manually.
  </FAQItem>

  <FAQItem question="Can Himalaya use OAuth2 on a headless server without a browser?">
    Not directly. The OAuth2 flow requires a browser for the consent screen. You can work around this by completing the flow on a local machine first, then manually injecting the access and refresh tokens into the server's keyring using `secret-tool store`.
  </FAQItem>

  <FAQItem question="Does Himalaya automatically refresh expired OAuth2 access tokens?">
    Yes. If a valid refresh token is stored in the keyring, Himalaya will automatically request a new access token when the current one expires. However, refresh tokens themselves can expire (especially with Google apps in "testing" mode), which requires manual re-authorization.
  </FAQItem>

  <FAQItem question="How do I configure both IMAP and SMTP with OAuth2 in the same Himalaya account?">
    You need to duplicate the OAuth2 configuration block for both the backend (IMAP) and the `message.send.backend` (SMTP) sections within the same `[accounts.name]` block. They can share the same keyring entries for tokens and secrets.
  </FAQItem>

  <FAQItem question="Do I need PKCE enabled for Himalaya OAuth2?">
    For Outlook/Microsoft accounts, yes. Add `backend.auth.pkce = true` to your config. Gmail does not require PKCE for desktop OAuth2 flows, though enabling it doesn't hurt. PKCE adds an extra layer of security against authorization code interception.
  </FAQItem>

  <FAQItem question="What is the difference between himalaya and himalaya-repl for OAuth2?">
    The configuration syntax differs slightly between versions. The standard `himalaya` CLI uses `backend.type = "imap"` with `backend.auth.type = "oauth2"`, while `himalaya-repl` uses the older format `backend = "imap"` with `imap.auth.type = "oauth2"`. Check which version you're running and match the config format.
  </FAQItem>

  <FAQItem question="Can an AI agent manage OAuth2 credentials for Himalaya without human intervention?">
    Not reliably. The initial OAuth2 consent flow requires a human in a browser. Refresh tokens can sustain automated access for weeks or months, but they eventually expire. For agents that need fully autonomous email, [LobsterMail](https://lobstermail.ai) provides self-provisioned inboxes with no OAuth2 required.
  </FAQItem>

  <FAQItem question="Is Himalaya OAuth2 more secure than using app passwords?">
    Yes. OAuth2 tokens are scoped to specific permissions, expire automatically, and can be revoked without changing your account password. App passwords grant broad access and don't expire until manually deleted. OAuth2 is the recommended method for any provider that supports it.
  </FAQItem>
</FAQ>
