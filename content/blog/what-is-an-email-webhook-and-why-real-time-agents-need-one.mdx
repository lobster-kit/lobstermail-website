Your AI agent needs to know when an email arrives. It could be a customer reply, a verification code, a lead responding to outreach, or a support ticket that needs triaging. The question is: how does the agent find out?

There are two approaches. The first is polling, where your agent repeatedly asks "any new mail?" every few seconds. The second is webhooks, where the email server calls your agent the moment something happens. One is like refreshing your inbox compulsively. The other is like having a notification ping you.

For agents that need to act fast, the difference between these two isn't academic. It's the difference between a 30-second response and a 3-millisecond one.

## How email webhooks actually work

A webhook is an HTTP callback. You give the email service a URL (your agent's endpoint), and whenever a specified event occurs, the service sends an HTTP POST request to that URL with the event details.

For email, the most common event is `email.received`. When someone sends a message to your agent's inbox, the email provider fires a POST to your webhook URL containing the sender, subject line, preview text, and metadata. Your agent processes the payload and decides what to do next.

Here's what that looks like in practice:

```json
{
  "event": "email.received",
  "timestamp": "2026-03-09T14:22:00Z",
  "data": {
    "emailId": "em_abc123",
    "inboxId": "in_xyz789",
    "from": "client@example.com",
    "subject": "Re: Project update",
    "preview": "Looks good, but can we adjust the timeline..."
  }
}
```

Your agent receives this, parses it, and takes action. Maybe it drafts a reply. Maybe it updates a CRM record. Maybe it escalates to a human. The point is it happens within milliseconds of the email landing, not whenever the next polling cycle comes around.

## Polling vs. webhooks: the real cost

Polling seems simpler at first. Set up a cron job, hit the inbox API every 10 seconds, check for new messages. But the costs add up in ways that aren't obvious.

**Latency.** If you poll every 30 seconds, your average response time is 15 seconds. For a customer waiting on an automated reply, that's an eternity. Webhooks deliver in under a second.

**Wasted API calls.** Most polls return nothing. If your agent checks every 10 seconds and gets one email per hour, that's 359 empty requests for every useful one. At scale, across dozens of inboxes, you're burning through rate limits and API quotas on dead requests.

**Scaling pain.** Ten inboxes polling every 10 seconds means 60 API calls per minute. A hundred inboxes means 600. With webhooks, you make zero calls until something actually happens. The email provider pushes to you.

**Complexity.** Polling requires you to track state: which emails have you already seen? What if a poll fails midway? Do you re-fetch? Webhooks are stateless from your side. Each event is self-contained.

The tradeoff is that webhooks require your agent to have a publicly accessible HTTPS endpoint. For agents running on cloud infrastructure, that's usually straightforward. For agents running on a laptop behind NAT, it takes more setup (tunnels, serverless functions, or a relay service).

## What events can trigger a webhook?

Different email providers support different events. The most common ones:

- **email.received** fires when a new email lands in the inbox. This is the one most agents care about.
- **email.delivered** confirms that an outbound email was accepted by the recipient's server.
- **email.bounced** tells you a sent email was rejected (a hard bounce like a 550 error, for instance).
- **email.opened** and **email.clicked** track engagement with sent emails, useful for sales and marketing agents.

Not every provider supports every event. Some only offer `email.received`. Others provide granular tracking down to individual link clicks. Pick based on what your agent actually needs to react to, not what sounds impressive on a feature list.

## Security: verifying webhook signatures

Here's where a lot of implementations go wrong. When your agent exposes an HTTPS endpoint to receive webhooks, anyone who discovers that URL can send fake payloads to it. An attacker could craft a POST request that looks like a legitimate incoming email and trick your agent into acting on it.

The fix is signature verification. A good email webhook provider includes a cryptographic signature with every request, typically an HMAC-SHA256 hash of the request body using a shared secret. Your agent recomputes the hash and compares it to the one in the header. If they don't match, the request is forged.

```typescript
import { createHmac } from 'node:crypto';

function verifyWebhook(body: string, signature: string, secret: string): boolean {
  const expected = createHmac('sha256', secret)
    .update(body)
    .digest('hex');
  return expected === signature;
}
```

Skip this step and you're building on sand. Any agent that takes action based on email content (sending replies, accessing databases, triggering workflows) needs to verify that the email actually arrived.

## Handling failures gracefully

Webhooks are fire-and-forget from the provider's perspective. They send the POST and expect a 2xx response. If your endpoint is down, slow, or returns an error, the event could be lost.

Good providers implement retry logic with exponential backoff. If the first delivery fails, they wait a few seconds and try again. Then a minute. Then five minutes. After some number of consecutive failures (10 is common), they disable the webhook entirely so they're not hammering a dead endpoint.

Your agent needs to account for this. A few things to consider:

1. **Respond quickly.** Accept the webhook, return 200, and process the payload asynchronously. Don't do heavy work inside the request handler.
2. **Be idempotent.** You might receive the same event twice (retries, network hiccups). Use the event ID or email ID to deduplicate.
3. **Monitor your endpoint.** If your webhook gets disabled after too many failures, you won't know until emails start piling up unprocessed. Set up health checks.
4. **Have a fallback.** Even with webhooks, it's worth running a lightweight poll (once per minute, say) as a safety net. Catch anything the webhook might have missed.

## Common mistakes to avoid

Beyond the security and reliability concerns above, there are a few more traps that trip up developers building webhook-driven email agents.

**Ignoring content size limits.** Some webhook payloads include the full email body, attachments and all. If your endpoint can't handle large payloads, you'll get silent failures. Set appropriate body size limits on your server and use the email ID to fetch full content separately when needed.

**Hardcoding webhook URLs.** If you move servers or change domains, hardcoded URLs break instantly. Store your webhook registration details in configuration, and make sure your provisioning scripts can update the URL with your email provider automatically.

**Not logging events.** When something goes wrong (and it will), you need to trace what happened. Log every incoming webhook payload with its timestamp, event ID, and processing result. This makes debugging ten times easier than trying to reproduce issues in production.

## When agents need real-time email

Not every agent needs sub-second email response times. A weekly report summarizer can poll once a day and be fine. But several use cases fall apart without real-time delivery.

**Customer support triage.** A support agent that takes 30 seconds to acknowledge a ticket feels broken. One that responds in under 2 seconds feels like magic.

**Verification flows.** When your agent signs up for a service and needs to click a confirmation link in the email, every second of delay is a second the user is waiting.

**Multi-agent handoffs.** In systems where agents coordinate by email, polling latency compounds. Agent A emails Agent B, which emails Agent C. If each hop has a 15-second average poll delay, the chain takes 45 seconds before anyone acts.

**Time-sensitive alerts.** Monitoring agents that watch for outage notifications, payment confirmations, or security alerts can't afford to check every 30 seconds. The whole point is reacting before a human notices.

## Setting up your first email webhook

The general pattern is the same across most providers:

1. Stand up an HTTPS endpoint on your agent's server.
2. Register the URL with your email provider, specifying which events you want.
3. Store the webhook secret securely (it's usually shown only once).
4. Implement signature verification in your handler.
5. Process events and return 200 quickly.

If you're building an agent that needs its own email inbox with webhook support out of the box, [LobsterMail](/) handles steps 2 through 4 for you. Your agent registers a webhook with one API call and starts receiving verified payloads immediately. But the pattern works with any provider that supports webhooks.

The real decision isn't which provider to use. It's whether your agent's workflow genuinely benefits from real-time email delivery, or whether polling every minute is good enough. For most agent-to-human communication, real-time wins. For batch processing and analytics, polling is simpler and perfectly adequate.

Start with what your agent actually needs to do, then pick the delivery mechanism that fits.
