Your agent just processed 200 emails. One of them triggered a weird reply. You have no idea which one, what the agent decided, or why.

This is the reality for most teams running AI email agents in production. The agent works until it doesn't, and when it breaks, you're staring at a wall of unstructured text trying to reconstruct what happened three hours ago. Structured logging fixes this. Not by adding more output, but by making every log entry machine-readable, searchable, and traceable from the moment an email arrives to the moment the agent acts on it.

Here's how to do it right.

## Structured logging best practices for AI email agents

1. Use JSON format with a canonical schema including `timestamp`, `agent_id`, `action`, `thread_id`, and `result`.
2. Mask or hash all PII fields (email addresses, names) before log emission.
3. Assign a `correlation_id` to every email thread so multi-step actions stay linked.
4. Tag each log entry with an explicit log level: `ERROR`, `WARN`, `INFO`, or `DEBUG`.
5. Include an `intent_label` field to record the agent's classification decision.
6. Store security-relevant events (injection attempts, auth failures) in a separate audit trail.
7. Set distinct retention policies for production logs (90+ days) and debug logs (7-14 days).
8. Enforce role-based access controls on log queries so only authorized teams see sensitive data.

That's the short version. The rest of this guide covers each practice in detail, with schemas you can copy and mistakes I've seen teams make repeatedly.

## Why unstructured logs fail for email agents

Traditional logging, the kind where you write `console.log("Processing email from user")`, works fine when a human is reading output in real time. It falls apart when an autonomous agent handles hundreds of emails per hour across multiple inboxes.

The problem is threefold. First, unstructured logs can't be filtered programmatically. If you want every log entry where the agent flagged a message as suspicious, you're stuck with regex. Second, you lose the thread. An email might trigger a classification step, then a draft, then a send, then a follow-up check. In flat text logs, those events are interleaved with everything else the agent did that minute. Third, debugging a failed workflow means manually piecing together timestamps and hoping nothing got truncated.

Structured logging (JSON objects with consistent fields) solves all three. Each event is a self-contained record. You can filter, aggregate, and correlate without guessing.

## A JSON schema for email agent events

Every log entry your agent emits should contain these fields at minimum:

```json
{
  "timestamp": "2026-03-09T14:23:01.442Z",
  "agent_id": "inbox-agent-prod-01",
  "correlation_id": "corr_8f3a2b1c",
  "thread_id": "thread_abc123",
  "action": "classify_inbound",
  "intent_label": "verification_code",
  "log_level": "INFO",
  "result": "success",
  "duration_ms": 342,
  "metadata": {}
}
A few notes on specific fields:

**`correlation_id`** ties together every step the agent takes for a single email thread. When an email arrives, generate a correlation ID and pass it through classification, drafting, sending, and any follow-up. This is the single most useful field for debugging. When something goes wrong, you query by `correlation_id` and get the full story in order.

**`intent_label`** records what the agent decided the email was about. "verification_code", "sales_inquiry", "spam", "prompt_injection_attempt". Without this field, you can't answer the question "how many emails did the agent misclassify last week?"

**`action`** should be a fixed enum, not freeform text. Values like `receive_inbound`, `classify_inbound`, `draft_reply`, `send_outbound`, `flag_injection` are specific enough to filter on and general enough to stay consistent across agent versions.

The `metadata` object is your escape hatch for action-specific data. For a `send_outbound` action, it might contain `{"recipient_hash": "a1b2c3", "subject_length": 45}`. For a `classify_inbound` action, it might contain `{"confidence": 0.92, "model": "gpt-4o"}`.

## Masking PII before it hits the log pipeline

This is where I see the most mistakes. An agent processes an email from `jane.doe@company.com`, and the full address ends up in the log. Multiply that across thousands of emails, and your log store becomes a PII liability.

The fix: hash or redact PII fields at the point of emission, before the log leaves the agent process.

```typescript
import { createHash } from "crypto";

function hashEmail(email: string): string {
  return createHash("sha256").update(email).digest("hex").slice(0, 12);
}

// In your log entry:
// recipient_hash: hashEmail("jane.doe@company.com") → "a1b2c3d4e5f6"
Hash the email address. Hash the sender name. Never log the raw subject line if it might contain personal data. For debugging, the hash gives you a consistent identifier you can correlate across events without storing the actual address.

This matters for GDPR compliance specifically. Article 30 requires records of processing activities, but those records shouldn't contain more personal data than necessary. If your logs are subpoenaed or breached, hashed fields limit the exposure.

You can log the envelope (hashed sender, timestamp, size) without touching the body. ## Audit trails vs. operational logs

These serve different purposes and should live in different places.

**Operational logs** help you debug. They answer "what happened and why did it fail?" They're high-volume, short-lived, and queried by engineers. Set retention to 14-30 days for INFO level, 7 days for DEBUG.

**Audit trails** help you prove compliance. They answer "what did the agent do with this person's data?" They're lower-volume, long-lived, and might be queried by legal. Set retention to 1-3 years depending on your regulatory requirements.

The distinction matters because audit trail entries need to be immutable. Once written, they shouldn't be modifiable. Operational logs can be rotated, sampled, and compressed aggressively.

For audit trails, log every outbound action (email sent, email deleted, inbox created) with the correlation ID, a timestamp, and the agent's identity. Skip debug noise. An auditor doesn't care about your retry backoff logic.

## Log levels for email agent events

Not every event deserves the same severity. Here's a practical mapping:

| Event | Level |
|---|---|
| Email received and queued | `INFO` |
| Classification completed | `INFO` |
| Outbound email sent | `INFO` |
| Classification confidence below threshold | `WARN` |
| Prompt injection pattern detected | `WARN` |
| Outbound send failed (non-retryable) | `ERROR` |
| Agent unable to reach mail server | `ERROR` |
| Unhandled exception in agent loop | `FATAL` |

When an agent fails to classify an incoming message, that's a `WARN`, not an `ERROR`. The agent can still fall back to a default behavior. An `ERROR` means something broke and the workflow can't continue. This distinction matters because alerting on `ERROR` is practical. Alerting on every `WARN` will bury you in noise.

For prompt injection specifically, log at `WARN` level with the action `flag_injection` and include the injection risk score in metadata, but never log the actual injection payload. You don't want attack strings sitting in your log store where they could be re-ingested by another system. We wrote more about this attack surface in [prompt injection through email: what agents need to watch for](/blog/prompt-injection-email-agents).

## Controlling log volume and cost

An agent processing 1,000 emails per hour might generate 5,000-10,000 log entries (multiple steps per email). At scale, that's real money in log ingestion and storage.

Three strategies that work:

**Sampling for DEBUG level.** In production, you don't need DEBUG logs for every email. Sample 5-10% of threads by correlation ID (not random entries, full threads) so you can still debug representative workflows without paying for 100% volume.

**Tiered storage.** Hot storage (Elasticsearch, Datadog) for the last 7 days. Warm storage (S3 with Athena) for 7-90 days. Cold storage (Glacier) for audit trails beyond 90 days. Most queries hit the last 24 hours.

**Field cardinality limits.** Don't put unbounded values (like full email subjects) in indexed fields. Hash them or move them to the metadata blob. High-cardinality fields inflate index sizes and slow down queries.

## Connecting to observability platforms

Structured JSON logs plug directly into most observability tools. Langfuse and Traceloop work well for LLM-specific tracing (they understand token counts, model latency, prompt/completion pairs). Datadog, Grafana Loki, and Azure Monitor handle the infrastructure side.

The integration pattern is simple: emit JSON to stdout, use a log shipper (Fluentd, Vector, or the platform's native agent) to route entries to your backend. Tag each entry with `service: email-agent` and `environment: production` for filtering.

If you want to replay a failed workflow, the correlation ID is your key. Query all events for a single correlation ID, sort by timestamp, and you have the exact sequence of decisions the agent made. This is far more useful than trying to reproduce the failure by re-running the agent, because the email content and timing will never be identical.

## Getting started

If you're building email agents on LobsterMail, the SDK handles inbox provisioning and email delivery. The logging layer is yours. Start with the JSON schema above, add a correlation ID generator, hash your PII fields, and ship to whatever observability platform you already use. You don't need a custom framework. You need consistent field names and the discipline to use them.

The agents that run reliably in production aren't the ones with the fanciest models. They're the ones where, when something breaks at 2 AM, you can pull up a correlation ID and see exactly what happened in under a minute.
