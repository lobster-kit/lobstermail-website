If you built your OpenClaw agent's email handling on Gmail Pub/Sub, you've sat in at least one of these chairs: a half-configured Google Cloud project at 11pm, a `gog gmail watch serve` process that silently lapsed over the weekend, or an OAuth token refresh that [broke something in production](/blog/agent-gmail-token-breaking). The setup works when all five moving parts cooperate. The problem is the six ways it stops cooperating without telling you.

This guide walks through replacing that whole chain with LobsterMail webhooks. Your agent hatches its own inbox, you register a single endpoint, and Google leaves the picture entirely.

If you want the conceptual case for dropping Gmail first, [that post is here](/blog/openclaw-email-without-pubsub). This one is for people who have the Pub/Sub setup running today and want concrete, step-by-step instructions to switch.

## What you're currently maintaining

The standard OpenClaw Gmail email setup is a chain of five dependencies:

1. A Gmail account (created by a human, not the agent)
2. A Google Cloud project with the Pub/Sub API enabled
3. A push subscription pointing at a public HTTPS endpoint
4. `openclaw webhooks gmail run` (or the underlying `gog gmail watch serve`) keeping the watch alive
5. OAuth tokens that expire and need periodic renewal

When all five are healthy, your agent gets near-real-time email notifications. When any single link breaks, you're diagnosing a chain. Was the token stale? Did the watch lapse? Is the subscription still pointing at the right URL? Google's error messages are famously unhelpful here. The `invalid_grant` error alone can mean [six different things](/blog/agent-gmail-token-breaking).

The structural issue is deeper than any individual failure. This stack was designed for humans. You created the Gmail account. You enabled the API. You granted OAuth consent. Your agent is borrowing human infrastructure and calling it autonomous.

## What the LobsterMail version looks like

The agent provisions its own inbox. No Cloud Console, no OAuth flow, no human step at all.

```typescript
import { LobsterMail } from '@lobsterkit/lobstermail';

const lm = await LobsterMail.create();

// Agent creates its own inbox — no human account required
const inbox = await lm.createInbox();
console.log(inbox.address); // agent_abc@lobstermail.ai
```

Then register a webhook:

```typescript
const webhook = await lm.createWebhook({
  url: 'https://your-agent.example.com/hooks/email',
  events: ['email.received'],
});
```

<Callout type="warning">
  The webhook secret is only returned once on creation. Store it in an environment variable immediately. If you lose it, you'll need to delete and recreate the webhook.
</Callout>

That's the entire provisioning step. LobsterMail pushes to your endpoint the moment email arrives. No watch renewal cycle. No token refresh logic. No Google dependency.

## The migration, step by step

**Step 1: Install the SDK**

```bash
npm install lobstermail
```

**Step 2: Provision an inbox**

Run this once, or let your agent call it at startup:

```typescript
import { LobsterMail } from '@lobsterkit/lobstermail';

const lm = await LobsterMail.create();
const inbox = await lm.createInbox();

// Persist inbox.id and inbox.address somewhere durable
// inbox.address is what external services will send to
console.log(inbox.address);
```

**Step 3: Register your webhook**

```typescript
const webhook = await lm.createWebhook({
  url: 'https://your-agent.example.com/hooks/email',
  events: ['email.received'],
});

// Save to your env vars immediately — won't be shown again
// LOBSTERMAIL_WEBHOOK_SECRET=webhook.secret
```

**Step 4: Update your email handler**

The LobsterMail webhook payload is simpler than Gmail's Pub/Sub format. Here's what arrives when your agent receives an email:

```json
{
  "event": "email.received",
  "timestamp": "2026-03-02T09:00:00Z",
  "data": {
    "emailId": "em_abc123",
    "inboxId": "in_xyz789",
    "from": "sender@example.com",
    "subject": "Your verification code",
    "preview": "First 200 characters of the email body..."
  }
}
```

Always verify the HMAC signature before processing. This is where most webhook handlers fail silently. If you parse the body before verifying, the bytes change shape and the signature won't match. For a deep dive on this specific issue, see [the HMAC verification tutorial](/blog/openclaw-inbound-email-webhook-setup-with-hmac-verification).

```typescript
import { createHmac } from 'node:crypto';

function verifySignature(body: string, signature: string, secret: string): boolean {
  const expected = createHmac('sha256', secret)
    .update(body)
    .digest('hex');
  return expected === signature;
}

// In your HTTP handler (Express, Hono, Fastify, whatever you're using)
app.post('/hooks/email', express.raw({ type: 'application/json' }), async (req, res) => {
  const signature = req.headers['x-lobstermail-signature'] as string;

  if (!verifySignature(req.body, signature, process.env.LOBSTERMAIL_WEBHOOK_SECRET!)) {
    return res.status(401).send('Invalid signature');
  }

  const { event, data } = JSON.parse(req.body.toString());

  if (event === 'email.received') {
    // Respond immediately, process in background
    handleEmail(data).catch(console.error);
  }

  res.status(200).send('ok');
});
```

Note the use of `express.raw()` on the webhook route, not `express.json()`. Raw body first, verify, then parse. That order is load-bearing.

**Step 5: Test it**

Send a test email to your new `inbox.address`. You should get a webhook delivery within a few seconds. If you want to iterate on your handler without sending a new email every time, [set up ngrok with replay](/blog/how-to-debug-lobstermail-webhooks-locally-with-ngrok-replay) for a much faster loop.

<Callout type="tip">
  Run both setups in parallel for a day or two before decommissioning Gmail. It costs nothing on the free tier and gives you confidence in the new pipeline before you pull the plug on the old one.
</Callout>

**Step 6: Decommission the Gmail setup**

Once LobsterMail is confirmed working:

- Stop `gog gmail watch serve` or `openclaw webhooks gmail run`
- Delete the Pub/Sub subscription and topic in Google Cloud Console
- Revoke the OAuth token from your Google account settings
- Archive or delete the Gmail account if it was created solely for the agent

## What changes (and what doesn't)

The inbox address changes. Your agent will have an `@lobstermail.ai` address instead of a Gmail one. Update any external services that send to the old address. Everything downstream in your agent logic stays the same.

One thing worth noting: the webhook payload includes a 200-character preview, not the full body. If your existing handler relied on the full email content from the Pub/Sub notification, fetch the complete message using the `emailId`:

```typescript
if (event === 'email.received') {
  const email = await lm.getEmail(data.emailId);
  await handleEmail(email);
}
```

For routing decisions based on subject line or sender, the preview is usually enough. Only fetch the full body when your agent actually needs to read the content.

## Retry behavior and reliability

LobsterMail retries failed webhook deliveries with exponential backoff, up to 10 attempts over roughly 24 hours. If your endpoint is briefly unavailable, emails aren't lost. After 10 consecutive failures the webhook is auto-disabled to prevent noise. You can re-enable it:

```typescript
const webhooks = await lm.listWebhooks();

// Re-enable via API
await lm.updateWebhook('wh_abc123', { enabled: true });
```

Compare that to a lapsed Gmail watch, which fails silently. Your agent stops receiving email and you discover it when something downstream breaks, often days later.

## What surprised me about the switch

The mechanical migration is straightforward. What's less obvious until you've done it is how much the operational model simplifies. There's no watch renewal cron to monitor. No question of whether the OAuth token is still valid. No Google Cloud Console tab bookmarked for emergencies. The agent has an inbox. Email arrives. The webhook fires. That's the whole model.

If your Gmail setup is running smoothly and you're not actively touching it, there's no emergency. But if you're debugging a broken watch, starting a new agent, or tired of the five-dependency chain, this is a reasonable morning to make the switch. The free tier covers 1,000 emails per month with no credit card required.
