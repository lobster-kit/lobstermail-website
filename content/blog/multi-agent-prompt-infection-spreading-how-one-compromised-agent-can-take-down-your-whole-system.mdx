## What is multi-agent prompt infection?

Multi-agent prompt infection is an attack where a compromised AI agent spreads malicious instructions to other agents it communicates with. Unlike standard prompt injection (which targets a single model), prompt infection enables LLM-to-LLM propagation. A single poisoned message can cascade through an entire agent network, behaving like a self-replicating computer virus that hijacks each agent it reaches.

The concept moved from theoretical to demonstrated in late 2024 when researchers published "Prompt Infection: LLM-to-LLM Prompt Injection within Multi-Agent Systems." Their findings were uncomfortable. Self-replicating infections significantly outperformed non-replicating ones. Larger agent populations spread the infection faster, not slower. And the growth pattern followed a logistic curve, the same S-shaped model epidemiologists use for biological contagions.

By early 2026, OWASP had ranked prompt injection as the #1 critical vulnerability in its Top 10 for LLM Applications, appearing in over 73% of production AI deployments assessed. The multi-agent variant makes this worse by orders of magnitude.

## How prompt infection actually spreads

A traditional prompt injection attack is a one-shot deal. An attacker embeds malicious instructions in text that a single LLM processes. Maybe it's a hidden instruction in a web page, a comment in a document, or a line buried in an email body. The model reads it, follows the injected instructions, and the attack is contained to that one interaction.

Prompt infection removes that containment. Here's the mechanism:

1. Agent A receives a message containing an injected payload. The payload tells Agent A to include specific instructions in all of its future outputs.
2. Agent A processes the payload and begins embedding the malicious instructions in its responses.
3. Agent B receives Agent A's output as input. The embedded instructions now infect Agent B.
4. Agent B starts propagating the same payload to Agents C, D, and E.

The result looks less like hacking and more like an epidemic. Each infected agent becomes a carrier. The researchers found that in a network of 20 agents, a single infection point could compromise the entire population within a handful of communication rounds.

What makes this particularly dangerous is that the agents don't need direct connections. If Agent A talks to Agent B, and Agent B talks to Agent C, the infection jumps the gap. Indirect communication paths are enough. In mesh topologies where most agents communicate with several peers, infection saturates the network almost immediately. Chain topologies (where agents only talk to one neighbor) slow the spread but don't stop it.

## Email is the obvious attack surface

Think about how agents interact with the outside world. Most agent-to-agent communication happens through structured APIs or tool calls. Those channels are relatively controlled. But email? Email is the internet's open door.

An AI email assistant that reads incoming mail, extracts information, and passes it to downstream agents is a perfect infection vector. The attacker doesn't need access to your internal systems. They just need to send an email.

Consider a plausible scenario: your agent monitors a shared inbox for customer requests. It reads each email, classifies the intent, and routes tasks to specialized agents (a billing agent, a support agent, a scheduling agent). An attacker sends an email with injected instructions hidden in the body text. Your intake agent processes it, gets infected, and starts embedding the payload in every task it routes. Within minutes, your billing agent is exfiltrating data, your support agent is sending unauthorized responses, and your scheduling agent is creating chaos.

This isn't hypothetical architecture. It's how most agent-based email workflows are built today.

## The logistic growth model

The researchers modeled prompt infection spreading using the logistic growth equation, the same formula used for modeling how diseases spread through populations. The infection starts slowly (a few agents compromised), accelerates through exponential growth as each carrier infects new hosts, and eventually plateaus when most agents in the network are already compromised.

The practical implication: you don't have a gradual warning period. The infection appears manageable in the early rounds, then suddenly most of your system is compromised. By the time you notice something is wrong, containment may already be impossible.

Larger deployments are more vulnerable, not less. More agents means more communication paths, which means faster propagation. A 50-agent system doesn't have 2.5x the risk of a 20-agent system. It has dramatically more, because the number of potential infection paths grows exponentially with population size.

## What LLM tagging does (and doesn't do)

One defense the research community has explored is LLM Tagging, where each message between agents carries metadata identifying its source and processing history. The idea is that an agent can detect when it's receiving instructions that originated from another LLM rather than from a legitimate human operator or system prompt.

LLM Tagging reduces infection rates in controlled experiments. But it doesn't eliminate them. The core problem is that LLMs are fundamentally instruction-following machines. If a sufficiently well-crafted payload tells an agent to ignore tags, strip metadata, or reformat its output to look like a system-level instruction, the tagging defense can be bypassed.

Tagging is a layer, not a solution. It works best combined with other measures: input sanitization, output monitoring, and (here's where infrastructure matters) architectural isolation.

## Designing email infrastructure to contain infection

The most effective defense against multi-agent prompt infection spreading through email isn't better prompts or smarter models. It's architecture.

When every agent shares a single email pipeline, a compromised agent can poison the entire flow. But when each agent has its own isolated inbox with independent authentication, the blast radius shrinks dramatically. A compromised intake agent can't access the billing agent's inbox. It can't forge messages from the scheduling agent's address. Each inbox is a separate security boundary.

## Practical steps for auditing your agent workflows

If you're running multi-agent systems that process email (or any external input), here's what I'd recommend:

**Map your communication topology.** Draw out which agents talk to which other agents. Identify the high-connectivity nodes, the agents that receive input from many sources and pass output to many destinations. Those are your highest-risk infection points.

**Isolate inboxes.** Every agent that handles email should have its own inbox with its own credentials. Shared inboxes are shared attack surfaces.

**Score before processing.** Don't let your agents process raw email content without some form of injection risk assessment. Whether you use LobsterMail's built-in scoring or build your own classifier, the check needs to happen before the content enters your agent pipeline.

**Log everything.** If an infection does spread, you need to trace it back to the source message. Log the full content and metadata of every email your agents process, every inter-agent message, and every action taken. Without observability, you're debugging blind.

**Test with adversarial inputs.** Send your own agents emails containing known injection payloads. See what happens. Better to find the gaps yourself than to let an attacker find them for you.

## This problem is getting worse, not better

Barracuda Networks flagged agentic AI as "the 2026 threat multiplier reshaping cyberattacks." The concern isn't that agents are inherently insecure. It's that agents are being deployed faster than security practices can keep up. Every new multi-agent system that processes external input without architectural isolation is another network waiting to be infected.

The prompt infection research was a wake-up call. The question isn't whether your agents *can* be compromised through email. It's whether your infrastructure limits the damage when (not if) it happens.
