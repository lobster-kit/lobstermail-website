Every developer who's tested email flows has written some version of this code:

```typescript
await triggerSignupEmail();
await new Promise(r => setTimeout(r, 5000));
const emails = await getEmails();
expect(emails[0].subject).toMatch(/welcome/i);
```

Five seconds of sleep. Sometimes it works. Sometimes the email takes six seconds and the test goes red. You bump the timeout to eight seconds, the suite slows down, and your CI bills go up. One teammate sets it to fifteen seconds "just to be safe." Now a 40-test email suite takes ten minutes, and nobody trusts the green check anymore because everyone knows at least two of those tests passed by luck.

The root problem is that `setTimeout` has no idea when the email arrives. You're guessing. And every guess is either too short (flaky) or too long (slow). Polling loops are slightly better but still wasteful. You're burning CPU cycles checking an inbox every 500 milliseconds when the email might arrive in two seconds or twenty.

`waitForEmail()` is the LobsterMail SDK's answer. It holds execution until a matching email actually lands in the inbox, then returns immediately. No polling loop. No arbitrary sleep. If nothing arrives within your timeout window, it throws a descriptive error that tells you exactly what it was waiting for.

## How it works

Install the SDK:

```bash
npm install @lobsterkit/lobstermail
```

The pattern is three steps: create a dedicated test inbox, trigger the flow that sends email to it, then call `waitForEmail()` to block until the email lands.

```typescript
import { LobsterMail } from '@lobsterkit/lobstermail';

const lm = await LobsterMail.create();
const inbox = await lm.createSmartInbox({ name: 'test-agent' });

// Trigger whatever flow sends an email — signup, password reset, invoice
await yourService.signUp({ email: inbox.address });

// Block until the email arrives (30-second timeout)
const email = await inbox.waitForEmail({
  timeout: 30000,
  filter: { subject: /welcome/i },
});

expect(email.subject).toMatch(/welcome/i);
expect(email.body).toContain('Confirm your account');
```

The inbox address is a real `@lobstermail.ai` address that receives real email. Your service sends to it like any other address. `waitForEmail()` resolves the instant the email arrives, not one millisecond later.

## Filtering incoming email

Without a filter, `waitForEmail()` returns the first email that shows up after the call. For simple flows, that's fine. When there's more than one email in flight, you'll want to narrow it down:

```typescript
const email = await inbox.waitForEmail({
  timeout: 30000,
  filter: {
    subject: /verification code/i,     // regex match on subject
    from: 'no-reply@yourservice.com',   // exact sender match
  },
});
```

Subject filters accept strings (exact match) or regex. The `from` filter accepts a full address or a domain fragment, so `'@yourservice.com'` matches any sender on that domain.

If multiple emails arrive before your filter matches, `waitForEmail()` checks each one in order and returns the first match. Non-matching emails aren't consumed. Subsequent calls can still see them.

## Pulling verification codes and links

The most common thing you'll do once the email arrives is extract a code or a link:

```typescript
const email = await inbox.waitForEmail({
  timeout: 30000,
  filter: { subject: /your verification code/i },
});

// Extract a 6-digit code from the plain text body
const match = email.body.match(/\b(\d{6})\b/);
if (!match) throw new Error('No verification code found in email body');
const code = match[1];

await yourService.verifyCode(code);
```

`email.body` gives you the plain text version. For HTML emails with embedded links, use `email.htmlBody`:

```typescript
const linkMatch = email.htmlBody.match(/https?:\/\/[^\s"]+\/verify[^\s"<]*/);
if (!linkMatch) throw new Error('No verification link found');
const verificationUrl = linkMatch[0];
```

<Callout type="tip">
  Every incoming email includes injection risk scoring at `email.securityMetadata.injectionRisk`. You probably don't need it for most tests, but if you're writing assertions for your prompt injection filtering logic, the field is there.
</Callout>

## Setting up Jest and Vitest

`waitForEmail()` is async and throws on timeout, so it slots into both frameworks without any adapter code. The one thing you need to configure is the test timeout. Jest's default of 5 seconds won't work for real email delivery.

Set it globally:

```typescript
// jest.config.ts
export default {
  testTimeout: 60000,
};
```

Or per test:

```typescript
it('sends a welcome email on signup', async () => {
  const inbox = await lm.createSmartInbox({ name: 'signup-test' });
  await yourService.signUp({ email: inbox.address });

  const email = await inbox.waitForEmail({
    timeout: 45000,
    filter: { subject: /welcome/i },
  });

  expect(email.body).toContain('Get started');
}, 60000);
```

Set the SDK's `timeout` shorter than the framework timeout on purpose. That way the SDK throws a descriptive error (inbox address, filter used, elapsed time) before Jest's generic "Exceeded timeout" fires. When you're debugging at midnight, context in the error message saves you from guessing whether the email never sent or just arrived late.

Vitest uses the same `timeout` option on `it()` and `test()`. The pattern is identical.

## Playwright E2E flows

For end-to-end tests where you're driving a browser through a real signup form, the inbox address goes straight into the email field:

```typescript
import { test, expect } from '@playwright/test';
import { LobsterMail } from '@lobsterkit/lobstermail';

test('email verification flow', async ({ page }) => {
  const lm = await LobsterMail.create();
  const inbox = await lm.createSmartInbox({ name: 'playwright-test' });

  await page.goto('https://yourapp.com/signup');
  await page.fill('[name="email"]', inbox.address);
  await page.fill('[name="password"]', 'TestPass123!');
  await page.click('[type="submit"]');

  const email = await inbox.waitForEmail({
    timeout: 30000,
    filter: { subject: /verify your email/i },
  });

  const link = email.htmlBody.match(/https?:\/\/[^\s"]+\/verify[^\s"<]*/)?.[0];
  expect(link).toBeTruthy();

  await page.goto(link!);
  await expect(page.locator('h1')).toContainText('Email verified');
});
```

Each test gets a fresh inbox. No cleanup needed. No shared state leaking between tests. Disposable addresses are disposable, and inbox isolation means a flaky signup-flow test can't corrupt the assertions in your password-reset test running in parallel.

## Handling timeout errors

When `waitForEmail()` times out, it throws an `EmailTimeoutError` with the context you need to debug:

```typescript
try {
  const email = await inbox.waitForEmail({ timeout: 30000 });
} catch (err) {
  if (err instanceof EmailTimeoutError) {
    console.error(
      `No email in ${err.timeoutMs}ms for ${err.inboxAddress} (filter: ${JSON.stringify(err.filter)})`
    );
  }
  throw err;
}
```

In CI, I usually let the error propagate without catching. The test output includes the inbox address, the filter, and the wait duration. That points you straight at your service's outbound email logs instead of leaving you wondering whether the test itself was wrong. Compare that to a bare `setTimeout` failure, which tells you nothing except "the assertion ran before the data was ready."

## When receive() is the better choice

`waitForEmail()` is for flows where you don't know when the email will arrive. If you're running unit tests against fixtures, or your test triggers email synchronously and the message is already sitting there, `inbox.receive()` returns whatever's in the inbox without waiting.

A few other cases where `receive()` fits better: asserting that no email was sent (an empty array is a clean pass), or batch-checking a set of emails that all arrive together. The [sandbox testing guide](/blog/testing-agent-email-sandbox) goes deeper on fixture patterns and running your entire email test suite without touching production infrastructure. If you're building tests for an agent that signs up for external services on its own, [agent self-signup explained](/blog/agent-self-signup-explained) covers the provisioning flow. And if you need `waitForEmail()` for human-in-the-loop approval gates rather than testing, the [waitForEmail() pattern post](/blog/the-waitforemail-pattern-human-in-the-loop-email-for-openclaw-agents) covers that use case.

<FAQ>
  <FAQItem question="What does waitForEmail() do exactly?">
    It blocks execution until an email matching your filter arrives in the specified inbox, then returns the email object. If no matching email arrives within the timeout window, it throws an `EmailTimeoutError` with the inbox address, filter, and elapsed time.
  </FAQItem>

  <FAQItem question="How long should I set the timeout?">
    30 seconds is a reasonable default for most email flows. For CI environments where delivery might be slower, 45-60 seconds gives more headroom. Set your test framework's own timeout a few seconds higher than the SDK timeout so you get the SDK's descriptive error rather than a generic framework timeout.
  </FAQItem>

  <FAQItem question="What happens when waitForEmail() times out?">
    It throws an `EmailTimeoutError` that includes the inbox address, the filter you specified, and how long it waited. This makes it much easier to debug than a generic timeout because you can see exactly what the test was waiting for and check your service's email logs accordingly.
  </FAQItem>

  <FAQItem question="Can I filter by email body content?">
    Not directly in the filter object. The current filters are `subject` and `from`. For body-based matching, call `waitForEmail()` with a subject filter, then assert on `email.body` or `email.htmlBody` after it returns.
  </FAQItem>

  <FAQItem question="Does waitForEmail() work with any email service sending to @lobstermail.ai?">
    Yes. Your service just needs to send to the inbox address. From your service's perspective, it's a normal email address. No special integration or webhook required on the sending side.
  </FAQItem>

  <FAQItem question="How is waitForEmail() different from receive()?">
    `receive()` returns whatever emails are currently in the inbox and exits immediately. `waitForEmail()` blocks and waits for a new email to arrive. Use `receive()` when the email should already be there; use `waitForEmail()` when you don't know how long delivery will take.
  </FAQItem>

  <FAQItem question="Can I wait for multiple emails in a single call?">
    Not with one `waitForEmail()` call. For multiple emails, call it multiple times with different filters, or use `receive()` after your flow completes to check everything at once. The [sandbox testing guide](/blog/testing-agent-email-sandbox) has examples of multi-email assertion patterns.
  </FAQItem>

  <FAQItem question="Does this work on the free tier?">
    Yes. The free tier supports 1,000 emails per month, which covers most test suites comfortably. Each test inbox receives a small number of emails, so even a large suite running daily stays well within those limits.
  </FAQItem>

  <FAQItem question="Is there a cost to creating lots of test inboxes?">
    Inboxes themselves don't have a per-inbox cost. You pay for emails sent and received, not for the number of addresses you create. Spinning up a fresh inbox per test is the recommended pattern and doesn't add meaningful cost.
  </FAQItem>

  <FAQItem question="Can I use waitForEmail() in CI/CD pipelines?">
    Yes, and it's a common use case. Make sure your pipeline's job timeout is longer than your test timeout, and set the `LOBSTERMAIL_TOKEN` environment variable so the SDK doesn't try to auto-signup on every run.
  </FAQItem>

  <FAQItem question="What test frameworks does waitForEmail() support?">
    Any framework that supports async/await works: Jest, Vitest, Mocha, Playwright, Cypress (with async support). The SDK is framework-agnostic. The main thing to configure is the test timeout.
  </FAQItem>

  <FAQItem question="Can I use a custom domain address for testing?">
    Yes. If you have a custom domain configured with LobsterMail, you can provision test inboxes on that domain using the same SDK methods. See the [Docs](/docs/getting-started) for custom domain setup.
  </FAQItem>

  <FAQItem question="Is waitForEmail() safe to use in parallel tests?">
    Yes. Each inbox is isolated. A `waitForEmail()` call on one inbox only sees emails delivered to that specific inbox. Running parallel tests with separate inboxes works cleanly with no cross-contamination between test cases.
  </FAQItem>
</FAQ>
